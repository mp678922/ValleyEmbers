# 文字選擇冒險遊戲企劃

## 企劃目標

製作一款以文字敘事為核心的選擇指令冒險遊戲。玩家是一名來自山谷外、被山谷村收留的男性外來者，在幾乎全由年輕女性組成且與世隔絕的小村莊中慢活、與村民互動、協助村莊發展，並在需要資源時離開村莊，前往山谷森林、危險地區或遺跡探索。

本企劃目前從最小可行版本開始，先確立核心循環、基礎能力、村莊發展與村民設定，後續再逐步擴充世界觀、地圖、事件、遺跡、道具與結局。

## 技術執行方向

第一版預計以 Node.js 提供本機網頁伺服器，玩家透過瀏覽器遊玩。伺服器負責提供靜態網頁與 JSON 資料檔，遊戲狀態運算先放在瀏覽器端處理，方便快速迭代場景與資料。

存檔策略：

- 遊戲紀錄以 JSON 格式保存。
- 第一版使用瀏覽器 `localStorage` 保存本機存檔。
- 玩家必須按下 `存檔` 才把目前狀態寫入本機存檔；一般行動只改變目前運行中的遊戲狀態。
- 玩家可從網頁匯出目前紀錄為 `.json` 檔。
- 玩家可從網頁匯入先前匯出的 `.json` 記錄檔，並覆寫目前本機存檔。
- 第一版先使用單一 `localStorage` 存檔；未來若需要多存檔欄位，可再改成多組本機存檔或伺服器端存檔。

## 核心體驗

遊戲由兩種節奏交替構成：

- 村莊慢活：交談、幫忙、建設、製作、照顧村民需求。
- 村外探險：探索野外、採集資源、進入遺跡、承擔生命與體力風險。

玩家每回合會看到一段場景描述，以及數個可選行動。每個行動可能造成以下結果：

- 推進時間
- 推進劇情
- 移動到新場景
- 取得或失去道具
- 觸發危險
- 消耗體力
- 損失或恢復生命
- 提升或降低村民好感
- 改變村莊發展狀態
- 解鎖新的選項
- 導向結局

玩家的主要壓力來自時間、生命、體力、攜帶重量、村莊需求與村民關係。森林普通探索中生命歸零時不直接死亡，而是被神秘善良野獸救回村莊，損失時間，且身上可懲罰遺失的道具各自有機率消失；體力不足時，玩家無法執行特定特殊行動；時間推進會影響村民作息、地點開放、探索風險與每日事件；村莊若長期缺乏男丁與外部物資，可能導致防衛空洞、重勞務停擺、外界勢力施壓、村民離開或進入壞結局。

## 主角設定

- 身份：來自山谷外、被山谷村收留的男性外來者。
- 起點：主角原本生活在山谷之外，但他的過去情報不作為開場重點。開場只呈現他不願回憶過去，只知道眼前那道耳語最重要，因而攀下深谷，最後在谷中遭遇野獸襲擊，醒來後被山谷村的少女們圍在共同宿舍中。主角會被呼喚的核心原因，是他來自山谷外，不屬於山谷村與山谷之神之間的舊契約，因此能成為打破規則的外來變數。
- 處境：村莊願意收留主角，但每位村民對他的好感不同。
- 敘事功能：主角既是受助者，也是村莊重新取得外部資源與防衛能力的關鍵。
- 設計重點：主角不能一開始就被全村接納，必須透過行動、承諾與冒險逐步建立位置。

### 初始主角參數

主角初始 save data 以 `docs/data/save/save-template.json` 為準。

- 生命：`life: 10`，`maxLife: 10`。
- 體力：`stamina: 10`，`maxStamina: 10`。
- 最大攜帶重量：`maxCarryWeight: 10`。
- 攜帶技能：`carrySkill: 0`。
- 採集技能：`gatheringSkill: 0`。
- 探索技能：`explorationSkill: 0`。
- 近戰武器技能：`meleeWeaponSkill: 0`。
- 遠程武器技能：`rangedWeaponSkill: 0`。

初始攜帶物品：

- `sword`：`1`，顯示名為老舊的劍。
- `fluorite`：`0`，顯示名為螢石。

初始道具只有老舊的劍；初始螢石為 `0`。伊蓮初見面事件會給 `10` 個螢石，並告知螢石可在村莊內用來交換物品。其他補給必須透過村莊設施、交易、委託或探索取得。

## 遊戲起點

新遊戲從固定開場事件開始，之後才回到村莊廣場的自由行動狀態。

開場流程：

- 深谷耳語：用主角視角說明過去不值一提，也不願回憶；他只明白眼前的事最重要，卻不知道原因。那道耳語吸引他攀下深谷，直到他被野獸襲擊。企劃內部真相是主角身為契約外的人，能避開山谷村血脈中的瘴氣偏斜與舊儀式規則；納薇菈察覺這個變數後，以微弱力量引導他靠近山谷。
- 村中醒來：主角醒來時被山谷村的少女們圍著。她們你一言我一語討論這位陌生人，直到艾妲靠近並詢問主角名字。
- 命名主角：玩家輸入主角名字，名字寫入 save data 的 `player.name`。後續對白若角色自然呼喚主角，可使用主角名字。
- 預設名稱：主角預設名稱為「旅人」，玩家可在開場命名介面更改。
- 養傷介紹：主角在共同宿舍逐漸養傷，以主角觀點認識艾妲、米菈、賽拉、諾絲、托莉與伊蓮，並看清山谷村人手不足、資源緊張、上一代失蹤、森林危險等處境。

開場結束時，主角意識到自己受了她們照顧，應該找點事情做來報答她們的恩情。故事正式回到村莊廣場，玩家可自由移動、等待、打開道具選單、找村民互動、操作設施或前往森林外圍。

## 遺跡呼喚

遺跡會透過夢境與耳語吸引主角。呼喚主角的根本理由不是主角天生特殊，而是他來自山谷外，不屬於山谷村與山谷之神締結的舊契約。山谷自古存在瘴氣，在秩序正常時可以被神職與祭司制度調和；失衡後，瘴氣主要影響山谷內部出生、森林與遺跡周邊。主角因此成為契約外的變數，能碰觸某些村民難以接近、或會被瘴氣干涉的線索。

納薇菈那一方會察覺主角這個變數，並用殘存力量把他推向山谷。開場時玩家只會感覺那是來自遺跡深處的耳語；後續主線可以逐步揭露，耳語中混有兩股力量：一股來自瑟菈彌，像是在確認誰能承擔失衡的規則；另一股來自被封印的納薇菈，像是在引導主角找到打破契約的方法。主角不是天選之人，也不是活祭候選，而是被納薇菈賭上的契約外來者。

- 夢境：在夜晚或休息後出現，內容偏象徵性，例如倒塌的祭壇、沒有臉的調查隊、被黑霧覆蓋的搖籃、呼吸般發亮的石門、盛滿穀物卻染著血的祭壇。
- 耳語：在靠近遺跡、閱讀禁忌記錄、做出關鍵選擇時出現，語句短而模糊，像是在引導主角前進，也像是在測試他是否理解「維持規則必須有人承擔」。
- 功能：夢境提供長線謎團與下一個探索方向；耳語提供當下壓力、警告或誘惑。
- 風險：玩家若過度相信耳語，可能更快接近真相，也可能觸發更危險的事件。
- 村民反應：伊蓮會認為這是遺跡選中主角的證據；賽拉則會懷疑主角可能被瘴氣、遺跡或山谷之神利用。

## 世界觀背景

山谷村是一座位於封閉山谷中的小村，四周被山谷森林、險峻地形與古代遺跡影響包圍，長年與外界往來極少。遺跡深處沉睡著被稱為山谷之神的古老存在。更深層的背景中，山谷之神其實是姊妹孿生神：納薇菈，承穢與豐饒之神；瑟菈彌，契約與循環之神。村人後世多半只記得山谷之神這個模糊稱呼，已經沒有人知道森林裡的黑貓就是當年的豐饒之神納薇菈。

納薇菈司掌承穢、豐饒、庇護與災厄吸納。山谷自古就有瘴氣，正常情況下並不等同於詛咒，而是需要被承接、沉澱、調和的山谷污穢。納薇菈負責承接山谷的病氣、死氣、飢荒、污穢、瘴氣與黑暗，再把它們轉化成水源、作物、庇護與生命延續；也正因為她的職責總是接觸最黑暗的部分，後世村人逐漸誤以為她是帶來災厄的邪神。瑟菈彌司掌契約、循環、償還與秩序。她維持的是山谷運作的規則，不是自己的私欲；村人祭祀的本質也不是滿足神的需求，而是維持這套規則能繼續運轉。

很久以前，山谷村曾以巨大的代價向山谷之神換取富饒，讓貧瘠山谷得到穩定收成、水源與庇護。這份富饒不是憑空產生，而是兩位神與神殿祭司團共同維持秩序，讓瘴氣被處理、被調和，最後不再直接傷害山谷生命。當年的村人沒有正確理解兩位神的分工，只把神恩、償還、瘴氣與災厄都混成同一套恐懼。

神殿祭司團祀奉兩位女神，實際職責是協助承接、記錄、分流與調和瘴氣，並讓山谷的契約與循環能持續運轉。祭司團需要永續傳承，因此山谷村每十年必須補充一名新的祭司。這名祭司不是被殺害的祭品，而是進入神殿成為祭司團的一員，接續前人的知識、儀式與瘴氣調和工作。

後來村人因誤解而封印納薇菈，這才是山谷村真正的罪孽。封印納薇菈的同一場動亂中，祀奉她的女神像被完全搗毀，神殿祭司團也被全數殺害；從那一刻起，山谷同時失去承穢與豐饒的神職、祭司團的技術傳承，以及調和瘴氣的日常秩序。女神像被毀是納薇菈失去神力的主因，她已經沒有能穩定承載神格與神職的依憑。唯一留下來的，是一名天真的神殿祭司女孩曾送給神明的黑貓布偶；納薇菈只能藉由這個被愛意留下的小東西，以黑貓姿態勉強顯現，但她幾乎已經什麼都辦不到。納薇菈無法回到瑟菈彌身邊後，山谷失去承穢與庇護的神職平衡；瑟菈彌則陷入失去納薇菈的歇斯底里，仍試圖維持契約與循環，卻在長年接觸失控瘴氣後逐漸被侵蝕，失去原本的理智。

這份富饒並不是無償恩惠，但真正需要的不是活人祭品。神殿每十年需要一名新祭司入殿，補足祭司團的傳承斷層，協助維持山谷的契約、循環與承穢規則；因為某種尚未揭露的神殿限制，祭司一旦正式入殿，就必須與外界斷絕聯繫，不能回到村中生活，也不能向村人解釋神殿內部發生了什麼。對村人來說，進入神殿的人像是從此被山谷取走，才逐漸被稱為「活祭」。所謂活祭是多年恐懼、誤解與掌權者包裝後留下的謊言，不是兩位神真正索取人命。

後來村莊因恐懼、內部分裂或試圖擺脫神殿責任而停止送出祭司，也有人刻意把神殿祭司制度說成殺人儀式，藉此煽動村人背離神殿。山谷失去祭司團維護後，契約與循環開始失衡；納薇菈又已被誤封，無法承接外溢的瘴氣、污穢與災厄。從上一代開始，村中再也沒有男孩出生，夜晚黑影也逐漸肆虐。所謂詛咒的表面規則，目前被認為只影響村中出生，不會改變山谷外來者的性別，也不會讓其他地區立刻出現相同異常；其本質其實是瘴氣在失衡秩序下對山谷生命循環造成的偏斜。起初村民以為只是偶然，直到連續多年所有新生兒皆為女孩，村莊才意識到問題與遺跡深處的神殿規則有關。

為了尋找調和瘴氣與恢復山谷平衡的方法，上一代的成年人與守護者組成調查隊前往古代遺跡。調查隊包含出生偏斜明顯惡化前已成年的男性、年長者、工匠與熟悉傳說的人。他們想確認山谷之神是否真的存在、所謂活祭是否只是神殿祭司制度被扭曲後的謊言，以及是否有不用讓祭司與外界永久斷絕也能維持山谷平衡的方法。他們帶走了村莊大部分經驗與戰力，卻再也沒有回來。

調查隊失蹤後，山谷村並不是立刻只剩六名年輕女孩。仍有極少數上一代成員曾經留下來，勉強維持村莊交接、照護、記錄與日常秩序。最後一位仍在村中活動的上一代成員，已在故事開始約一年前死亡或消失；對現在的六名女孩來說，這一年才是「真的沒有人能再替她們承擔」的開始。這個時間點要成為村莊情緒的重要錨點：許多床位、飯碗、工具、巡邏路線、未寫完的記錄與沒有收好的私人物品都還留著，但使用它們的人已經不在。

伊蓮一系保存的舊資料並不只是傳說，也包含歷代學者對山谷失衡的研究、修補紀錄與懺悔。這些學者未必知道完整真相，但他們隱約明白山谷有些地方「位置錯了」：被封印的神、斷絕的祭司團、被誤稱為活祭的制度、失控的瘴氣，以及被迫離散的人與記憶。伊蓮因此知道山谷需要修補；這個修補不是償還罪孽，也不是讓村子用苦難抵消過錯，而是讓失去的一切回到原本應在的位置，使瘴氣、神職、祭司傳承與村莊生活重新取得平衡。

現在山谷村只剩六位年輕女性守著家園。她們不是村莊裡「比較重要的六個人」，而是山谷村最後還能站在日常裡互相支撐的六位成員。故事定位上，六位女孩全都是女主角，不是主角身邊的功能配角。她們不是無力者，而是被迫提早承擔村莊運作、防衛、農務、修繕與照護的人；伴隨主角冒險時，每個人都必須用自己的職責推動關鍵選擇、準備、線索或後果。主角的到來打破了村莊長久以來沒有男性、也幾乎沒有外人進入的現狀，讓村民重新面對瘴氣失衡、遺跡與上一代失蹤的真相。村莊場景需要保留上一代人的生活痕跡，讓玩家感覺這裡不是被抽象設定「縮小」的村莊，而是一個曾經有更多人生活、工作、爭吵、留下東西，最後才慢慢空下來的地方。

村莊場景敘事必須永遠記得目前總人口只有這六名女孩。除非明確描寫過去、回憶、夢境或上一代留下的痕跡，不得把山谷村寫成仍有普通村民群眾、路人、人潮、熱鬧市集、很多人圍觀、街上人聲鼎沸或「村裡大家都在場」的狀態。廣場、宿舍、藥草棚、工坊倉庫、開墾區與墓地的日常氛圍應以空位、回音、被繼續維護的舊物、少數人在不同地點分頭工作、遠處只聽見一兩個熟悉聲音來表現。可以使用「大家」指六位女孩這個小團體，但文意必須清楚不是泛指大量村民。若場景需要熱度，應來自六位女孩努力維持生活節奏的痕跡，而不是人口很多造成的熱鬧。

整體敘事基調必須同時保留悲傷、堅強與苦中作樂。山谷村不是熱鬧村莊縮小版，而是幾乎走到盡頭後仍努力維持早飯、修繕、巡邏、看板留言和小爭執的地方。六名女孩的核心態度不是相信未來一定會好，而是已經知道痛苦太多、傳承斷裂、村莊注定逐漸走向衰亡，仍選擇在此時此刻勉強歡笑，盡最後的力氣留下一點痕跡，像是在最後時刻替孤單的山谷點綴一點光。角色對話不應只寫成輕鬆日常；即使是笑鬧，也要能感覺到她們在用玩笑替恐懼留出呼吸空間。個性較淘氣、活潑或嘴硬的女孩，可以更明顯地用玩笑、吐槽、惡作劇、逞強或裝作沒事來表現「苦中作樂」，但底色仍是她們知道村莊正在變少，卻還是不肯放手。

## 禁忌之書主線

主線需要安排玩家尋找四本禁忌之書。四本書先作為真相揭露的階段節點，取得地點、守護條件、閱讀代價、關聯角色與具體委託流程後續再規劃。

世界觀與主線結構採取開放世界沙盒方向。玩家取得禁忌之書的委託順序、探索路線與觸發方式可以是非固定的，但實際獲得與揭露的禁忌之書一定按照以下四本的順序推進。處理規則很簡單：不管玩家用哪條路線解決禁忌之書相關委託，第一次完成相關委託就取得第一本，第二次完成相關委託就取得第二本，依此類推。也就是說，委託入口可以自由，揭露內容由 save data 內的 `mainStory.forbiddenBooks.nextRevealIndex` 決定。

- 第一本禁忌之書：揭露山谷之神曾為山谷村帶來富饒。
- 第二本禁忌之書：揭露活祭是謊言；神殿每十年真正需要的是一名與外界斷絕聯繫、加入祭司團並維持山谷規則的新祭司。
- 第三本禁忌之書：揭露詛咒本質是瘴氣失調；黑影、出生異常與山谷瘋狂化都來自失控瘴氣侵蝕。
- 第四本禁忌之書：揭露擊敗山谷之神的方法，作為後期主線與結局分歧的關鍵。

## 基礎能力

### 生命

- 代表玩家是否還能繼續冒險。
- 初始值暫定為 `10`。
- 最小值為 `0`。
- 森林探索中生命歸零時，不直接死亡，而是由森林中的神秘善良野獸暗中救回村莊。
- 生命歸零後會強制結束本次探索，身上可懲罰遺失的道具各自有 `33%` 機率消失，日期推進 1 天，並在隔天 `06:00` 醒來。
- 生命不能透過一般等待、休息或睡眠恢復。
- 生命只能透過治療相關動作恢復，例如治療藥、繃帶、米菈治療、治療事件或明確標記為治療的特殊選擇。

### 體力

- 代表玩家執行特殊行動的資源。
- 初始值暫定為 `10`。
- 最小值為 `0`。
- 體力不足時，不能執行需要體力的行動。
- 體力可透過等待、睡眠、食物、村民協助、道具或安全場景恢復。
- 等待每累積 `30` 分鐘恢復 `1` 點體力，不恢復生命；不足 `30` 分鐘的累積休息值會保留，但體力已滿時不會把額外恢復量存到之後使用。

### 攜帶重量

- 代表玩家外出時能攜帶多少裝備、消耗品與探索收穫。
- 初始最大攜帶重量暫定為 `10`。
- 出發前攜帶的老舊的劍、士兵長槍、獵弓、投石索、陷阱、火把、食物與治療道具都會占用重量。
- 探索中取得的木材、藥草、獸皮等物資也會占用重量。
- 超過最大攜帶重量時，玩家必須丟棄部分物資、停止收集，或透過向村莊移動逐步退回安全區域。
- 攜帶重量的設計目的，是讓玩家不能每次都帶滿所有解法與補給，必須在安全、收益與深入能力之間取捨。

### 技能

第一版先保留少量技能，不設計攻擊力與防禦力。

- 探索技能：影響森林不同層級的前進效率、迷路風險與探索事件判定。
- 近戰武器技能：影響老舊的劍、士兵長槍、近身壓制與徒手處理怪物的成功門檻。
- 遠程武器技能：影響獵弓、投石索、投擲物、遠距牽制與避免近身危險的成功門檻。
- 負重訓練：每級使最大負重增加 `5`，第一批來源是諾絲的工坊負重訓練委託。
- 採集技能：影響蒐集物資的效率、品質或數量。

技能不是傳統等級制戰鬥能力，而是讓玩家在沒有正確道具時，仍能以較高成本或較高風險處理危險。

## 森林探索與遭遇系統

第一版先製作森林探索區，不製作遺跡探索。遺跡保留為後續主線與高風險區域。主線規劃上，玩家必須深入三層山谷森林後才會抵達神殿遺跡；神殿遺跡不是普通探索層，而是已經滿布瘴氣的高危主線區域。

森林探索的核心體驗是膽小鬼遊戲：玩家越敢深入，能取得的獎勵越好，但生命、體力、補給、時間與本次探索物資的風險也越高。玩家需要記住怪物特性，選擇正確的攜帶物資與遭遇處理方式。

夜晚與深夜的森林會出現失控瘴氣凝成的黑影。黑影不是獨立於戰鬥之外的事件，而是夜間森林戰鬥的特殊替換結果：只要夜晚或深夜在森林中原本會發生戰鬥，就有 `33%` 機率把原本敵人替換成黑影。黑影採用一般遭遇戰流程與逃跑機制，但難度固定為 `99`，沒有一般弱點；火把是唯一有效的對應戰鬥工具，選用火把可直接擊敗黑影。黑影被擊敗後不掉落任何戰利品。夜晚或深夜位於冒險區時，主文字區塊要在場景敘事後以紅字顯示不安敘述，並且排在黑貓現身或離開資訊之前。

### 森林層級

山谷森林分為三個層級：

- 山谷森林外圍：初期區域，主要取得木材、樹脂、乾苔、石頭與少量布條，危險以小型野獸與基礎探索事件為主。
- 山谷森林中層：較危險區域，主要取得木材、月露草、石頭、銳利碎石與少量布條，會出現更明確的怪物特性與迷路事件。
- 山谷森林深處：高風險區域，主要取得月露草、乾苔、銳利碎石、石頭與特殊素材，會出現詛咒或神殿遺跡相關前兆。玩家只有穿越外圍、中層與深處三層森林後，才有機會接近神殿遺跡。

下一層不是永久解鎖地點。玩家每次從村莊出發，都必須依序穿越：

```text
山谷村 -> 山谷森林外圍 -> 山谷森林中層 -> 山谷森林深處 -> 神殿遺跡
```

`神殿遺跡` 不視為一般森林層級，也不應用普通採集收益節奏。它是三層森林之後的主線區域，內部已經滿布瘴氣；玩家進入前應需要足夠準備、線索或劇情條件。抵達遺跡前，森林深處要先透過黑影徵兆、異常植物、破碎祭司記號、呼吸不順或視野扭曲等敘事，讓玩家感覺瘴氣濃度正在上升。

### 本次探索進度與前進效率

- 探索進度只代表本次外出的暫時進展，不作為長期狀態保存。
- 每次離開探索區後，本次探索進度歸零。
- 不記錄路線掌握度或熟悉度。
- 探索技能越高，向深處移動時越容易有效前進，迷路或無效前進的機率越低。

範例：

| 狀態 | 通過成本方向 |
|---|---|
| 本次探索剛開始 | 花費時間長，體力消耗高，事件風險高 |
| 本次已找到前進路線 | 可前往下一層，但仍需消耗時間與體力 |
| 本次進展順利 | 通過成本降低，較容易避開低價值事件 |

### 探索行為

森林探索中可提供以下行為：

- 向深處移動：增加本次探索進度，也可能觸發探索事件或怪物遭遇；目前消耗 `10` 分鐘與 `1` 點體力。
- 向深處緩慢移動：增加本次探索進度 `1%`，也可能觸發探索事件或怪物遭遇；目前消耗 `10` 分鐘，不消耗體力。
- 搜尋：取得該層資源，低機率觸發怪物或危險事件。每次移動到新的探索位置後，只能搜尋一次；若該位置已經搜尋過，`搜尋` 需鎖定並顯示 `附近已經沒有物品。`
- 等待/休息一段時間：每累積 `30` 分鐘恢復 `1` 點體力，不恢復生命；在探索區等待可能提高遭遇風險。探索區的休息與一般等待視為同一個指令，只是依目前場景套用不同風險。
- 道具：可使用目前場景有直接效果的消耗品、查看目前可用道具，或進入丟棄道具流程。丟棄道具直接開啟當前場景的 `遺落的道具` 編輯介面；若地上已有遺落物，就編輯同一份地面容器，若地上尚無遺落物，則以空地面容器草稿開始。`遺落的道具` 按下確認後才真正套用取回與放下結果，並一律回到原場景；丟下的道具會暫時留在當前場景，但只保留 1 小時，期間再丟新道具會併入同一堆並重設倒數。
- 大喊：耗時 `1` 分鐘，讓當前危機感知提高 `20%`，再用提高後的危機感知檢定是否遭遇敵人；可用於主動提高遭遇機會，但不保證立刻遇敵。
- 向村莊移動：先降低目前層級的探索進度；只有當進度已到 `0%` 時，才能退回上一層；只有位於最外層且進度為 `0%` 時，指令名稱改為 `返回村莊`，並真正回到村莊。`向村莊移動` 目前消耗 `10` 分鐘與 `1` 點體力；顯示為 `返回村莊` 時只消耗 `10` 分鐘，不消耗體力。
- 向村莊緩慢移動：降低本次探索進度 `1%`；若目前層級進度已到 `0%`，則照向村莊移動的退層或回村規則處理。此指令目前消耗 `10` 分鐘，不消耗體力。
- 移動到廢棄小屋：只有在山谷森林外圍且本次探索進度介於 `45%` 到 `55%` 時出現。廢棄小屋是森林中的安全地點，停留期間不觸發森林遭遇；離開廢棄小屋時回到山谷森林外圍，並將本次探索進度固定為 `50%`。

探索區的可用指令由 `docs/data/commands/commands.json` 的 `baseCommandSets.exploration` 生成，不在每個探索場景中硬寫採集或深入選項。村莊地點與探索區使用不同指令組；共用指令如 `useItem` 也必須透過 `usableIn` 標明可執行範圍。

第一版安全探索地點：

- 廢棄小屋：位於山谷森林外圍 `45%` 到 `55%` 之間。它不是村莊地點，也不清空本次探索狀態；進入後可使用道具與等待，等待不套用探索遭遇風險。離開時固定回到山谷森林外圍 `50%`，再繼續一般探索流程。

第一版 `搜尋` 規則：

- 探索物資池資料放在 `docs/data/exploration/forage-loot.json`。
- 每次移動到新的探索位置時，重置一次 `搜尋` 權限。
- 每次移動到新的探索位置時，也從 `scenes[].sceneryDescriptions` 隨機抽一則森林景色描述，保存到 `save.exploration.sceneryText`；同一位置重繪時沿用，不固定提及陪同角色。
- 在同一個探索位置只能執行一次 `搜尋`。
- 本次找到 `0` 個物品的機率為 `20%`，找到 `1` 個物品的機率為 `50%`，找到 `2` 個物品的機率為 `30%`。
- 物品數量決定後，再從目前區域的候選物品池依權重抽出不重複物品。
- 第一版候選物品包含：樹脂、布條、乾苔、木材、月露草、石頭、銳利碎石。
- 若玩家已在該位置搜尋過，`搜尋` 需鎖定並顯示 `附近已經沒有物品。`
- 若搜尋找到物資，先進入暫時的道具整理介面。上方顯示剛找到的道具，下方顯示背包；玩家可把找到的道具放入背包，也可把背包裡的道具放到遺落道具。按下確認後才套用整理結果，沒有撿起來的物資會放進目前探索位置的 `遺落的道具`。
- 搜尋也可以揭露隱藏地點。第一版規則為：在山谷森林外圍深入程度 `10%` 時執行 `搜尋`，若尚未發現 `隱藏的洞穴`，會優先揭露洞穴並寫入旗標 `forest_hidden_cave_revealed`，不抽一般搜尋物資。
- `隱藏的洞穴` 是森林外圍內的安全地點，不觸發敵人；玩家可從森林外圍 `10%` 位置進入，離開後回到同一個森林外圍位置。
- 隱藏地點內固定物件使用一般設施狀態保存，不走探索臨時資源點狀態。洞穴內目前有 `礦脈` 與 `土堆`；礦脈每日最多提供 `礦石 x1`，離開再進來不會重複刷新，需等每日重置後才可再次取得。

第一版探索臨時資源點規則：

- `莓果叢`、`月露草叢`、`雜草叢`、`碎石堆`、`土堆` 在探索區中可作為臨時資源點出現。
- 這些探索版資源點不是村莊內的固定設施，而是每次移動時重新抽出的臨時物件。
- 每次 `前進` 後，抽取本次要出現幾個臨時資源點：`0` 個 `50%`、`1` 個 `30%`、`2` 個 `20%`。
- 每次 `後退` 後，臨時資源點總出現率折半；第一版依同樣比例換算為：`0` 個 `75%`、`1` 個 `15%`、`2` 個 `10%`。
- 一次若抽到 `2` 個資源點，可以出現兩個相同類型。
- 每次移動都重新抽取，本次出現的臨時資源點彼此獨立。
- 離開探索區後，不保留這些臨時資源點的資料；只有村莊內原本固定存在的設施保留長期狀態。
- 探索臨時資源點資料可暫存在 `save.exploration.temporaryResourceNodes`，但在成功退回村莊、探索失敗回村或其他離開探索的流程時必須清空。
- 探索區的 `遺落的道具` 有第二種清除條件：主角變換探索位置或離開探索區時立刻清除。清除時一般物品消失，裝備與委託道具依失物箱規則移入宿舍 `失物箱`。

### 簡化戰鬥

戰鬥不採用逐回合攻防，也不設計攻擊力與防禦力。戰鬥是遭遇事件中的選項判定。

玩家遇到怪物後，系統依照玩家攜帶的物資、近戰武器技能、遠程武器技能與已知情報，提供可選的處理方式。每隻怪物都有適合記憶的對應規則。近戰武器與空手硬撐不因體力不足而禁止選擇；目前只有逃跑在選擇時要求 `1` 點體力。

主角遭遇過的敵人需以隱藏存檔欄位 `player.encounteredEnemyIds` 記錄敵人 id。只要敵人已經出現在遭遇場景就算遭遇過，後續不論戰勝、險勝、失敗、大失敗或逃跑，都不再移除這筆紀錄。這個欄位不直接顯示給玩家，可供後續怪物知識、角色對話、委託或敘事解鎖判斷使用。

賽拉可透過 `尋求知識` 提供敵人與森林情報。`關於敵人` 只顯示主角已遭遇過的敵人，並以兩頁說明呈現：第一頁補足外觀與行為，第二頁提示應對方式。設計檢查順序是 `戰術物品`、`遠程應對`、`近身搏鬥`，必要時納入 `撤退`，但玩家可見文字不得顯示這些流程標籤，也不得用「我按順序說」之類的流程模板。正式敵人應優先使用手寫情報，把弱點藏在該敵人的動作、距離、身體特徵與賽拉的觀察裡；若某階段沒有好解法，也要用自然台詞說出，不可寫成欄位式報告。賽拉的提示必須含糊，不直接說出應對道具名稱；她可以描述行動方向、距離、節奏、翅膀、衝撞、貼近、火光感等線索。看過第二頁後，該敵人 id 寫入 `player.knownEnemyWeaknessIds`，代表主角已知曉弱點。之後再遭遇該敵人時，戰鬥畫面可出現 `賽拉的情報` 按鈕；這只是查閱弱點台詞的視窗，不是戰鬥行動，不改變結果、不消耗時間或道具。戰鬥中打開這個視窗時，標題只顯示敵人名，前導為 `你記得賽拉曾提及{敵人名}：`，後續引用賽拉台詞與當時情境時使用斜體呈現。

若賽拉談到 `巨獸`，第二頁情報除了提示遠程牽制方向，也要暗示撤退或拉開距離是合理選項；不要直接顯示逃跑成功率或機率數字。

遭遇畫面的搏鬥把握提示只評估目前近身作戰手段，也就是近戰武器與徒手硬撐能否達到 `險勝` 以上；遠程削弱、戰術物品、逃跑、武器選擇暗示與未來機率都不納入這個提示。若遠程命中改變敵方當前難度，搏鬥把握只依更新後的近身判定重新顯示。可能時用綠色文字顯示 `你似乎有把握在近身戰勝對方。`，不可能時用紅色文字顯示 `你目前毫無把握戰勝。`

戰鬥結果分為五個階段：

- 大勝利：選對會消耗的戰鬥工具，直接擊敗敵人。這個判定優先於一般戰鬥力比較。
- 勝利：戰鬥力大於難度。
- 險勝：戰鬥力等於難度。
- 失敗：戰鬥力減難度為 `-1` 到 `-2`。
- 大失敗：戰鬥力減難度小於或等於 `-3`，主角陣亡並套用生命歸零的撤退回村結果。

近戰判定時，近戰武器若對該敵人有效，戰鬥力 `+2`；近戰武器技能再讓近戰武器戰鬥力 `+1`。先判定大勝利工具，再比較 `戰鬥力 - 難度`：大於 `0` 為勝利，等於 `0` 為險勝，`-1` 到 `-2` 為失敗，小於或等於 `-3` 為大失敗。

近身作戰不推進遊戲時間，但按下後要播放只覆蓋指令區塊的假讀條。老舊的劍顯示 `無畏揮砍...`；士兵長槍顯示 `衝鋒突刺...`；徒手顯示 `奮力搏鬥...`。

大失敗不是額外死亡分支，只是最嚴重的戰鬥失敗。大失敗造成的陣亡等同生命歸零，會啟動探索戰敗回村事件；事件負責播放救回敘事、結束本次探索、推進到隔天清晨，並套用可遺失物品的懲罰。其他戰鬥結果若造成生命損失，也只有生命歸零時才強制結束本次探索並啟動同一事件。

五階段結果的第一版損益表：

| 結果 | 條件 | 生命 | 體力 | 戰利品 | 其他處理 |
|---|---|---:|---:|---|---|
| 大勝利 | 選對會消耗的戰鬥工具 | 0 | 0 | 取得 `loot.common`，並取得全部符合條件的 `loot.special` | 消耗該戰鬥工具 1 個，戰鬥立即結束 |
| 勝利 | 戰鬥力大於難度 | 0 | -1 | 取得 `loot.common`，`loot.special` 以一般機率判定 | 戰鬥結束 |
| 險勝 | 戰鬥力等於難度 | -1 | -1 | 取得 `loot.common`，不取得 `loot.special` | 戰鬥結束，但顯示驚險描述 |
| 失敗 | 戰鬥力減難度為 `-1` 到 `-2` | -2 | -1 | 不取得戰利品 | 敵人逃走或主角撤退，遭遇結束並停留在探索區 |
| 大失敗 | 戰鬥力減難度小於或等於 `-3` | 歸零 | 歸零 | 不取得戰利品 | 套用森林守護獸救回與本次探索失敗規則 |

`loot.special` 的一般機率第一版暫定為 `50%`，之後若需要可改成敵人個別欄位。戰鬥戰利品不在戰鬥結果頁立刻加入背包，而是先顯示本次獲得的戰利品數量；玩家從戰鬥結果頁按下 `蒐集戰利品` 後，進入類似搜尋整理的介面，才決定哪些放入背包、哪些留到探索位置的 `遺落的道具`。已消耗的彈藥、火把、陷阱、補網或其他消耗品不會因任何結果返還。若主角沒有倒下，戰鬥結束後當前危機感知降低 `30%`，最低不低於 `0%`；若險勝或失敗造成生命歸零，仍套用森林守護獸救回規則。

戰鬥結束會先進入 `戰鬥結果` 畫面，讓玩家閱讀完整結果與當下的狀態變化；若有戰利品，確認按鈕顯示為 `蒐集戰利品`，並先進入戰利品整理頁；若沒有戰利品，確認按鈕顯示為 `確認`，直接回到場景。回到場景後，本次戰鬥的狀態變化資訊不再保留在場景狀態變化區塊。戰鬥敘事與是否選對應對方式屬於主文字區塊，不屬於狀態變化區塊。戰鬥中的主文字情報必須先維持沉浸感，再承載必要判斷：敵人外觀、壓迫感、戰勝可能、遠程命中、工具是否有效、賽拉情報與弱點回想，都要寫成主角看見敵人動作後的判斷、身體感受、記憶浮現或角色曾說過的自然台詞，不得寫成 `敵人弱點`、`目前戰力`、`有效道具`、`情報來源`、`狀態` 這類欄位式資料。遭遇戰開始時，主文字區塊應先用中標題標示敵人名稱，不額外顯示 `敵人` 小標籤，再以緊湊文字顯示遭遇敘事；敵人出現、外觀、壓迫感與對峙描述可用單換行分隔，不用空行。敵人的壓迫感只作為一般情境文字，不獨立做成欄位。玩家採取戰鬥行動後，主文字區塊改顯示該行動的新資訊，不重複初次遭遇描述；遠程行動必須直接明寫 `命中` 或 `沒有命中`，其中 `命中。` 獨立顯示為綠字，`沒有命中。` 獨立顯示為紅字。若後續選擇逃跑，成功時戰鬥結果頁以綠字顯示 `逃脫成功。` 並不得保留逃跑前上一個戰鬥行動的回饋文字或顏色；失敗時留在遭遇場景，主文字區塊刷新為紅字 `逃脫失敗。` 與一般格式的追擊、受傷敘事，連續逃脫失敗時敘事必須依失敗次數變化。搏鬥把握只用主文字區塊的一般文字換色顯示，避免靜態敵人情報混入狀態變化區塊。

遠程攻擊不直接進入戰鬥結果判定，而是在進入近戰前削弱敵人。遠程攻擊命中時會讓該敵人難度 `-1`。若敵人懼怕該彈藥，命中 `+2`。當遠程武器技能加命中大於等於敵人的遠程閃避時，遠程判定成功；若命中值未達門檻，除了絕對閃避敵人外，仍有 `33%` 機率命中。玩家可在進入戰鬥前嘗試遠程削弱；若命中後敵人難度降到 `0`，或本次遭遇累計遠程命中達 `3` 次，敵人直接死亡。飛獸絕對閃避彈藥，不受遠程削弱，但玩家仍可能浪費第一發彈藥才意識到遠程牽制派不上用場。

遠程削弱規則：

- 遠程削弱嘗試本身不消耗時間。
- 每次遠程削弱都會消耗對應武器彈藥：獵弓消耗 `1` 支箭矢，投石索消耗 `1` 顆石頭。
- 遠程削弱不推進遊戲時間，但按下遠程武器後要播放假讀條。這類戰鬥中短讀條只覆蓋指令區塊，避免主文字區塊與標題閃爍。獵弓顯示 `拉弓瞄準...`；投石索顯示 `投擲投擲...`。
- 戰鬥結果結算時，本次遭遇每支射出的箭矢與每顆投出去的石頭都各自判定是否回收；基礎回收率為 `10%`，遠程武器技能每級再增加 `3%`。
- 沒有對應彈藥時，該遠程削弱選項不可使用。
- 每次遠程削弱有 `25%` 機率讓主角體力 `-1`。
- 若敵人 `absoluteAmmoEvasion: true`，遠程選項仍可執行一次，但必定失敗並消耗彈藥，之後本次遭遇不再提供遠程空隙。
- 除絕對閃避外，未達命中門檻的遠程嘗試仍有 `33%` 機率命中。命中後可繼續使用遠程牽制；未命中才關閉本次遭遇的遠程空隙。
- 同一場遭遇累計遠程命中達 `3` 次時，無論剩餘難度多少，都直接以大勝利擊敗敵人。
- 同一場遭遇中，遠程命中的主文字回饋要依累計命中次數明顯變化。第二次命中必須改寫成戰況進一步推進的敘事，例如敵人姿態、節奏或攻勢被打亂；不能只把第一擊文案的數字改成第二次。第三次命中會直接結算勝利，因此文案只需表達最後一擊讓敵人支撐不住，不再追加持續壓制或後續判斷情報。文案仍需直接寫出 `命中` 與目前是第幾次命中。
- 難度最低降到 `0`，不繼續降成負數。

戰鬥中的行動提示需要反映玩家是否選對相剋手段，但只使用簡短固定句，不追加完整戰鬥敘事，並且必須顯示在戰鬥結果正文，不得放進狀態變化區塊。狀態變化區塊只顯示生命、體力、物品、戰利品或其他實際數值變化。近身相剋顯示綠字 `使用{武器}作戰時發揮些許優勢。`，近身選錯顯示紅字 `使用{武器}作戰並不順遂。`；戰鬥工具相剋顯示 `使用{工具}應對{敵人}發揮了極大的功效。`，戰鬥工具選錯顯示 `使用{工具}應對{敵人}絲毫沒有作用。` 若遠程攻擊直接擊倒敵人，結果頁綠字短句顯示 `你用{遠程武器}完美地壓制對手。`，不得把前一步遠程命中敘事整段搬到結果頁。戰鬥結果畫面先用中標題簡單顯示 `勝利`、`險勝`、`失敗` 三種結果；大勝利併入勝利，大失敗併入失敗。結果頁不要再加 `戰鬥結果` 小標籤或 `你面對的是...` 這類遭遇對象重述。

會被消耗的戰鬥工具若選對，直接進入大勝利。目前會消耗且可直接作為戰鬥工具的道具為補網、陷阱、火把。煙霧彈是例外；它在戰鬥中使用時不進入大勝利判定，而是必定成功逃跑。戰術物品不推進時間，但按下後要播放只覆蓋指令區塊的假讀條：陷阱為 `佈置陷阱...`，火把為 `點燃火把...`，補網為 `張開補網...`，煙霧彈為 `拋出煙霧彈...`。

戰鬥工具的分類弱點固定如下：

- 飛獸：絕對怕補網，對應 `capture_net`。
- 四足獸：絕對怕陷阱，對應 `trap`。
- 雙足獸：絕對怕火把，對應 `torch`。
- 巨獸：體型過大，不套用分類必勝工具，也沒有近戰武器或陷阱弱點；牠們遠程閃避低，但難度比同地區一般敵人高 `3` 點，設計上鼓勵先用遠程削弱或累計遠程命中，而不是直接近身硬打。

黑影作為夜間瘴氣敵人時也沿用這條規則：黑影的 `difficulty` 為 `99`，`fearedCombatTools` 只有 `torch`，因此一般武器、彈藥與其他戰鬥工具幾乎無法戰勝黑影；火把命中正解時直接大勝利，但黑影的 `loot.common` 與 `loot.special` 都為空，不產生戰利品。

面對戰鬥時可選擇逃跑。逃跑不進入五種戰鬥結果判定，固定消耗體力 `-1`，不推進時間，但按下後要播放只覆蓋指令區塊的假讀條，文字為 `尋找退路...`。一般敵人有 `50%` 機率成功脫離戰鬥；巨獸體型龐大、轉向慢，逃跑成功率為 `100%`。若逃跑失敗，主角留在戰鬥中且生命 `-1`；同一場遭遇連續逃跑失敗時，主文字敘事需依失敗次數變化，不重複同一段文案。若逃跑失敗後生命歸零，套用森林守護獸救回與撤退回村結果。

### 敵人資料欄位

```json
{
  "id": "outer_forest_mist_bat",
  "name": "霧翼蝠",
  "region": "山谷森林外圍",
  "category": "飛獸",
  "difficulty": 2,
  "rangedDodge": 999,
  "absoluteAmmoEvasion": true,
  "fearedCombatTools": ["capture_net"],
  "effectiveMeleeWeapon": "無",
  "effectiveAmmo": "無"
}
```

敵人分類基礎規則：

- 飛獸：絕對閃避彈藥，`effectiveAmmo` 固定為 `無`。
- 四足獸：怕槍、箭矢。
- 雙足獸：怕劍、石頭。
- 巨獸：沒有近戰武器弱點，也不怕陷阱；遠程閃避低，通常怕箭矢，但仍需要靠遠程削弱或累計命中處理。

`effectiveMeleeWeapon` 可用值：`槍`、`劍`、`兩者`、`無`。其中 `槍` 目前對應可用道具為士兵長槍，`劍` 對應老舊的劍。

`effectiveAmmo` 可用值：`箭矢`、`石頭`、`兩者`、`無`。彈藥本身不歸類為戰鬥工具，但會影響獵弓或投石索是否可用。

### 森林敵人戰利品

三個森林區域共用骨類戰利品規則，依敵人難度決定：

- 難度 `1` 到 `3`：碎骨。
- 難度 `4` 到 `5`：獸骨。
- 難度 `6` 以上：硬獸骨。

特殊戰利品依敵人命名與外觀印象追加：

- 鳥類一律有羽毛。
- 命名看起來牙齒很鋒利的一律有獸牙。
- 命名看起來跟植物有關的一律有月露草。
- 命名看起來有角的一律有獸角。

黑影是失控瘴氣在夜間凝成的敵人，不套用骨類與特殊戰利品規則。即使以火把擊敗黑影，也不取得 `loot.common` 或 `loot.special`。

### 黑貓與森林守護獸

森林中存在一名只會在森林探索區偶然出現的極特殊 NPC，玩家可見名稱為黑貓。牠不說話，也不主動解釋自己的來意；牠只用視線、動作和突然消失的方向回應主角。每個探索位置生成時，黑貓有 `25%` 機率出現。黑貓出現時，探索場景主文字區塊要在場景敘事後追加一行綠色的現身情報，以主角看見牠的姿態和距離來表現，不寫成系統提示。牠不會進入村莊常駐名冊，也不以普通村民方式參與日常作息。六名核心村民的好感、送禮、交易、行程與委託流程不適用於黑貓；若牠暫時放在 `villagers.json` 內，只代表目前資料收納位置，不代表牠要吃一般村民運作邏輯。資料上使用 `npcType: "special_forest_npc"` 與 `socialRules` 明確排除普通好感、交談、送禮、交易與委託選單生成。特殊 NPC 的互動權限以 `docs/data/npc-interactions/npc-interaction-rules.json` 為準；若特殊 NPC 台詞資料中暫時保留 `affectionMin`，程式應忽略該條件。

黑貓的玩家可見稱號可顯示為來自森林的助手，這只是類似艾妲顯示為領袖的角色定位文字，不改變黑貓的真實設定。黑貓真身是巨大的黑豹，但牠不會主動透露。巨大的黑豹就是森林中的神秘善良野獸，也是主角在森林普通探索失敗時救助主角的存在。更深層的背景中，牠是納薇菈，承穢與豐饒之神，也是瑟菈彌的孿生姊姊。納薇菈曾因村人誤解而被背叛、封印；祀奉她的女神像被完全搗毀後，她失去大部分神力，只剩一名天真的神殿祭司女孩送給神明的黑貓布偶，讓她能以現在這副沉默黑貓的樣子勉強顯現。這不是完整化身，而是幾乎什麼都辦不到的殘影。如今已經沒有人知道黑貓就是當年的豐饒之神。背景設定中，主角初次出現在山谷村附近，也是牠先救助並把主角帶到能被村莊發現的位置。

賽拉知道黑貓存在，也明白黑貓不是威脅。她曾在森林巡守或危急撤退時受過黑貓幫助，因此她對黑貓的態度不是恐懼，而是克制的信任：她不會把黑貓說成可靠同伴，也不會鼓勵主角依賴牠，但若主角說黑貓引導或救了他，賽拉不會否定。這份認知不代表賽拉知道黑貓真身；她只知道森林裡有一個沉默存在，會在某些時候把人從不該死的地方推回來。

村裡女孩對黑貓有不同態度與好奇心，但都不應太早理解牠是納薇菈。艾妲把黑貓視為需要記錄但不能公開依賴的異常助力；米菈會好奇黑貓為什麼能靠近瘴氣與受傷的人；賽拉承認牠不是威脅，卻仍要求所有人以自己的準備活下來；諾絲會懷疑黑貓把東西送回村的方式是不是能被做成工具或路標；托莉最想親眼見牠，容易把牠當成森林裡的祕密朋友；伊蓮則會把黑貓放進舊傳說與村中目擊紀錄之間，先保留疑問，不急著定義。

這段設定與主線關聯很深，細節暫時保留，後續再規劃。

黑貓可提供三種特殊協助。請牠引開敵人注意時，牠會降低目前探索危險度 `30`，耗時 `1` 分鐘，回饋需讓玩家感覺到危機降低了一些，但不使用說話說明效果。請牠尋找臨時藏身處時，耗時 `30` 分鐘，主角體力恢復 `2`，危險度不因此改變。請牠把東西帶回村子時，畫面先進入類似倉庫的整理介面且不耗時，也不播放假讀條；玩家可把背包中的道具交給黑貓，確認整理後才播放只覆蓋指令區塊的假讀條並耗時 `1` 分鐘，這些道具會出現在宿舍失物箱中。第一次成功請黑貓帶回物品後，玩家回到村莊廣場時會觸發艾妲事件；不論艾妲當時是否按排班在廣場，她都會告知主角那些物品似乎被放在村口附近，已經被她收到宿舍失物箱，讓玩家理解黑貓帶回的物品會出現在失物箱。若玩家從整理介面返回，只取消草稿並回到黑貓三個特殊行動選項，不觸發黑貓離開。所有黑貓特殊行動都需要留下情境回饋。玩家可見提示必須用主角的自然感受暗示這件事，不要直白說明內部容器轉移規則。黑貓任一特殊協助完成後就會從目前探索位置消失；同一探索位置不重複出現，玩家移動到新的探索位置時才重新依出現率判定。黑貓特殊行動完成後必須回到原本探索場景，並在主文字區塊用紅色沉浸文字提示黑貓離開；帶東西回村只有在確認整理後才算完成並顯示黑貓離開。

主角第一次在森林探索區看見黑貓時，若黑貓已被該探索位置的出現率抽中，必須立即觸發初見事件，不需要玩家主動互動。事件描述應讓主角觀察到黑貓沒有敵意、姿態友善，甚至像是在用動作表示願意幫忙；但這份理解可以是主角單方面的解讀，不需要黑貓開口或直接證明。

森林普通探索中，生命歸零或戰鬥大失敗造成的陣亡都不進入死亡結局。主角會被黑貓真身，也就是巨大的黑豹，暗中救回村莊，並套用同一套撤退回村結果。

生命歸零時固定處理：

- 強制結束本次探索。
- 身上每件可懲罰遺失的道具各自有 `33%` 機率消失。
- 裝備、委託物品、主線重要道具與線索型道具不會因森林失敗懲罰消失。
- 已使用的消耗品不返還。
- 日期推進 1 天。
- 在隔天清晨 `06:00` 醒來。
- 醒來時必然有一名核心女孩正在照顧主角，照顧者從艾妲、米菈、諾絲、賽拉、托莉、伊蓮中隨機決定。
- 醒來後生命恢復到 `3`，體力恢復到最大值。
- 戰敗回村分成兩段事件：第一段是主角昏迷時渾渾噩噩感知到自己被某種大型生物帶走的共通描述；第二段是回到宿舍後，依 `save.events.lastExplorationDefeatCaregiverId` 直接啟動對應女孩的照護事件，播放 2 到 3 頁專屬關切與照護對白。
- 照顧者 id 屬於這次敗退事件的結果狀態，暫存在 `save.events.lastExplorationDefeatCaregiverId`。照顧者抽選使用一輪不重複規則，近期已抽過的角色暫存在 `save.events.explorationDefeatCaregiverHistory`，六名候選都輪過後再重新開始。照護事件優先於宿舍其他 `sceneEnter` 事件；照護事件結束後才回到宿舍並允許其他宿舍事件判定。
- 保留玩家已獲得的怪物情報或解法記憶。

森林守護獸的真身在第一版中先作為伏筆，不需要立刻揭露。玩家可透過黑貓的無言行動、生命歸零事件、巨大足跡、草藥痕跡與村民反應，逐步察覺牠和救助事件之間的關聯。

## 時間系統

遊戲採用 24 小時制。時間是核心資源，所有行動都必須有時間成本。

### 時間精度

- 顯示格式：`HH:mm`。
- 玩家可見時間不顯示秒。
- 一天長度：`24:00`。
- 日期從第 1 天開始計算。
- 時間超過 `23:59:59` 後進入下一天。

### 時間成本

每個行動都需要定義 `timeCostSeconds`。不同類型行動的時間成本不同。

範例：

- 簡短對話：`300` 秒，等於 5 分鐘。
- 深入交談：`1800` 秒，等於 30 分鐘。
- 查看工坊倉庫：`120` 秒，等於 2 分鐘。
- 修補柵欄：`3600` 秒，等於 1 小時。
- 從山谷村走到森林外圍：`600` 秒，等於 10 分鐘。
- 採集木材：`1800` 秒，等於 30 分鐘，並消耗 `1` 點體力。
- 短暫等待/休息：`1800` 秒，等於 30 分鐘。
- 睡眠：依照起床時間計算，通常推進到隔天早晨。

### 時段

系統可依目前時間判定時段：

- 清晨：`05:00:00` 到 `07:59:59`
- 上午：`08:00:00` 到 `11:59:59`
- 中午：`12:00:00` 到 `13:59:59`
- 下午：`14:00:00` 到 `17:59:59`
- 傍晚：`18:00:00` 到 `19:59:59`
- 夜晚：`20:00:00` 到 `23:59:59`
- 深夜：`00:00:00` 到 `04:59:59`

### 時間影響

時間會影響以下內容：

- 村民位置：不同時段村民會在不同地點。
- 可選行動：部分行動只在特定時間開放。
- 探險風險：夜晚與深夜外出風險提高。
- 體力管理：長時間活動可能需要休息或睡眠。
- 村莊事件：某些事件會在指定時間或天數觸發。

### 每日重置

每日重置點固定為 `06:00`。當時間推進跨過隔天 `06:00`，或森林失敗救回後直接醒在隔天 `06:00`，系統執行同一套每日重置流程。

每日重置包含：

- 角色交易庫存依 `dailyRestock` 補貨，最多補到 `stock.max`。
- 特殊商人的交易庫存依其資料規則補貨。
- 產物暫存型設施每日生成可取得產物，例如蘋果樹、莓果叢、雜草叢、補給箱。
- 看板每日留言重新抽取。
- 每日事件旗標重置。

每日重置不會改寫 `docs/data` 企劃資料；實際已購買庫存、產物暫存內容、已生成狀態與每日旗標都屬於存檔內容。

每日重置的保存與處理欄位放在 save data：

- `time.dailyResetSecond`：固定 `21600`，代表 `06:00`。
- `time.lastDailyResetDay`：最近已完成每日重置的日數。
- `dailyReset.lastProcessedDay`：每日重置流程最後處理到的日數。
- `dailyReset.lastProcessedAt`：最後一次處理每日重置時的 `day` 與 `secondsOfDay`。
- `dailyReset.pending`：時間推進跨過重置點但尚未完成處理時暫存為 `true`。
- `player.dailyFlags`：只在當日有效的玩家旗標，每日重置時清空；社交限制也放在這裡，例如 `chat_affection_gained:{villagerId}` 與 `gift_given:{villagerId}`。
- `tradeStocks[characterId][itemId]`：角色或特殊商人的當前庫存，依 `dailyRestock` 補到 `stock.max`。
- `facilities[facilityId].lastRestockDay` / `items`：產物暫存型設施用，`items` 是目前仍留在該物件上、可被玩家取得的產物。
- `facilities.notice_board.lastMessageRefreshDay` / `messageIdsToday`：看板每日留言用。

每日重置優先序：

1. 若目前時間推進跨過 `06:00`，先判斷 `time.lastDailyResetDay` 是否小於目標日。
2. 更新交易庫存，普通角色依 `villagers[].carriedItems[].stock.dailyRestock` 補貨，特殊商人依 `tradeRules` 與 `carriedItems` 補貨。
3. 依 `smallStorage.restockRule` 生成產物暫存型設施的可取得產物，例如蘋果樹、莓果叢、雜草叢、補給箱。若規則有 `onlyRestockWhenTotalBelow`，需先計算該物件目前保留的道具總數，含堆疊數量。
4. 重新抽取看板每日留言。
5. 清空 `player.dailyFlags`，並清理只屬於當日的事件旗標。
6. 寫入 `time.lastDailyResetDay`、`dailyReset.lastProcessedDay` 與 `dailyReset.lastProcessedAt`，最後把 `dailyReset.pending` 設回 `false`。

場景描述：同一地點會依時段呈現不同文字。

### 時間設計原則

- 所有選項都要顯示或可查詢時間成本。
- 時間成本越高，行動收益或劇情資訊通常也要更高。
- 秒級精度用於資料與系統計算；介面可視需求顯示到分鐘或秒。
- 玩家可以規劃一天的行程，但不需要每秒手動操作。

## 村莊系統

村莊是玩家的主要基地。村中核心成員皆為年輕女性，只有主角是男性。玩家不只是為自己生存，也要讓村莊變得更安全、更富足。

### 村莊主要危機

村莊目前最大的危機是失控瘴氣造成的出生偏斜、夜間黑影，以及上一代成年人與守護者前往遺跡後不歸。

距離故事開始約一年前，村中最後一位仍在日常裡活動的上一代成員死亡或消失。自那之後，六名女孩才真正成為山谷村全部可見的人手。這件事不只是背景數字，而是所有村莊場景的生活痕跡來源：多出來的碗、沒人再穿的外衣、空床位、沒人敢丟掉的工具、舊巡邏表、被繼續維護卻不再有人使用的工作角落，都應該在敘事中自然存在。

這個危機不只代表人數問題，也會影響：

- 重勞務：修橋、伐木、搬運石材與修復設施變得困難。
- 防衛：村外野獸、盜匪或外部勢力會認為村莊容易被欺壓。
- 探險：多數村民需要維持村莊日常，難以長時間離村犯險。
- 繼承與外界壓力：鄰近村鎮可能質疑村莊能否延續與自治。
- 內部接納：村民需要判斷主角究竟是可靠的同伴，還是新的風險。
- 真相壓力：上一代調查隊失蹤的原因仍未確認，神殿遺跡位於三層山谷森林之後，內部已經滿布瘴氣，可能藏著山谷之神、神殿祭司制度與瘴氣失控來源，也可能藏著更危險的秘密。
- 情感傷痕：留下來的女孩們大多失去了父母、師長或親人，因此對遺跡既恐懼又放不下。
- 異常徵兆：瘴氣失衡表面上只影響村中出生與夜間森林，但神殿遺跡能隔著三層森林吸引主角靠近，代表山谷之神、瘴氣或遺跡本身可能還有尚未揭露的規則。

### 上一代人的痕跡

山谷村需要在場景、物品與角色對話中持續保留上一代人的痕跡。這些痕跡不是單純裝飾，而是讓玩家理解「村莊曾經更完整」的證據。四位已確認亡故的上一代角色詳見 `docs/previous-generation-characters.md`，該文件負責記錄她們/他們與六名少女的關係、留下的生活痕跡與可演出用途。

- 宿舍：空床位、舊名字、沒有拆掉的床簾、仍被擦拭的櫃子、最後一位上一代成員留下的生活用品。
- 工坊倉庫：上一代工匠留下的工具、未完成的修補件、諾絲仍在使用但不完全理解來歷的標記。
- 藥草棚：舊藥方、上一代照護者的筆記、米菈仍照著整理的藥瓶位置。
- 村莊廣場：過去村會使用的長桌、過多的椅子、已經不會被全部坐滿的飯碗與看板留言規則。
- 開墾區：曾經由更多人照看的田埂、舊水渠、還能看出分工痕跡的工具架。
- 墓地與巡邏路線：死亡者與失蹤者不能全部混成背景。某些名字有墓，某些名字沒有墓，這會讓「死亡」與「消失」在情感上保持差異。

敘事上，村民不需要每次都直接說明誰死了或誰失蹤了。更好的做法是讓她們自然使用上一代留下的物品、避開某個名字、維持某個已經沒人要求的習慣，或在主角碰到某件東西時短暫沉默。

### 村莊物資

第一版不使用獨立的村莊資源機制。木材、食物、藥草、礦石等都以道具或委託狀態表示，詳細資料放在 `docs/data/items/items.json`。

若村莊建設、設施升級或委託需要材料，直接檢查玩家背包、倉庫箱或指定容器中的道具數量，不再使用 `villageResources` 或 `village.resources` 這類共享資源欄位。

## 資料分層原則

為了降低模型讀取與生成時的 token 消耗，資料必須拆成可引用的小表，不把同一份規則重複寫進每個場景或角色。

- `docs/data/commands/commands.json`：全域指令、預設時間成本與可重用指令組。
- `docs/data/locations/locations.json`：村內自由移動地點、出口、地點基礎指令與地點特殊行動。
- `docs/data/relationships/relationships.json`：角色關係摘要，以角色配對記錄，不重複塞進每個角色。
- `docs/data/villagers/villagers.json`：角色基本資料、喜好、需求、委託鉤子、解鎖內容與時段位置。
- `docs/data/scenes/scenes.json`：只放劇情、探索與對話場景，不放可由地點資料生成的標準地點場景。
- `docs/data/dialogues/dialogues.json`：條件式對話片段。
- `docs/data/quests/quests.json`：委託資料。
- `docs/data/items/items.json`：道具資料。
- `docs/data/enemies/enemies.json`：森林敵人資料、分類、難度、遠程閃避與武器克制規則。
- `docs/data/facilities/facilities.json`：地點內可互動設施與物件，例如莓果叢、雜草叢、廢料堆、鴉巢、倉庫箱與看板；也包含探索資源點模板，例如月露草叢。
- `docs/data/save/save-template.json`：`localStorage` 手動存檔與匯出/匯入 JSON 的範本，記錄玩家狀態、設施狀態、交易庫存、每日重置、委託進度與主線揭露進度。
- 未來若森林探索資料擴大，可新增森林區域與探索事件資料檔；第一版也可以先把少量探索事件放在 `scenes.json` 驗證玩法。

資料撰寫規則：

- JSON 資料檔是可執行數值與完整資料的權威來源；Markdown 企劃文件負責記錄設計意圖、規則方向與資料格式。若兩邊出現重量、時間、庫存、恢復量、資料筆數等差異，以 JSON 為準，並回頭修正文檔摘要。
- 場景若只是固定地點，例如宿舍、藥草棚、工坊倉庫、開墾區，應放在 `locations.json`。
- 地點共用指令以 `docs/data/commands/commands.json` 為準，應透過 `baseCommandSet` 生成，不在每個場景重寫。目前村莊地點的 `standardLocation` 只包含 `useItem` 與 `wait`。
- 指令必須用 `usableIn` 標明可執行地點分類，例如 `villageLocation`、`exploration`、`dialogue` 或 `specialMerchant`。村莊地點與探索區不可共用同一批預設行動，除非該指令明確標記可在兩邊執行。
- 地點中的可互動清單應由資料生成，不硬寫在每個地點場景。清單包含目前地點的設施與目前在場角色；設施來自 `facilities.json`，以 `facility.locationIds` 匹配目前 `locations[].id` 或探索 scene id；角色由目前時間、角色行程與 `allowPresentVillagers` 生成。
- 跨資料表引用優先使用英文 id。顯示用中文只放在 `name`、`label`、`location` 或敘述文字；程式判斷用欄位使用 `locationIds`、`targetVillagerId`、`returnSceneId`、`facilityId`、`itemId` 等 id 欄位。
- 對話場景只作為進入角色互動的資料入口，不硬寫 `chat`、`gift`、`trade`、`quest`、`leave` 舊選項；實際選單由 `baseCommandSet: "dialogueMenu"`、角色 `npcType`、交易資料、委託前置條件與特殊規則生成。
- 角色關係只寫在 `relationships.json`，角色檔不重複保存完整關係矩陣。
- 企劃文件只寫規則、決策與資料格式，不複製完整資料內容。

## 道具系統

道具資料以 JSON 管理，放在 `docs/data/items/items.json`。

第一版道具類型：

- `consumable`：可消耗道具，例如乾糧、小治療藥、強效繃帶。
- `equipment`：可攜帶裝備，例如老舊的劍、士兵長槍、獵弓、投石索。
- `tool`：探索或地形處理工具，例如繩索。
- `material`：製作或委託材料，例如月露草。
- `resource`：可攜帶的建設或製作材料，例如木材、礦石；不是獨立村莊資源系統。
- `currency`：交換用貨幣，例如螢石。
- `key_material`：主線或遺跡進度材料，例如禁忌記錄、神殿鑰記或其他明確設計過的進度物。
- `quest_item`：角色委託道具，例如舊工具箱。
- `clue`：線索型道具，用於保存已明確設計用途的調查線索。

螢石是山谷村的通用貨幣，只用於交換與交易，不可作為禮物。

每個道具都需要有 `price` 欄位，價格以螢石為單位。可賣給商人的道具使用 `price.sellable: true` 與 `price.amount`；非賣品使用 `price.sellable: false`，`price.amount` 可為 `null`。無法在探索、採集、掉落、委託獎勵或一般交易中取得的故事重要道具，通常設定為非賣品。裝備、委託物品、主線重要材料與線索型道具預設不賣。

每個道具都需要有 `canGift` 欄位，決定是否可作為送禮道具。貨幣、彈藥、裝備、委託道具、線索、主線材料、會卡住委託或探索流程的補給與工具，預設不可送禮。第一版可送禮道具以 `items[].canGift` 為準，目前包含莓果、蘋果、生菜、花朵、礦石、獸角與骨系戰利品等。命中角色喜好的禮物使好感 `+3`，其他可送禮物預設 `+1`。

凡是新增的可食用道具，若尚未另外設計特殊效果，臨時預設效果一律為體力 `+1`。

森林失敗懲罰可用道具屬性表示，建議欄位為 `canDisappearOnExplorationFailure`。一般消耗品、材料、建設材料與貨幣可設為 `true`；裝備、委託物品、主線重要道具與線索型道具設為 `false`。若可遺失道具不應永久消失，可加上 `explorationFailureLossTarget: "lost_and_found_box"`，表示探索失敗抽中遺失時改送入宿舍失物箱；空藥水瓶使用這個規則。

## 設施系統

設施資料以 JSON 管理，放在 `docs/data/facilities/facilities.json`。設施不是可自由移動的地點，而是存在於地點內的可互動物件或功能點。

第一版設施類型：

- `small_storage`：程式運行用的產物暫存設施，會依規則保存已生成但尚未被玩家取得的道具；玩家只能取得，不能放入。例如蘋果樹、莓果叢、雜草叢、廢料堆、鴉巢、礦脈、補給箱與一次性舊箱子。探索資源點模板如月露草叢也可沿用此結構。玩家介面不得把自然物件描述成倉庫。
- 藥草棚新增固定設施 `醫藥補給箱`，每天補入 `1` 個月露草；若一直未取，最多累積 `3` 個。
- `infinite_source`：無限來源設施，不保存產物、不每日補貨，每次互動直接生成固定道具並加入背包，例如探索中的土堆可花 `30` 分鐘取得 `soil x1`，工坊倉庫木材堆可花 `1800` 秒進行一次劈材、消耗 `1` 點體力並取得 `wood x1`。每次取得仍需檢查玩家背包負重。
- `container`：可收納、取出或交換物品的容器，例如倉庫箱。
- `information`：提供公告、情報或委託資訊的物件，例如看板。
- `upgradeable`：有等級並會影響其他功能的設施，例如田地。第一版不從設施本身提供直接升級指令，等級提升改由委託獎勵處理。
- `crafting`：可用來製作物品的設施，例如製藥台。
- `special_access`：特殊入口或解鎖通道，例如通往蜥蜴羅索藏身處的井。

設施可掛在一個或多個地點上，也可進一步附屬在其他設施下。設施掛載使用 `locationIds`，內容必須對應 `locations.json` 的 `id`，例如田地位於 `reclamation_area`，倉庫箱位於 `workshop_storage`。若某個設施需要自己的出入口與自由移動流程，才升格為 `locations.json` 內的地點。

凡是能讓玩家取得道具的設施，第一版統一用 `small_storage` 作為程式暫存模型：設施自身保存 `items`，每日依規則生成可取得產物，或以 `smallStorage.initialItems` 指定一次性初始內容。一次性寶箱取完後寫入 `smallStorage.oneTimeClaimFlag` 對應的玩家旗標，之後不再重新生成內容。玩家只能取得，不能把道具放回該設施。這是內部資料模型，不是玩家可見的物件設定；蘋果樹仍是蘋果樹，玩家行動是摘取，不是開倉庫。這類已生成產物與剩餘內容屬於 save data，不寫回 `facilities.json`。

`舊箱子` 不是宿舍專屬物件，而是一種可重複配置的一次性寶箱規則。任何地點都可以新增自己的舊箱子實例，內容由該設施的 `smallStorage.initialItems` 指定，領取狀態由該設施自己的 `smallStorage.oneTimeClaimFlag` 記錄。若該舊箱子設定 `smallStorage.withdrawAllAtOnce: true`，畫面只提供一個 `取出` 指令；執行時巡迴箱內所有物品，主角負重可承受的加入背包，無法承受且設定 `smallStorage.overflowToDroppedItems: true` 的物品留到當前地點的 `遺落的道具`，並在結果中告知無法全部拾取。

目前舊箱子實例：

- 宿舍舊箱子：初始內容由資料表指定為 `投石索 x1` 與 `石頭 x2`，領取狀態用 `dormitory_old_chest_claimed` 記錄。投石索不再由 NPC 販售。
- 墓地舊箱子：初始內容由資料表指定為 `煙霧彈 x3`，領取狀態用 `graveyard_old_chest_claimed` 記錄。

墓地固定存在四塊上一代墓碑，資料放在 `docs/data/facilities/facilities.json`：

- `老舊的墓碑`：對應羅恩。早期只寫巡路人與夜間危險，不直接揭露他是艾妲父親、為救賽拉被黑影殺死，也不直接揭露士兵長槍關聯。
- `風化的墓碑`：對應葛倫。銘文由托莉撰寫，重點是水渠、田埂與開墾區記號。
- `生苔的墓碑`：對應艾芙拉。碑面文字由米菈留下，互動時應以主角停在墓前、看見字跡與苔痕的方式讀出她留下的話；墓碑邊每天可取得 `乾苔 x1`。
- `嶄新的墓碑`：對應瑪蓮。這是最後一位死亡的上一代成員之墓，銘文由伊蓮撰寫。

墓碑文字必須保持玩家視角可讀，不要把所有隱藏真相一次寫出來。銘文撰文者皆為村莊現存女孩，應能反映她們與亡者的關係。

玩家可見敘事文字的準則：玩家可見敘事不是資料卡，玩家不應在一般主文字、道具說明、角色台詞、事件正文或知識對話中看到 `墓碑資訊`、`銘文`、`撰文者`、`用途`、`狀態`、`來源`、`管理者`、`效果`、`取得方式`、`角色定位`、`好感需求` 這類欄位式項目。這種列項會讓物件、道具與人物從主角的觀察經驗變成系統整理好的資料檔案，削弱沉浸感。正式文案應以主角觀點或角色自然語氣寫成連續敘事：場景物件先描寫材質、磨損、位置、擺放方式、使用痕跡、附近聲音、氣味、光線或主角停步時的感受；道具先描寫重量、質地、氣味、磨損、包紮方式、可食用狀態、手感或使用聯想；角色台詞先落在她此刻看見的東西、手上的動作、所在位置、過去經驗或對主角的反應，再自然說出資訊。需要保留的資訊可以藏在句子裡，例如「碑上只留下羅恩的名字與巡路人的稱呼」、「最後幾個字像是艾妲用很穩的手刻下的」、「木箱邊緣被諾絲常用的木炭記號標過」、「藥瓶排列方式讓你想起米菈平常整理傷藥的習慣」、「伊蓮把螢石推近時順口提到它在村裡常被拿來換小東西」，而不是分行列出。只有背包、倉庫、交易、製作、Debug、狀態摘要、狀態變化區塊、可用按鈕列表或其他明確操作介面可以使用列項。墓碑、看板、舊箱子、床、工作台、藥草棚器具、田地、水渠、路標、遺跡痕跡、上一代留下的舊物、道具說明與角色台詞，都應優先使用敘事式描述。

墓碑是這條準則的高優先例子。早期墓碑文案只能讓玩家感覺到上一代留下的空位、女孩們仍在維護這些空位，以及有些真相被沉默保護；不得用完整死亡原因、關係表或功能說明取代敘事。

宿舍有一只 `失物箱`，作為重要物品的找回設施。當 `遺落的道具` 滿 `1` 小時被清除時，普通物品照常消失；分類為 `equipment` 或 `quest_item` 的物品會移入 `失物箱` 保存，玩家之後可從宿舍取回。玩家可以把背包道具暫時放入當前場景的 `遺落的道具`，但放入後同樣受 `1` 小時消失與探索換位清除規則影響；`失物箱` 不提供玩家手動放入，只接收這類過期遺落物。倉庫、丟棄道具與 `遺落的道具` 都採同一套整理草稿模式：每次只移動 `1` 個堆疊數量，按下確認才一次套用並回報結果，然後回到原場景；返回會取消尚未確認的整理。

產物取得必須逐個道具處理，不可把同一物件上的多個產物合成一次全部取得。例如莓果叢目前有 `berry x3` 時，或月露草叢目前有 `moondew_herb x3` 時，介面應只顯示一個採集按鈕並標示剩餘數量；每次點擊只取得 `1` 個道具。每個取得按鈕都必須先檢查玩家背包負重，若取得後會超過 `player.maxCarryWeight`，按鈕不可使用並提示「你已經無法負荷這個重量。」

村莊廣場有一棵蘋果樹，每天會結出 `1` 顆可摘的蘋果；成熟的蘋果會留在樹上，直到玩家摘走。

村莊廣場有一個補給箱，每天會隨機補入 `2` 樣可消耗的戰鬥工具。補給池第一版為 `trap`、`torch`、`capture_net`。當補給箱內道具總數含堆疊已達 `6` 個時，當天不再補給；低於 `6` 時才會在每日重置補給，但不得補到超過 `6` 個。

村莊廣場有一個看板，可以看到今天大家的留言。看板互動每天最多觸發一次隨機留言事件；事件池由 `facilities.notice_board.dailyMessages.eventPool` 指向 `events.json`，第一版內容是女孩們留在看板上的留言拌嘴。抽選時只從尚未被永久 `seen` 旗標鎖住的事件中選擇；成功播放後寫入每日旗標 `notice_board_message_read_today`，當天再次互動不再觸發新留言。每個看板留言事件看過後都寫入自己的永久旗標，因此每則留言只會看一次。

宿舍有一張屬於主角的床位，玩家可透過「睡覺」快速度過 `1` 到 `6` 小時；體力恢復效率與等待相同，但不恢復生命。

夜晚宿舍可出現期間限定行動「參與紡織工作」，成本為 `30` 分鐘。這個行動的回饋應寫成事件式宿舍演出：在場女孩們一邊整理線束、布條或繩股，一邊彼此拌嘴、玩笑、關切主角，或對主角表現出自然的好奇；主角不說台詞，只用手上的動作、點頭、眼神、停頓，或敘事概略交代他的應對。紡織演出正文放在 `docs/data/events/events.json` 的宿舍紡織事件中，固定準備 `6` 段，每段都拆成 `2` 頁文字。輪番規則採一輪六段制：每次觸發時只會從本輪尚未播放過的事件中抽選，直到六段都看過才重置；新一輪開始時，第一段仍要避免和上一輪最後一段相同。紡織工作的時間推進與完成獎勵應掛在第二頁，讓狀態變化出現在第二段收束後；完成後由事件頁隨機取得 `繩索 x1` 或 `布條 x1`。

藥草棚有一座製藥台，互動後可以開啟製藥選單，只顯示 `workstationFacilityId = alchemy_table` 的已學會配方。

工坊倉庫有一座工作台，互動後可以開啟工坊製作選單，只顯示 `workstationFacilityId = workbench` 的已學會配方。

工坊倉庫內另外有一個木材堆，可反覆進行 `劈材`；每次花 `1800` 秒、消耗 `1` 點體力並取得 `wood x1`，不需要每日補貨。

工坊倉庫有一個廢料堆，每天可翻找出 `2` 份布條；尚未取走的布條會留在廢料堆裡，直到玩家帶走。

墓地有一個鴉巢，每天會隨機留下 `1` 個可取得物，可能是 `禽蛋 x1` 或 `羽毛 x1`。禽蛋是可食用道具，使用後恢復 `2` 點體力。

會直接消耗體力的行動，在介面上應明確顯示體力需求；體力不足時原則上應阻擋執行，而不是等玩家按下後才失敗。

開墾區有一塊田地，田地可升級但本身不提供直接功能；田地等級會決定其他農務與村莊功能。田地升級不提供直接設施指令，改由托莉的村莊委託處理。第一次升級委託為 `田地第一次整土`，向托莉提交 `soil x3` 後，田地等級 `+1`。田地最高 `5` 級。

### 製作與配方

製作系統第一版採「工作台讀配方表」的方式，不把配方直接硬寫進設施或道具資料。

- 配方資料放在 `docs/data/recipes/recipes.json`。
- 每筆配方至少要有五個核心欄位：做出什麼、需要哪些材料、在哪個工作台、耗時、是否學會。
- `是否學會` 的實際狀態屬於玩家存檔，不屬於工作台狀態；第一版新遊戲不預設學會任何配方，配方必須透過事件、委託或後續規則寫入 `save.player.knownRecipeIds`。
- 工作台只負責篩選自己能做的配方，例如製藥台只顯示 `workstationFacilityId = alchemy_table` 的配方。
- 玩家選擇配方時，必須一次檢查材料是否足夠；足夠才扣除材料、推進時間、加入成品，不足時只顯示缺少哪些材料，不扣任何東西也不推進時間。
- 第一版製作不做失敗率；只要材料足夠且配方已學會，就必定成功。
- 製作負重檢查以「扣除材料並加入成品後」的背包結果為準；不能只用目前背包直接加入成品來判斷，避免消耗重材料製作輕成品時被錯誤阻擋。
- 若消耗品存在可回收容器或殘留物，應在道具資料中明確寫出使用後返還物，而不是口頭默認。

開墾區有一個雜草叢。進入雜草叢時只顯示未知尋找狀態，不提前顯示可能取得的道具。玩家每日重置前只能尋找一次，按下 `尋找物品` 後才決定結果：可能取得 `1` 個月露草、`1` 朵花，或什麼都沒有。若找到的道具會讓玩家超過負重，該道具暫時留在雜草叢並顯示為已發現；玩家可以之後回來取回。若到每日重置時仍未取回，已發現道具會消失。

探索時也可能遇到臨時出現的莓果叢、月露草叢、雜草叢、碎石堆與土堆。這些探索版資源點每次前進或後退都重新抽取是否出現，不共享村莊內固定設施的狀態，也不在離開探索後保留。礦脈目前不作為探索臨時資源點，而是固定存在於森林外圍的隱藏洞穴內。

土堆不再固定出現在開墾區；它只作為探索臨時資源點出現。土堆是無限來源設施，玩家每次花 `30` 分鐘可取得 `1` 個泥土。泥土重量為 `1`，取得前必須檢查玩家背包負重；超過負重上限時不可取得，並提示「你已經無法負荷這個重量。」

倉庫箱可存放物品，初始負重容量為 `20`。倉庫箱升級不提供直接設施指令，改由諾絲的村莊委託處理。第一次升級委託為 `倉庫箱第一次加固`，向諾絲提交 `wood x5` 後，倉庫箱等級 `+1`。倉庫箱最高 `5` 級，每升一級容量增加 `10`。

開墾區有一口井。井是蜥蜴羅索的藏身處，也是白天找到蜥蜴羅索的入口；玩家必須完成蜥蜴羅索的乾糧委託後，才能在清晨、上午、中午或下午進入井底找他。夜晚與深夜時他會離開井底到開墾區做交易，因此井底入口不顯示，也不能同時在井底與開墾區出現。

### 村莊設施

第一版村內可自由移動的主要地點暫定五個；具體地點內設施與互動物件以 `docs/data/facilities/facilities.json` 為準：

- 村莊廣場：村民集合、分配工作、觸發公共事件與交換情報。
- 宿舍：集體住宿、夜間事件、舊資料保存、宿舍管理與村莊方針記錄。
- 藥草棚：製作恢復生命的物品，處理治療與藥草委託。
- 工坊倉庫：將村外物資轉化為工具與設施升級，並管理可存放的道具與材料。
- 開墾區：村莊邊緣的農務、開墾、巡視與危險預警地點；可通往墓地，並可銜接村外路線，但不作為單純出口存在。
- 墓地：位於開墾區外側的安靜坡地，暫時作為可移動地點保留，後續可承接弔念、線索、夜間事件或主線觸發。

### 村莊子場景與自由移動

村莊暫時先建立成一組可自由移動的子場景。玩家進入村莊後，不應只沿著單一路線推劇情，而是能在村莊內查看不同地點、找角色、處理委託與準備外出。

第一版村莊子場景暫定：

- 村莊廣場：村莊樞紐與大家聚集的點，公共討論、農務分配、簡易交易與村民集合事件都集中在這裡。
- 宿舍：村民集體住宿地點，由伊蓮管理；同時保存舊資料、點名表、住宿規則與夜間事件。
- 藥草棚：米菈治療、製藥與藥草委託的主要地點。
- 工坊倉庫：諾絲修繕、建設、工具製作、倉庫收納與材料交付的主要地點。
- 開墾區：托莉協助農務與雜務、賽拉巡視村外動靜的地點，也能作為前往村外路線前的準備區。

村莊內移動規則：

- 村內子場景之間可自由移動，通常消耗 `120` 到 `300` 秒。
- 每個子場景都直接顯示可互動清單，包含目前地點的設施與目前在場角色。
- 角色是否出現由當前時段、角色行程與 `allowPresentVillagers` 決定；角色行程的地點值使用 location id。設施是否出現由 `facilities.json` 的 `locationIds` 決定。
- 子場景的移動、使用道具、等待與地點特殊行動由 `locations.json` 與 `commands.json` 組合生成。移動主要由 `locations[].exits` 生成，不列入 `standardLocation` 基礎指令。
- 若角色此時不在該地點，玩家可以透過一般交談或事件提示得知其目前可能位置，但仍需自行移動過去。
- 村外探索入口可從村莊廣場直接進入；開墾區仍可作為出發前的準備與巡視地點。村外探索可受時間、體力、好感、委託與危險狀態影響。

## 村民系統

每個村民都必須有明確設定、性格、需求與可互動內容。第一版核心村民皆為年輕女性，且六位核心村民就是山谷村目前最後的成員。村民不是單純商店或委託板，而是村莊發展與「村莊還沒有結束」這件事本身的核心。故事上，六位核心村民都視為女主角；任何主線階段都不應只讓其中一人承擔敘事重量，其他人退化成交易、製作、治療或提示按鈕。

六名女主角伴隨主角冒險的方式可以是實體同行、出發前準備、回村後處理後果、提供關鍵判讀，或在主線選擇中要求主角回應她們的立場，但每個人都必須有不可替代的職責：

- 艾妲：負責村莊決策、資源分配、對外風險承擔與「是否繼續冒險」的政治責任。她讓主角的冒險不只是個人探索，而是整個村莊共同押上的選擇。
- 米菈：負責治療、瘴氣病理、藥草與身體代價判讀。她讓冒險後果落到傷口、病症、照護與是否還能承受下一次外出。
- 諾絲：負責工具、修繕、負重、裝備與失敗後的物件修復。她讓冒險不是憑勇氣硬闖，而是需要具體器物與可被修補的準備。
- 賽拉：負責路線、敵情、退路、夜間危險與森林判斷。她讓冒險有紀律、有代價，也讓主角理解活著回來本身就是責任。
- 托莉：負責食物、開墾區、村莊日常動力與對外界的好奇。她讓冒險不只通往真相，也連回「村子還想活下去」的普通願望。
- 伊蓮：負責舊資料、神話判讀、宿舍秩序與真相能否被大家承受。她讓冒險取得的線索變成能被理解、記錄與傳下去的故事。

主線事件設計時，每一階段至少要安排一名主要推動者與一名受到後果影響的女孩；完整主線則必須讓六名女主角都各自碰到不可替代的真相碎片。玩家不能只靠伊蓮解讀故事，也不能只靠賽拉處理冒險，否則會削弱「六位女孩共同支撐山谷」的核心。

村民日常的敘事語氣必須避免只有歡樂或只有苦難。她們可以鬥嘴、留言、開玩笑、互相吐槽，也可以因小事生氣或惡作劇，但這些輕鬆段落要像是她們在廢墟邊緣努力維持生活節奏。她們的笑不是無知或天真，而是在已經看見衰亡時，仍想替彼此、替主角、替山谷留下「我們曾經在這裡活過」的痕跡。艾妲、米菈、賽拉這類較克制的角色，悲傷與堅強會藏在責任、準備和警戒裡；托莉、諾絲這類較淘氣、活潑或嘴硬的角色，苦中作樂會更外顯，常用吵鬧、逞強或裝作不怕來遮住不安；伊蓮則更常用故事、玩笑話或語氣上的迂迴，把殘酷的真相說得讓人還能聽下去。

六名女孩的性格需要保留反差感，避免單一標籤化。艾妲外表穩重，內在卻容易被日常小事牽動；米菈待人溫柔，面對病理與瘴氣時能冷靜到固執；諾絲嘴上只講效率，實際最捨不得丟掉有痕跡的舊物；賽拉看似不信任未知，卻能克制地信任黑貓這類無法解釋的存在；托莉看似最天真，卻最努力把絕望裝飾成還能笑的樣子；伊蓮看似飄忽，實際最精準地維持宿舍秩序與記錄。後續台詞和事件應優先演出這些外顯印象與內在矛盾，而不是只重複職業定位。

玩家可見文案的共同準則集中記錄於 `docs/narrative-guidelines.md`。本節保留角色對話與知識提醒的核心規則；後續新增場景、物件、道具、戰鬥情報、事件正文或其他主文字區塊文案時，應先依該敘事準則檢查。

角色台詞必須避免模板化的格言收尾。不要頻繁使用「記住」「不要把 X 當成 Y」「真正的 X 不是 Y」「如果你想活著回來」這類可替換式警句。每段知識或提醒應先落在具體情境：看見的痕跡、角色手上的動作、某個地點、某件道具、一次村中經驗，然後再給出自然的行動建議。賽拉的語氣應偏向巡守實務，說腳印、風向、翅膀、前腳、距離和退路；伊蓮的語氣應偏向學者與舍監，說記錄、舊習慣、村裡誰會怎麼反應，以及她如何把殘酷事實講得能被聽下去。

角色對白應模擬角色設定中的性格演出，而不是只把資訊翻成直白說明。直觀表達是先讓玩家聽懂核心意思，不是把所有角色都寫成同一種說明口吻。艾妲可以先確認事務再下判斷；米菈可以短句、停頓、先看傷勢再說話；諾絲可以嘴硬、抱怨、用「嘖」「喂」等短促反應遮住稱讚；賽拉可以壓低聲音、先講路線與危險；托莉可以急著補話、用「欸」「那個」表現年輕與雀躍；伊蓮可以輕聲、含笑、用舍監與記錄的語感包住提醒。可適時加入狀聲詞、語氣詞或停頓，例如「嗯」「唉」「嘖」「喂」「欸」「……」，但必須服務於角色當下反應，不得每句都加，也不得用語氣詞取代資訊。若去掉角色名後所有台詞聽起來都像同一個人，應重寫。

所有玩家可見文案都應盡量直觀表達，不只限於女孩對白。場景說明、場景物件、道具說明、角色台詞、事件正文、知識對話、戰鬥情報、敵人描述、操作後回饋與好感事件，都不應長期把主要資訊藏在弦外之音或話中有話裡。含蓄可以作為情緒、猶豫、嘴硬、害羞或保留秘密的手段，但每段文字仍應讓玩家明確知道這段在說什麼：看見了什麼、危險在哪裡、物件能做什麼、角色為什麼改變態度、主角被要求做什麼、或下一步該注意什麼。若一句話讀完後只能得到模糊氣氛，卻不知道實際狀況或角色判斷，就應重寫。正式文案要優先使用具體主詞、具體動作、具體原因與具體結果，例如「你回來時把路況說清楚，我比較能準備藥」比「你讓明天有了方向」更適合承載信賴變化；「陷阱要放在牠前腳落地的位置」比「把位置放對」更適合承載戰鬥情報。比喻、象徵和詩性句子應少量使用，且必須接在已經說清楚的事後面，不能取代核心資訊。

避免讓玩家可見文案像 AI 生成的空泛句子：不要堆疊「缺口、方向、位置、故事、沉默、明天、答案、代價、真相」這類抽象名詞來假裝深意；不要連續使用「像是、也許、大概、不是 A 而是 B」讓語義變得漂浮；不要讓角色或敘事用不符合當下情境的高概念句子回應普通互動。每句文字應能回答至少一個問題：主角看見了什麼、聽見了什麼、摸到什麼、這東西能做什麼、危險從哪裡來、角色擔心什麼、角色希望主角怎麼做、或這句話推進了哪個角色關係。若答案不明確，優先改成直接說明。

### 年齡與性別規則

- 第一版核心村民皆為年輕女性，且她們是村莊最後仍在日常中活動的六位成員。
- 村莊內目前唯一男性是玩家主角。
- 上一代成年人與守護者只存在於背景、回憶、道具記錄、場景痕跡或遺跡線索中，不作為村莊當前常駐村民。故事開始約一年前，村中最後一位仍在日常裡活動的上一代成員已死亡或消失。
- 若新增村民，必須先確認是否符合「年輕女性村民」規則；不符合者只能作為回憶、失蹤者、外來者或特殊劇情角色。

### 村民互動

玩家可透過以下方式影響村民：

- 交談：了解個性、背景、需求與村莊問題。
- 送禮：用道具提升好感或觸發特殊反應。
- 交易：使用螢石向角色購買其攜帶物品。
- 協助：完成村民委託，改善村莊狀態。
- 邀請建議：讓村民影響村莊下一個發展方向。

伊蓮額外提供 `尋求知識` 行動，作為沉浸式教學與設定查詢入口。第一版分成三組口語問法：`想尋找物品`、詢問應先記住的生存規矩、詢問山谷舊事。道具來源只顯示材料、資源、可吃道具與可消耗戰鬥工具，並優先排目前委託需求物，其次排玩家曾經取得過的道具；程式以 `player.obtainedItemIds` 記錄取得歷史。來源內容由既有企劃資料推導，用伊蓮的自然對白回答，不顯示條列來源清單。知識文案可用 `§` 分頁。若玩家詢問如何擊敗敵人，伊蓮應把主角引導去問賽拉，並以故事式語氣提醒玩家把遭遇時的距離、聲音、傷痕與敵人第一個動作說清楚，不直接取代賽拉提供敵人弱點情報，也不要用固定格言句替代具體資訊。伊蓮繼承的是歷代學者留下的智慧與懺悔；她談到主線修補時，語義應落在「讓錯位的一切回復原位」，而不是「償還罪孽」或「用苦難抵消過錯」。

賽拉也提供 `尋求知識`，但主題集中在森林安全與敵人應對。`森林的注意事項` 可明確告知廢棄小屋可作休息點與中繼站、森林外圍約 `10%` 時使用搜尋可發現隱藏洞穴；作戰工具提示則必須保持賽拉式含糊表述，只描述飛行、衝撞、貼近等應對方向，不直接說出道具名稱。夜晚黑影情報應提醒主角夜裡森林會出現不像普通野獸的黑暗威脅，暗示照亮或逼退黑暗的準備重要，但不得揭露 `33%` 替換規則。

賽拉的事件與委託只要提到森林、外出準備或村外危險，就必須以鄭重語氣警告夜晚森林的危險性。這類文字可以提到「夜裡不是白天的延長」「不要把白天經驗當護身符」「需要能逼退黑暗的準備」，但不得直接說出黑影出現機率或敵人替換規則。

### 對話子指令

與角色對話時，先進入該角色的對話場景，再顯示角色專屬子指令。子指令會根據時段、地點、好感、旗標與委託狀態開放。

第一版對話子指令暫定：

- `chat`：閒聊或一般交談，通常小幅增加好感或取得背景資訊；透過交談增加的好感每名角色每天最多 `+1`。
- `gift`：送禮，消耗玩家道具，依角色喜好增減好感；每名可送禮角色每天只能收一次禮物。
- `trade`：交易，使用螢石購買該角色目前攜帶且已解鎖的物品。
- `quest`：查看、接取、回報或推進該角色相關委託；若角色目前沒有可用委託，不顯示此指令。
- `leave`：結束對話，回到目前子場景。

對話設計原則：

- 場景中不使用「與在場角色互動」這個中介指令。角色會直接出現在場景可互動清單中，玩家選擇角色後進入該角色的對話選單。
- 從互動選單返回場景也不消耗時間。
- 同一角色在不同時段或地點，對話內容可以不同。
- 角色對話需依目前時間、地點、好感、主角剛才的行為、目前委託、旗標與道具狀態動態挑選。
- 交談使用台詞池；一次交談先篩出符合角色、指令、時段、地點、好感與旗標條件的候選台詞，再從最高優先度候選中依 `weight` 隨機抽出一段；同優先度有兩筆以上時，不可連續抽到上一句。
- 交談若會增加好感，必須先檢查 `chat_affection_gained:{villagerId}`；當天已透過交談取得好感時，仍可交談與抽台詞，但不再增加好感。
- 送禮必須檢查道具是否可送、角色是否喜歡、當天是否已送過，以及是否有特殊委託需求；成功送禮後寫入 `gift_given:{villagerId}`，當天不得再次送禮給同一角色。
- 六名核心女孩的好感達到 `5` 以上時，應各自觸發一次「逐漸信賴」事件。這批事件放在 `docs/data/events/events.json`，條件使用 `villagerAffectionMins: { villagerId: 5 }` 與各自的 `*_trust_village_change_seen` 旗標鎖定。事件大綱是：角色開始把主角視為可被納入村莊日常的人，並用自身職責角度稱讚最近山谷村因主角到來而往好處變化。觸發檢查點統一放在玩家按下 `返回`、離開該角色互動頁時；交談、送禮、治療、委託或其他互動本身應先完整顯示自己的回應，不被好感事件立刻打斷。這樣保留委託、治療、劇情選擇或其他未來途徑增加好感後，也能在離開人物互動時補觸發。因為這類事件沒有時間與地點條件，文案必須能在任何該角色可互動的時段與地點成立：不要寫死藥草棚、田邊、工坊桌、宿舍點名紙、特定天色或手邊道具；開場應以「玩家準備離開時被叫住」「剛剛互動後留下的停頓」「角色回想最近變化」這類可攜情境切入。事件文案也必須避免把村莊寫成人口很多的熱鬧狀態，重點放在六名女孩分頭支撐的空村裡，多出一個能帶回材料、回報狀況、修補缺口、延續明天安排的人。
- 交易使用螢石作為貨幣，購買時扣除玩家螢石並減少角色攜帶物品庫存。
- 角色攜帶物品由好感度解鎖；未達 `affectionMin` 的物品不顯示。
- 角色專屬互動可放在 `villagers[].interactionActions`。例如米菈的治療行動只在米菈本人互動選單中顯示，不在藥草棚地點行動中顯示。
- 角色攜帶物品有每日庫存上限與每日補貨量。被購買後當日庫存減少，隔天依 `dailyRestock` 補貨，最多補到 `max`。
- 委託指令負責把委託加入玩家已接委託、更新委託階段或領取獎勵。
- 好感不應只靠送禮堆高，完成承諾、遵守規矩、回報委託與劇情選擇也會影響好感。

### 蜥蜴羅索

蜥蜴羅索是魔物商販 NPC，資料上使用 `npcType: "special_merchant"`，不走六名核心村民的好感、送禮、日常行程與標準委託互動邏輯。他是長命種旅行商人，從很久很久以前就一直生活到現在；漫長人生中，他選擇在山谷這個濕度與溫度都正好的地方暫時休息。山谷的神話、詛咒、失蹤與衰亡對他來說都不是忌諱，也不值得評論，因為他認為自己想走時隨時可以走。

羅索的敘事態度是「什麼都與我無關，只要能做買賣就好」。他不想多說山谷的事，也不想評價村民、神殿或黑影；即使他知道一些古老痕跡，也應避免主動變成解說角色。村莊對他的存在同樣不大驚小怪：女孩們知道他在附近做買賣，偶爾需要工具或物資時會跟他交易，但不把他當成村莊成員，也不要求他表態、幫忙或留下。

- 出現地點：開墾區。
- 出現時間：夜晚與深夜會出現在開墾區；清晨、上午、中午與下午不在地面開墾區出現。
- 藏身處：開墾區的井底。
- 白天找人：完成蜥蜴羅索的乾糧委託後，玩家白天可以從井底找到蜥蜴羅索；夜晚與深夜不能從井底找他，因為同一時間他已在地面開墾區做交易。
- 收購規則：玩家可以把身上與倉庫箱中的可賣道具交換成螢石。是否可賣與售價以 `items.json` 的 `price` 欄位為準。
- 販售規則：蜥蜴羅索販售除了彈藥與裝備以外的所有戰鬥工具；目前包含陷阱、火把與補網。
- 價格單位：所有買賣都使用螢石。

蜥蜴羅索的解鎖委託沒有前置條件。委託內容是提交 `3` 個乾糧給蜥蜴羅索，獎勵是取得旗標 `lizard_merchant_day_access`，之後白天可從開墾區的井底找到他。

### 對話資料與動態組合

對話資料以 JSON 管理，放在 `docs/data/dialogues/dialogues.json`。每筆對話資料是一個可被條件挑選的片段，系統可在一次互動中抽取多筆片段組成完整回應。

對話挑選因素：

- 目前角色：只挑選 `villagerId` 符合的資料。
- 目前指令：依 `command` 區分 `chat`、`gift`、`trade`、`quest` 等用途。
- 目前時間：可用 `timeBlocks` 限制清晨、上午、中午、下午、傍晚、夜晚或深夜。
- 目前地點：可用 `locationIds` 限制角色在特定子場景時的台詞，值對應 `locations.json` 的 `id` 或探索 scene id。
- 好感範圍：可用 `affectionMin` 與 `affectionMax` 控制親近程度不同的反應。
- 主角方才的行為：可用 `recentPlayerActions` 判斷玩家剛查看、等待、受傷、送禮、完成工作、違規外出或回報委託。
- 委託狀態：可用 `questStates` 判斷委託未接、進行中、可回報、完成或失敗。
- 旗標與道具：可用 `requiredFlags`、`blockedFlags`、`requiredItems` 控制特殊資訊與反應。

一次對話的組合方式：

1. 先挑選必要片段，例如委託可回報、主線警告、角色對剛才行為的即時反應。
2. 再挑選情境片段，例如時間、地點、天氣、目前村莊狀態或角色心情。
3. 最後挑選角色性格片段，例如口頭禪、關心、吐槽、保留態度或好感反應。
4. 同一次輸出建議組合 1 到 3 筆片段；重要劇情可增加到 4 筆，但避免冗長。
5. 若多筆資料符合，依 `priority` 由高到低選取；同優先度可依 `weight` 做加權隨機或輪替。
6. 已顯示過的低優先度日常片段可短時間降權，避免連續重複。

對話資料欄位範例：

```json
{
  "id": "mira_after_player_injured_evening",
  "villagerId": "mira",
  "command": "chat",
  "category": "recent_reaction",
  "priority": 80,
  "weight": 1,
  "conditions": {
    "timeBlocks": ["傍晚", "夜晚"],
    "locationIds": ["herb_shed"],
    "affectionMin": 1,
    "recentPlayerActions": ["returned_injured", "used_healing_item"],
    "questStates": {
      "mira_moondew_herb": ["active", "readyToReport"]
    }
  },
  "lines": [
    "米菈先看你的傷口，才抬頭看你。她的聲音很輕，但語氣比平常更堅定。",
    "「我知道你想幫忙，可是你每次把自己弄成這樣，我都得重新計算藥草還夠不夠。」"
  ],
  "effects": {
    "flags": ["mira_warned_about_injury"]
  },
  "tags": ["米菈", "受傷反應", "藥草棚"]
}
```

### 角色行程

每個角色都有依時段變化的出現地點。第一版先使用時段位置，不做逐分鐘移動；同一時段內角色固定出現在指定子場景，直到時間跨入下一個時段後重新判定。

行程資料先放在村民資料的 `defaultLocations` 欄位，欄位值使用 location id 或探索 scene id，不使用中文地點名作為程式匹配鍵。未來若需要更細緻，可擴充為 `schedule` 陣列，加入天數、天氣、委託階段、事件旗標與例外行程。

行程判定規則：

- 以目前 `timeBlock` 讀取角色的 `defaultLocations`。
- 若角色有進行中的委託事件，可用委託階段覆蓋預設地點。
- 若角色因劇情暫時離村，該角色不會出現在村莊子場景。
- 夜晚仍依 `defaultLocations` 顯示角色；若核心女孩位於共同宿舍，玩家仍可互動。深夜若角色位於共同宿舍，視為不可互動，不顯示人物入口；非共同宿舍位置仍依資料判定是否可互動。若未來需要其他不可打擾狀態，必須由角色事件、旗標或明確條件個別指定。
- 子場景描述可以根據目前在場角色調整文字。

### 村民資料欄位

村民角色資料以 JSON 管理，放在 `docs/data/villagers/villagers.json`。

企劃文件只保留資料格式與設計規則；村民的完整設定、喜好、委託鉤子、解鎖內容與預設作息位置以資料檔為準。角色的可抽取對話片段則放在 `docs/data/dialogues/dialogues.json`，不要直接塞進村民基本資料。

角色關係以 `relationships.json` 的角色配對資料管理，方便對話、委託與事件判斷誰會替誰說話、誰會擔心誰、誰會在特定決策上衝突。

```json
{
  "id": "mira",
  "name": "米菈",
  "role": "藥草師",
```

## 指令與選擇

遊戲以「選擇指令」進行。每個場景提供多個行動，每個行動都有清楚的代號與描述。

### 場景基本指令

多數可自由停留的場景都應提供一組基礎指令。正式指令清單以 `docs/data/commands/commands.json` 為準；Markdown 只解釋用途，不另外新增舊指令。

第一版村莊地點的 `standardLocation` 基礎指令為：

- `useItem`：打開道具選單，從背包選擇可在目前場景使用的物品，或改為批量丟棄道具；被丟下的道具會形成臨時地面設施，1 小時後連同內容一起消失，若先被全部取回則立即消失。
- `wait`：等待/休息一段時間，用於切換時段、等角色出現、觀察變化、觸發定時事件；等待每 `1` 分鐘累積 `1` 點隱藏休息值，休息值每滿 `30` 點恢復 `1` 點體力。在探索區等待需承擔遭遇風險。選擇此指令只會打開等待時間選擇，不會立刻推進時間。

村莊移動不放在 `standardLocation` 中，而是由 `locations[].exits` 生成「移動到」分組。`commands.json` 仍保留 `move` 指令定義，作為出口或移動型選項的共用 action type。出口若需要先播放警告、檢查或過場，可加上 `eventBeforeMove`；事件完成後由事件選項決定實際 `targetSceneId`，不再假設按下出口一定到達 `to`。

探索區不使用上述完整村莊指令組。探索區第一版使用：

- `exploreDeeper`：向深處移動。
- `exploreDeeperSlow`：向深處緩慢移動。
- `forage`：搜尋。
- `wait`：等待/休息一段時間。
- `useItem`：打開道具選單。
- `shoutAttractEnemy`：大喊。
- `retreat`：向村莊移動。
- `retreatSlow`：向村莊緩慢移動。

時間成本原則：

- 進入場景選單、進入角色互動選單、離開角色互動選單，預設 `timeCostSeconds` 為 `0`。
- 使用道具依道具本身的 `timeCostSeconds` 決定；例如吃乾糧很短，治療或解讀道具較久。
- 等待必須讓玩家先選擇等待長度，再按確認執行；確認前不推進時間。
- 第一版等待時間用 slider bar 控制，最小 `1` 分鐘、最大 `60` 分鐘，步進 `1` 分鐘。
- 只有真正採取行動時才推進時間；打開選單或查看可用選項不應懲罰玩家。
- 若指令需要先進入附帶介面才能完成，例如交易草稿、倉庫整理、黑貓帶回東西、等待時間選擇、睡覺時間選擇、戰利品整理或未來多步驟確認，入口只代表準備流程，不消耗時間、體力或道具，也不播放假讀條。玩家可返回反悔；返回只取消草稿或回到上一層。只有按下確認後才算真正執行，並按照該指令原本的成本、時間與讀條規則處理。

範例：

```text
你在藥草味濃厚的小屋中醒來，手臂纏著繃帶。門邊的獵戶巡守賽拉正盯著你，藥草師米菈則把熱水放到床邊。

1. 向米菈詢問自己在哪裡
2. 試著向賽拉解釋自己的來歷
3. 勉強起身查看屋外（消耗 1 體力）
```

玩家可輸入：

```text
1
```

或未來在圖形介面中點選對應選項。

## 回合流程

每次玩家做出選擇時，系統依序處理：

1. 讀取目前場景。
2. 顯示場景描述與可選行動。
3. 玩家選擇一個行動。
4. 檢查行動條件。
5. 若條件不符合，提示原因並停留在目前場景。
6. 若條件符合，套用消耗與效果。
7. 推進行動時間成本。
8. 更新玩家、村莊與村民狀態。
9. 檢查時間觸發事件。
10. 檢查生命是否歸零，或戰鬥結果是否為大失敗。
11. 若森林普通探索中生命歸零或戰鬥大失敗，套用森林守護獸救回規則。
12. 若主線、遺跡或特殊劇情明確定義死亡條件，才進入死亡結局。
13. 若仍可行動，移動到下一個場景或刷新目前場景。

## 行動類型

### 普通行動

不需要消耗體力，通常用於村莊互動、移動、等待、使用道具或推進劇情。進入選單類指令不消耗時間；真正執行內容時才依行動設定推進時間。

範例：

- 前進
- 返回
- 與村民閒聊：正式聊天內容，會耗時。
- 查看工坊倉庫資源
- 觀察遺跡符號
- 使用背包道具
- 打開等待時間選擇：不耗時，選定 1 到 60 分鐘並確認後才推進時間。

常見時間成本：數分鐘到 30 分鐘。

### 選單行動

只用於開啟或關閉選單，不代表角色真的做了花時間的事情。

範例：

- 開啟使用道具選單
- 返回上一層場景
- 查看可接委託列表

常見時間成本：`0` 秒。

### 特殊行動

需要消耗體力，通常帶來較高收益、避開危險、取得隱藏資訊或爭取村莊外的稀有資源。

範例：

- 攀爬斷崖
- 強行打開生鏽鐵門
- 搜查廢墟
- 奔跑逃離危險
- 搬運木材回村
- 抄錄遺跡中的禁忌記錄

常見時間成本：30 分鐘到數小時。

### 恢復行動

用於恢復生命或體力。生命恢復必須是治療相關動作；一般等待、休息與睡眠只處理體力或時間推進。村莊內的恢復通常較安全，但可能消耗食物或時間；村外等待風險較高。

範例：

- 在家睡覺：可快速度過 `1` 到 `6` 小時，恢復效率與等待相同。
- 等待累積 `30` 分鐘：恢復 1 體力，不恢復生命。
- 睡覺：可快速度過 `1` 到 `6` 小時，依等待規則恢復體力，不恢復生命。
- 使用草藥或治療藥：恢復生命。
- 吃乾糧：恢復 3 體力。
- 請米菈治療：從米菈本人互動選單進入，需要米菈目前在場，並消耗指定藥草或符合後續好感規則；目前效果為恢復主角最大生命的 `70%`，不超過生命上限。若主角生命已滿，該治療行動不可使用。

常見時間成本：數分鐘到整晚。

### 村莊行動

用於推動村莊發展，通常需要指定道具、指定設施狀態或特定村民好感。

範例：

- 修復工坊倉庫
- 擴建藥草棚
- 整理村外路標
- 舉辦村民會議
- 分配食物與材料

常見時間成本：30 分鐘到半天。

## 委託系統

委託用於記錄玩家已接下的承諾、角色委託、村莊目標與主線進度。委託模板資料以 JSON 管理，放在 `docs/data/quests/quests.json`。企劃資料檔不是存檔內容；玩家接收到的委託會存進遊戲狀態中的 `quests.active`，完成後移入 `quests.completed`，失敗或放棄則移入 `quests.failed`。

### 委託類型

- `main`：主線委託，推動詛咒、遺跡與上一代失蹤真相。
- `villager`：角色委託，與特定村民需求、好感和解鎖內容相關。
- `village`：村莊發展委託，通常要求道具、設施狀態或特定旗標條件。
- `exploration`：村外探索委託，要求前往特定地點、帶回材料或發現線索。
- `tutorial`：系統教學委託，協助玩家理解行動成本與基礎循環；第一版不使用固定開場事件教學。
- `special`：特殊 NPC、特殊設施或特殊規則解鎖委託，例如蜥蜴羅索的井底入口。

### 委託資料欄位

委託資料先以最小核心欄位為主，之後再依需要擴充目標、委託類型、給委託角色與後續委託。

核心欄位：

- `accepted`：模板預設接取狀態，第一版通常為 `false`；真正接取狀態保存在 save data。
- `repeatable`：是否可重複。
- `completed`：模板預設完成狀態，第一版通常為 `false`；真正完成狀態保存在 save data。
- `title`：委託名稱。
- `description`：委託說明。
- `prerequisites`：前置條件，玩家必須滿足後才可以接取該委託。
- `rewards`：委託獎勵。

擴充欄位可包含：

- `id`：委託資料識別碼。
- `type`：委託類型。
- `giver`：給委託角色。
- `objectives`：委託目標。
- `nextQuestIds`：完成後可能接續的委託。
- `tags`：分類與檢索標籤。

```json
{
  "id": "mira_moondew_herb",
  "title": "藥棚的月露草",
  "description": "米菈需要兩株月露草來補充藥草棚庫存。",
  "accepted": false,
  "repeatable": false,
  "completed": false,
  "type": "villager",
  "giver": "mira",
  "statusOnAccept": "active",
  "prerequisites": {
    "affection": {
      "mira": 1
    },
    "flags": []
  },
  "objectives": [
    {
      "id": "collect_moondew",
      "description": "向米菈提交 2 株月露草",
      "type": "submitItems",
      "targetId": "moondew_herb",
      "requiredCount": 2
    }
  ],
  "rewards": {
    "items": {
      "empty_potion_bottle": 1
    },
    "villagerAffection": {
      "mira": 2
```

### 接取與保存規則

- 委託可由對話子指令 `quest`、場景事件、主線旗標或村莊會議觸發。
- 多頁事件使用 `docs/data/events/events.json` 管理。事件可由進入場景、與特定角色互動、移動前攔截或其他後續擴充觸發源啟動；每頁可包含文字、狀態效果與分支選項。
- 事件頁的分支選項可接到另一個事件，例如 `AAA -> AAA-A` 或 `AAA -> AAA-B`；也可指定 `targetSceneId` 讓事件結束後直接移動到某個地點。不要在同一頁直接做任意頁碼跳轉。
- 移動前事件由 `locations[].exits[].eventBeforeMove.eventId` 指向事件資料。這類事件的最後選項可決定仍然前往、折返原地或改去其他地點，並可各自套用時間、旗標與結果訊息。
- 事件若直接交付實體道具，仍必須檢查負重；超重的道具會落到目前所在地的 `遺落的道具`，並在狀態變化區塊明確提示。
- 事件若教會配方，應把配方 id 寫入 `player.knownRecipeIds`，並在狀態變化區塊顯示學會的配方名稱。
- 委託若教會配方，使用 `rewards.recipeIds` 記錄配方 id，完成委託時寫入 `player.knownRecipeIds`，並在狀態變化區塊顯示學會的配方名稱。
- 委託只有在 `prerequisites` 全部滿足時才可接取。
- `quests.json` 是只讀委託模板，不是 save data。程式不應在遊戲進行中改寫企劃資料檔。
- 委託接取時，系統從委託資料表複製必要狀態到 `quests.active`。
- 進行中的委託需保存 `questId`、`status`、`currentStepIndex`、各目標完成數量與接取時間。
- 同一委託預設不可重複接取；若需要可重複委託，使用 `repeatable: true`。
- `accepted` 與 `completed` 在模板中只代表初始預設值，一開始當然通常都是未接取與未完成；玩家實際委託狀態以 save data 的 `quests.active`、`quests.completed`、`quests.failed` 為準。
- 委託完成時發放獎勵，獎勵第一版重點為物品與村民好感，也可附帶設施狀態或旗標。
- 委託獎勵若包含物品，必須檢查背包或容器容量；若包含好感，使用 `villagerAffection` 調整角色好感。
- 委託獎勵若包含設施升級，使用 `rewards.facilities[facilityId].levelDelta` 調整 save data 內的設施等級，並同步寫入對應完成旗標。第一版不從 `facilities.json` 生成直接升級指令。
- 第一次倉庫箱升級由諾絲委託 `倉庫箱第一次加固` 處理：提交 `wood x5`，獎勵 `storage_box level +1`。
- 第一次田地升級由托莉委託 `田地第一次整土` 處理：提交 `soil x3`，獎勵 `field level +1`。
- 諾絲的第一個負重訓練委託為 `工坊的負重訓練`：無前置條件，向諾絲提交 `wood x2` 後，獎勵 `carrySkill +1`。
- 諾絲的第二個負重訓練委託為 `工坊的負重訓練（二）`：前置條件為 `carrySkill >= 1`，向諾絲提交 `soil x2` 與 `stone_ammo x2` 後，獎勵 `carrySkill +1`。玩家可理解為學會固定零散重量與不穩定物品的進階搬運訓練。
- 托莉有一個防身訓練委託：無前置條件，向托莉提交 `berry x3` 後，獎勵 `meleeWeaponSkill +1`。文案重點是開墾區靠近村外，托莉用莓果作為練習後補給，陪玩家做基礎近身防身練習。
- 賽拉有一個近戰訓練委託：前置條件為 `meleeWeaponSkill >= 1`，向賽拉提交 `cracked_bone x3` 後，獎勵 `meleeWeaponSkill +1`。
- 賽拉有一個煙霧彈配方委託：向賽拉提交 `dry_moss x1` 後，獎勵 `smoke_bomb_recipe`。
- 委託狀態會影響角色對話；同一個角色在委託未接、進行中、可回報、完成後，應能抽取不同對話片段。

### 事件、委託與旗標的分工

`events/events.json`、`quests/quests.json` 與 `flags/flag-sources.json` 必須各自負責不同層級，避免同一段進度同時被多個資料表宣告。

- `events/events.json` 負責入口與當下演出：誰觸發、何時觸發、播放哪些事件頁、每頁立即套用哪些效果、事件分支接到哪個事件或哪個 `targetSceneId`。
- `quests/quests.json` 負責可追蹤承諾：委託是否可接、進行中目標、可回報條件、完成時獎勵與後續委託。
- `flags/flag-sources.json` 負責完成後鎖定與解鎖索引：記錄哪個事件、委託或互動會寫入旗標，以及該旗標解鎖或阻止什麼內容。
- 若一段內容只是一次性敘事，使用事件加旗標鎖定，不建立委託。
- 若玩家需要在側欄追蹤目標、收集數量或回報對象，必須建立委託，事件只能作為接取、回報或轉場演出。
- 旗標不得替代委託進度。旗標只保存「已發生 / 已解鎖 / 已阻止重播」這類完成後狀態。
- 新增旗標時，必須同步在 `flags/flag-sources.json` 登記 `sourceType`、`sourceId`、`description` 與 `unlocks`；玩家可見旗標才額外寫 `playerVisible` 與 `playerDescription`。

## 狀態資料草案

正式存檔範本放在 `docs/data/save/save-template.json`。第一版使用 `localStorage` 保存同一份 JSON；玩家按下 `存檔` 時才寫入本機存檔。匯出時直接輸出目前 save data，匯入時以玩家選擇的 JSON 覆寫目前本機存檔。

存檔只保存運行中會變化的狀態，不保存企劃模板內容。`docs/data/*.json` 仍是靜態規則與資料來源，遊戲過程不得把委託進度、設施等級、交易庫存或每日採集狀態寫回企劃 JSON。

核心欄位：

- `schemaVersion`：存檔格式版本，例如 `save@v1`。
- `storage`：記錄第一版使用 `localStorage`、手動存檔、JSON 匯出與匯入覆寫策略。
- `time`：日數、當日秒數、時段、`dailyResetSecond: 21600` 與最近每日重置日。
- `player`：生命、體力、技能、背包、永久旗標、每日旗標與近期行為。
- `player.knownRecipeIds`：玩家目前已學會的配方 id 清單。
- `player.encounteredEnemyIds`：主角曾經遭遇過的敵人 id 清單，屬於隱藏紀錄，不在一般介面顯示。
- `player.knownEnemyWeaknessIds`：主角已從賽拉情報中知曉弱點的敵人 id 清單，屬於隱藏紀錄；遭遇對應敵人時可用來顯示 `賽拉的情報` 查閱按鈕。
- `currentSceneId`：目前所在場景或地點 id。
- `exploration`：本次探索是否進行中、目前區域、探索進度、臨時收穫與遭遇狀態。
- `exploration.forageState`：記錄目前探索位置是否已搜尋過；移動到新位置時重置。
- `exploration.sceneryText` / `exploration.sceneryPositionKey`：記錄目前探索位置抽到的景色描述與位置鍵，避免畫面重繪時描述反覆變動。
- `exploration.temporaryResourceNodes`：記錄本次探索目前位置臨時抽出的資源點；離開探索區時清空。
- `facilities`：以 facility id 保存等級、容量、倉庫內容、每日採集/搜尋使用狀態與特殊入口解鎖。
- `villagers`：只保存六名核心村民的好感等核心社交狀態；特殊 NPC 不放普通好感。
- `tradeStocks`：以角色或商人 id 保存當前庫存，普通角色與特殊商人都在每日重置時補貨。
- `quests`：玩家實際委託狀態，分為 `active`、`completed`、`failed`。
- `mainStory.forbiddenBooks`：保存禁忌之書相關委託完成次數、下一本揭露序號與已揭露書本 id。
- `dialogueHistory`：近期出現過的對話與顯示次數。
- `dailyReset`：每日重置流程的最後處理日、處理時間與 pending 狀態。
- `gameState`：例如 `playing`、`ending` 或未來擴充狀態。

範例結構以 `docs/data/save/save-template.json` 為準；後續企劃新增會變動的系統時，必須同步檢查是否需要更新該存檔範本。

## 場景資料草案

正式可自由停留地點以 `docs/data/locations/locations.json` 為準；`docs/data/scenes/scenes.json` 只保留森林探索與對話入口。需要連續播放多頁文字、條件觸發與事件分支時，改放進 `docs/data/events/events.json`。

### 場景類型

- `hub`：樞紐場景，例如村莊廣場，可通往多個子場景。
- `location`：一般地點資料，例如宿舍、藥草棚、工坊倉庫或開墾區；正式資料放在 `locations.json`。
- `dialogue`：角色互動選單場景，本身不代表已經花時間對話，正式選擇交談、送禮、委託等子指令時才計算時間。
- `story`：劇情事件場景，通常由委託、旗標或主線觸發。
- `exploration`：村外探索場景，通常有更高時間、體力與生命風險。

村莊自由移動的地點資料範例：

```json
{
  "id": "village_square_hub",
  "title": "村莊廣場",
  "location": "村莊廣場",
  "sceneType": "hub",
  "baseCommandSet": "standardLocation",
  "allowPresentVillagers": true,
  "description": "廣場連接宿舍、藥草棚、工坊倉庫、開墾區與森林外圍。你可以在這裡確認目前有哪些村民在場，也可以直接離開村莊前往森林探險。",
  "exits": [
    { "to": "dormitory", "label": "前往宿舍", "timeCostSeconds": 180 },
    { "to": "herb_shed", "label": "前往藥草棚", "timeCostSeconds": 180 }
  ]
}
```

## 最小可行版本範圍

第一版先做出以下內容：

- 一名男性玩家角色，身份是被村莊收留的外來者
- 生命與體力
- 攜帶重量
- 探索技能、近戰武器技能、遠程武器技能、攜帶技能與採集技能
- 6 名有設定、性格、需求與委託線的年輕女性村民；她們是山谷村最後的成員，不是龐大村莊中的代表人物
- 至少 10 個場景，其中包含村莊場景與村外探險場景
- 第一版村外探索區先以山谷森林為主，包含山谷森林外圍、山谷森林中層與山谷森林深處
- 森林探索支援本次探索進度、向深處移動、向村莊移動、慢速移動、搜尋、等待/休息、使用道具與大喊
- 戰鬥採用遭遇選項判定，不做逐回合攻防
- 怪物至少支援大勝利、勝利、險勝、失敗與大失敗五階段戰鬥結果
- 森林生命歸零時不死亡，由神秘善良野獸暗中救回村莊，日期推進 1 天，並讓可懲罰遺失道具各自有 `33%` 機率消失
- 生命歸零醒來時，隨機一名核心女孩照顧主角
- 村莊內約 5 個可自由移動的必要子場景
- 每名核心村民都有依時段變化的預設出現地點
- 與角色對話時提供交談、送禮、委託等子指令
- 一般場景 2 到 4 個選擇；樞紐場景可有較多移動選項
- 24 小時制時間系統，所有行動皆有時間成本
- 至少 3 個需要消耗體力的特殊行動
- 至少 2 種恢復方式
- 至少 3 個可升級或可修復的村莊設施
- 至少 4 條村民委託
- 委託資料表與已接委託保存狀態
- 委託完成獎勵至少支援物品與村民好感
- 至少 1 個死亡結局
- 至少 1 個村莊衰敗結局
- 至少 1 個村莊復興成功結局
- 可重玩並做出不同選擇

## 初始遊戲方向提案

暫定主題：被活祭謊言遮蔽的山谷村復興。

玩家原本是山谷外的人，因遺跡力量與納薇菈殘存引導交錯造成的異常牽引，意外闖入與世隔絕的山谷，並被逐漸衰敗的山谷村收留。村中只剩六位年輕女性作為最後的成員，因為山谷村多年來把每十年補充一名神殿祭司的制度誤傳成活祭；納薇菈被誤封時，祀奉她的女神像被完全搗毀，祭司團也被全數殺害，使山谷自古存在的瘴氣失去調和，瑟菈彌也逐漸被瘴氣侵蝕而失去原有理智，導致上一代再也生不出男孩，夜晚黑影逐漸肆虐。納薇菈如今只能藉由祭司女孩留給神明的黑貓布偶，以幾乎無力的黑貓姿態顯現。瘴氣失衡表面上只影響村中出生，不直接綁住山谷外來者，因此主角能成為舊契約外的變數。納薇菈察覺這個變數後，把主角推向山谷，但開場時玩家只會以為自己聽見的是遺跡深處的呼喚。後來上一代的成年人與守護者為了尋找山谷之神、神殿祭司真相與調和瘴氣的方法前往遺跡，卻一去不回。村莊本身相對安全，但缺乏人手、資源與防衛能力，村外則有山谷森林、廢棄礦道與古代遺跡。整體敘事應帶有悲傷、堅強與苦中作樂的氛圍：玩家需要白天與村民互動、理解每個人如何在玩笑與責任之間撐住村莊，並在準備充足時冒險離村，帶回木材、食物、藥草、礦石與失蹤調查隊的線索。

這個方向適合第一版，因為它能同時支持慢活互動、村莊建設、角色委託、外出風險、資源管理與多結局。

## 待決設計問題

下一步需要決定：

1. 襲擊主角的野獸是否受到詛咒影響，還是只是村外危險生態的一部分？

4. 探索技能、近戰武器技能、遠程武器技能、攜帶技能與採集技能要透過什麼方式訓練或提升？

6. 村莊發展要偏自由建設，還是以主線階段逐步解鎖？
7. 成功結局是「村莊復興」、「調和瘴氣」、「救回上一代」、「揭開活祭謊言」、「改寫神殿祭司制度」，還是多結局並存？
