# Data Structure Notes

The data is split to reduce repeated text and token usage.

- `commands/commands.json`: shared command definitions and reusable command sets.
- `locations/locations.json`: reusable free-move locations. Standard location commands and exits are generated from this file. `impressionInfos` provides dynamic scene impressions for the main text area.
- `relationships/relationships.json`: character relationship summaries by pair, not repeated inside every villager.
- `villagers/villagers.json`: compact character index, dynamic impression info, appearance, protagonist-facing profile text, schedule, likes, dislikes, needs, hooks, unlocks, and carried trade items.
- `scenes/scenes.json`: story, exploration, and dialogue scenes only. Do not duplicate standard location data here. Exploration scenes may use `impressionInfos` and `sceneryDescriptions` for dynamic environmental impressions.
- `events/events.json`: multi-page triggerable events. Use this for cutscene-style text pages, per-page effects, event-to-event branch choices, and event choices that end by moving to a `targetSceneId`.
- `quests/quests.json`: quest definitions, objectives, prerequisites, and rewards.
- `dialogues/dialogues.json`: conditional dialogue fragments for dynamic dialogue composition.
- `items/items.json`: item definitions.
- `exploration/forage-loot.json`: exploration forage pools, item-count weights, and per-position forage-lock rules.
- `exploration/resource-node-spawns.json`: transient exploration resource-node spawn rules for forward/backward movement.
- `recipes/recipes.json`: crafting recipe definitions, including result item, required materials, target workstation, time cost, and learning defaults.
- `enemies/enemies.json`: forest enemy definitions, categories, difficulty, ranged dodge, feared combat tools, and weapon/ammo effectiveness. Combat-tool weakness follows enemy category by default: 飛獸 -> `capture_net`, 四足獸 -> `trap`, 雙足獸 -> `torch`; 巨獸 is the exception, with low ranged dodge but difficulty 3 points above its region, no guaranteed combat tool, no trap weakness, and no melee-weapon weakness. The night black shadow is a special night-encounter replacement enemy and should not be treated as part of the normal region enemy pool.
- `facilities/facilities.json`: interactable facilities and objects inside locations, such as bushes, storage boxes, and notice boards.
- `flags/flag-sources.json`: small flag source table. Each record must include a Chinese `description` for author/debug lookup, and explain which scene, quest, or interaction creates a flag and what that flag unlocks. Flags are hidden from normal player output unless `playerVisible: true`; visible flags must use `playerDescription` for player-facing text and must not show the raw flag id.
- `npc-interactions/npc-interaction-rules.json`: minimal interaction matrix for core villagers and special NPCs.
- `knowledge/elaine-knowledge.json`: Elaine's immersive knowledge/help topics. Category labels are player-facing questions, while item-source answers are generated from existing item, facility, quest, event, recipe, trade, and exploration data.
- `knowledge/sela-knowledge.json`: Sela's forest survival topics. Enemy entries are generated from encountered enemy ids and `enemies.json`; forest-note topics are authored here.
- `save/save-template.json`: `localStorage` manual-save template for export/import, including runtime-only facility state, trade stock, quest progress, daily reset tracking, and current player state.

Token-saving rules:

- JSON files are the authoritative source for executable data values such as weights, time costs, stock counts, recovery amounts, and actual record counts. Markdown planning text explains intent and rules; if it differs from JSON on a concrete value, update the Markdown summary and keep JSON as source of truth.
- JSON files must be saved as UTF-8 without BOM to keep the data portable across tools and runtimes.
- Do not repeat standard location commands in every location scene. Use `baseCommandSet`; the current `standardLocation` set is defined by `commands.json` and contains `useItem` and `wait`.
- Use separate command sets for village locations and exploration areas. `baseCommandSets.standardLocation` is for village free-move locations; `baseCommandSets.exploration` is for exploration scenes. Commands must declare `usableIn` so the same command table can still prevent village-only actions from appearing in exploration areas.
- Do not hard-code visible interaction choices in every location. Generate a location interaction list from facilities whose `locationIds` include the current `locations[].id` or current scene id, plus present characters from schedules and `allowPresentVillagers`.
- Location and exploration scene main text should use dynamic `impressionInfos` when available. These entries describe what the protagonist notices in the current place and may use conditions such as `timeBlocks`, `locationIds`, `requiredFlags`, `blockedFlags`, `requiredDailyFlags`, and `blockedDailyFlags`. They must not expose internal systems, probabilities, reset rules, or implementation details.
- Do not hard-code character shop lists in scenes. Generate trade options from `villagers[].carriedItems`, filtered by affection and daily stock.
- Core villagers should keep clear `appearance`, `impressionInfos`, and `profileDescription` fields. `appearance` is for stable visual/art direction; `impressionInfos` is the data-driven pool for dynamic first-entry NPC impressions in the main text area and may use conditions such as `timeBlocks`, `locationIds`, `requiredFlags`, `blockedFlags`, `requiredDailyFlags`, `blockedDailyFlags`, `affectionMin`, and `affectionMax`; `profileDescription` is prose from the protagonist's perspective and should stay immersive rather than system-facing. Impression info must describe what the protagonist notices at that moment, not long biography, internal systems, or static debug-like labels.
- Special NPCs may use custom trade rules instead of core-villager affection rules. For example, `lizard_merchant` can buy sellable items from the player inventory and `storage_box`, and sells only combat tools that are not ammo or equipment.
- Special NPCs can opt out of normal generated menus. For example, `talkative_cat` is displayed as the silent black cat, uses `npcType: "special_forest_npc"` and `socialRules.canUseNormalTrade: false`, so normal affection, chat, gift, trade, and quest menu generators must exclude it.
- Special NPCs do not use normal affection state. If their dialogue records include `conditions.affectionMin`, ignore that condition instead of checking `save.villagers[npcId].affection`.
- Keep `flags/flag-sources.json` updated when adding a new player flag. The table should name the source scene/quest/interaction, the content unlocked by the flag, and a Chinese `description` for non-player-facing lookup. If the flag should be shown to players, set `playerVisible: true` and provide a player-facing `playerDescription`.
- Keep event, quest, and flag ownership separate. `events/events.json` owns triggerable scenes and immediate page effects; `quests/quests.json` owns trackable objectives and turn-in rewards; `flags/flag-sources.json` owns source lookup, replay locks, and unlock records after content has happened.
- Use events plus flags for one-time narrative beats. Use quests when the player needs an active objective, count, turn-in, or sidebar tracking. Do not use flags as a replacement for quest progress.
- Keep `npc-interactions/npc-interaction-rules.json` updated when adding a new NPC type or changing whether an NPC can chat, trade, give quests, receive gifts, use affection, or provide special information.
- Use `save.player.dailyFlags` for per-day social locks. The first version reserves `chat_affection_gained:{villagerId}` for the daily `+1` chat affection cap and `gift_given:{villagerId}` for the daily one-gift-per-character limit.
- Keep relationship text in `relationships.json` only.
- Keep `game-design.md` for rules and decisions; keep complete data in JSON files.
- Prefer English IDs and references over duplicated prose when a value can be resolved from another table. Cross-table references should use fields such as `locationIds`, `targetVillagerId`, `returnSceneId`, `facilityId`, and `itemId`; Chinese names stay in display fields such as `name`, `location`, `label`, and prose descriptions.
- Keep location-level interactable objects in `facilities.json`; do not promote them into full locations unless they need their own exits and free movement.
- Crafting recipes belong in `recipes/recipes.json`, not inside `facilities.json` or item records. Match each recipe to a workstation with `workstationFacilityId`; the facility only declares that it is a crafting station.
- Recipes that can be made from the `道具 > 製作道具` menu without a workstation must set `portableCrafting: true`; keep this limited to simple handwork such as sharp stones and arrows.
- Whether the player has learned a recipe is runtime/save state, not planning-template state. Recipe templates may declare `learnedByDefault`, while the actual learned list belongs in `save.player.knownRecipeIds`.
- `alchemy_table` and `workbench` are the first crafting-station ids. Use them as `workstationFacilityId` values instead of duplicating recipe lists inside facility descriptions.
- The base wood material is now `wood` / `木材`; do not introduce a parallel `wood_bundle` item id again unless the design later needs a separate bundled form.
- Consumables may declare `useReturnsItems` when using the item should leave a reusable remainder, such as `small_healing_potion` returning `empty_potion_bottle`.
- Exploration forage pools belong in `exploration/forage-loot.json`. The forage command should read item-count weights and region loot candidates from there instead of hard-coding region drops in command definitions.
- Temporary exploration resource nodes belong in `exploration/resource-node-spawns.json`. These are not the same as village fixed facilities even when they reuse the same facility ids or behavior templates.
- Facilities that give items to the player should use `facilityType: "small_storage"` with `smallStorage.canDeposit: false` and `smallStorage.canWithdraw: true`. Their current `items` live in save data. Daily restock follows `smallStorage.restockRule`; one-time treasure objects use `smallStorage.initialItems` plus `smallStorage.oneTimeClaimFlag` so the claimed state is recorded in `player.flags`.
- `scrap_pile` / 廢料堆 is the fixed workshop-storage source for `cloth_strip` / 布條. It restocks `cloth_strip x2` daily and should be presented as rummaging through usable scraps, not as a storage container.
- `crow_nest` is the fixed graveyard source for `bird_egg` or `feather`. It randomly restocks one item daily and should be presented as rummaging through a nest, not as a storage container.
- `舊箱子` is a reusable one-time treasure pattern, not a dormitory-only object. Add each instance as its own facility with its own location, description, `initialItems`, and claim flag. Use `smallStorage.withdrawAllAtOnce` for a single take command and `smallStorage.overflowToDroppedItems` when overweight leftovers should become dropped items at the current location.
- Facilities that can be gathered infinitely should use `facilityType: "infinite_source"` with `infiniteSource.itemId`, `count`, `timeCostSeconds`, and `label`. They do not use `smallStorage.items` or daily restock.
- `supply_box` is a village-square small storage facility. It randomly restocks two consumable combat tools from `trap`, `torch`, and `capture_net` at `06:00`, but skips restock while its total item count including stacked quantities is already `6` or more.
- Do not use a separate village resource table. Wood, herbs, ore, food, and similar materials are items, quest state, or container contents.
- Use `lost_and_found_box` for recoverable expired dropped items. When `discarded_items` expires, `equipment` and `quest_item` entries move there; other categories disappear.
- `quests/quests.json` is a read-only quest template table, not save data. Player quest progress belongs in the exported/imported save JSON.
- Every item must include a `price` object using `fluorite` as currency. Non-sellable items keep `price.sellable: false`; story-critical, quest, clue, and equipment items are usually non-sellable.
- Dialogue scenes should be menu entry records, not hard-coded lists of `chat/gift/trade/quest/leave` choices. Use `baseCommandSet: "dialogueMenu"` and generate actual menu availability from command rules, quest prerequisites, trade rules, and NPC type.
- Triggered narrative events belong in `events/events.json`, not in `locations.json` or dialogue-menu scaffolding. Use event triggers for things like entering a place, talking to a specific NPC, moving toward a specific location, or gated one-time scenes; let each event page own its text and state changes.
- Event trigger types currently used by the browser runtime include:
  - `sceneEnter`: checked when the current scene is a location or story/exploration scene.
  - `villagerInteract`: checked when entering a generated character interaction page such as `dialogue:aida:village_square_hub`.
  - `villagerInteractionLeave`: checked only when the player presses `返回` from a character interaction page back to its parent location. Use this for one-time affection or trust events that should happen after the player has seen the latest chat, gift, quest, treatment, or other interaction response.
  - `facilityInteract`: checked when opening facilities that point to an event pool, such as notice-board messages.
  - `moveToLocation`: checked by move-before events declared through location exits.
- Do not use post-action affection events that interrupt `chat` or `gift` immediately after the action. If a trust event depends on affection raised by conversation, gifts, quests, treatment, or later social actions, give it `villagerInteractionLeave` plus a `villagerAffectionMins` condition and a permanent seen flag. This keeps the direct interaction response visible first, then plays the trust event when the player chooses to leave the character page.
- If a `villagerInteractionLeave` trust event does not declare `timeBlocks` or a trigger `sceneId`, write it as location- and time-neutral. The event may happen anywhere the character can be interacted with, so avoid fixed props or places such as herb shelves, field seedlings, workshop benches, dormitory roll calls, specific daylight, or night-only atmosphere. Prefer portable openings like being called back before leaving, a pause after the latest interaction, or the character reflecting on repeated recent changes.
- Location exits may use `eventBeforeMove.eventId` to start a move-prevention or move-confirmation event before normal travel. The event choices decide the actual `targetSceneId`, so the player can continue, turn back, or be redirected without hard-coding that behavior in the location table.
- Runtime-only values such as facility level, storage box contents, small-storage contents, trade stock, quest status, and daily reset tracking must be saved in exported/imported save JSON. Do not write runtime progress back into planning JSON files.
