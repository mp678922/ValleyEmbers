# ValleyEmbers 介面設計規格

企劃版本：`interface-ui@2026-05-17.02`

本文是主程式的 UI 合約。主程式實作介面時，應優先遵守本文件的畫面結構、資料來源、互動元件與狀態呈現規則；具體遊戲資料仍以 `docs/data/*.json` 為準。

## 目的

- 讓主程式可以照同一份介面文件設計畫面，不因資料表擴充而反覆改 UI 架構。
- 避免在介面中硬寫舊選項、舊地點名、舊資源欄位或舊 NPC 規則。
- 讓玩家能清楚看見目前場景、可互動對象、可用設施、背包、存檔、時間與每日狀態。

## 基本原則

- 介面只顯示資料生成後的結果，不在畫面層硬寫 `chat/gift/trade/quest/leave`、設施按鈕、角色按鈕或地點出口。
- 跨表匹配一律使用英文 `id`，例如 `locationIds`、`facilityId`、`itemId`、`villagerId`、`returnSceneId`。
- 中文文字只用於顯示欄位，例如 `name`、`label`、`description`、`location`。
- 玩家可見的 `description`、提示與說明文字，預設應以主角當下能感受到或會如何理解的角度書寫，優先維持沉浸感。
- 不要在玩家可見描述中直接解說內部系統，例如每日 `06:00` 重置、固定生成數量、隱藏判定、解鎖邏輯、資料表來源或 runtime 狀態結構；這些只保留在規格、程式或除錯資訊裡。
- 同一畫面應先顯示「目前可做什麼」，再顯示狀態細節。不要把玩家需要點擊的行動藏在說明文字裡。
- 介面只操作 save data，不把 runtime 狀態寫回規劃 JSON。
- 不常駐顯示村民好感與全村設施狀態；這些資料只在對話、交易、委託、設施互動或除錯介面需要時局部呈現。
- 介面企劃與程式都必須有版本編號。版本可保存在程式常數、save data 或可檢查資料中，方便比對差異；玩家狀態區不顯示版本。

## 介面術語

主畫面主要區塊固定使用以下術語。後續企劃、回報問題與程式註記都應優先使用這組名稱，避免混用 `資訊欄`、`結果框`、`場景文字`、`行動列表` 等舊稱。

- 主介面：左側整個主要遊戲區塊，程式節點為 `scenePanel`，包含日期與時間、導覽、小標題、標題、主文字區塊、設施狀態區塊、狀態變化區塊與指令區塊。
- 導覽：麵包屑位置，對應畫面紅色圈選區，程式節點為 `sceneLocation`；事件播放期間與開場命名介面暫時隱藏。
- 小標題：標題上方的短標籤，程式節點為 `sceneSubtitle`。人物互動頁用來顯示角色身分，例如 `藥草師`。
- 標題：主場景大標題位置，對應畫面綠色圈選區，程式節點為 `sceneTitle`。
- 主文字區塊：場景敘事、人物回應、事件正文或操作後的敘事回饋位置，對應畫面藍色圈選區，程式節點為 `sceneDescription`。
- 設施狀態區塊：只在進入單一設施互動頁時顯示，位於主文字區塊下方、狀態變化區塊上方，程式節點為 `facilityStatus`。只顯示該設施目前最低限度狀態，不放任何互動。
- 狀態變化區塊：主文字區塊下方、指令區塊上方的位置；設施頁若顯示設施狀態區塊，則位於設施狀態區塊下方。對應畫面黃色圈選區，程式節點為 `actionResult`。只顯示上一個動作造成的實際影響或失敗原因，不顯示靜態場景資訊。
- 指令區塊：所有可點選行動、人物、物件與移動分組的位置，對應畫面褐色圈選區，程式節點為 `choiceList`。

## 主畫面結構

主程式第一版使用單頁遊戲介面，分為左側主介面與右側側邊狀態區。主介面可依語境稱為主要場景區，但若指整個左側區塊，統一稱作主介面。

```text
┌──────────────────────────────┬────────────────────┐
│ 主介面                         │ 側邊狀態區          │
│                              │                    │
│ 日期 / 時間                    │ 生命 / 體力          │
│ 導覽                          │                    │
│ 小標題                        │                    │
│ 標題                          │                    │
│ 主文字區塊                    │ 背包                │
│ 設施狀態區塊（設施頁限定）    │                    │
│ 狀態變化區塊                  │                    │
│ 指令區塊                      │ 存檔操作            │
└──────────────────────────────┴────────────────────┘
```

手機或窄畫面時，側邊狀態區移到主介面下方，指令區塊仍應出現在狀態詳情之前。

桌機雙欄版面中，主介面與側邊狀態區的高度不得互相拉伸；兩欄都應依自身內容高度排列。

## 主要場景區

主要場景區必須包含下列欄位：

- `sceneTimebar` / 日期與時間：顯示 `第 N 天` 與 `HH:MM + timeBlock`，位置固定在導覽上方；事件播放期間與開場命名介面暫時隱藏。
- `sceneTimebar` 背景需依目前時段變化，例如夜晚使用深藍色；文字顏色必須與背景保持高對比，確保日期與時間清楚可讀。
- `sceneLocation` / 導覽：顯示目前互動階層麵包屑，例如 `山谷村 > 村莊廣場 > 托莉`。進入更深層介面時可繼續顯示 `交易`、`送禮`、`道具`、`設施` 或 `委託` 等目前所在層級；事件播放期間與開場命名介面暫時隱藏。
- `sceneSubtitle` / 小標題：可選短標籤，位置在標題上方。人物互動頁使用角色身分，例如 `藥草師`；沒有小標題時不顯示。
- `sceneTitle` / 標題：顯示目前場景標題。
- `sceneDescription` / 主文字區塊：顯示場景描述、人物回應、事件正文與操作後的敘事回饋。不要把數值變化、物品增減、人物清單、設施清單或出口清單塞進主文字區塊。
- `facilityStatus` / 設施狀態區塊：只在玩家點進單一設施後顯示；一般場景頁不顯示，也不列出全場景設施狀態。此區只用欄位式短句顯示該設施目前狀態，例如 `等級：1 / 5`、`狀態：可摘取`、`狀態：蘋果已被摘取`、`容量：4 / 20`。不得顯示刷新時間、每日生成數量、抽選規則、內部 flag、資料來源或推測性階段文字。所有互動一律留在 `choiceList`。
- `actionResult` / 狀態變化區塊：位置在主文字區塊與指令區塊之間；設施頁若有 `facilityStatus`，則在 `facilityStatus` 下方。只在最新動作造成可顯示影響或失敗時顯示；沒有最新動作影響時隱藏，不用來常駐顯示目前場景、人物或設施資訊。
- `adventureStatus`：冒險區專用資訊列，只在冒險區顯示，取代原本的大標題位置，固定回報探索區域、深入程度與當前危機感知，不被上一個行動結果覆蓋。夜晚或深夜位於冒險區時，不安敘述不放在冒險狀態列，而是放進主文字區塊，以紅字顯示在場景敘事之後、黑貓現身或離開資訊之前；文案不得明說黑影、機率、替換規則或具體敵人。
- `choiceList` / 指令區塊：顯示分組後的目前可互動項目。

事件頁面規則：

- 事件沿用主畫面場景區播放，不另外做獨立視窗。
- 事件頁的標題顯示事件名稱，主文字區塊顯示當前頁文字。
- 無分支事件頁只顯示單一 `下一頁` 或 `結束`。
- 有分支事件頁使用 `選擇` 分組顯示選項。
- 事件頁選項可接到另一個事件，也可指定 `targetSceneId` 結束事件並前往指定地點；不得直接跳到同事件內任意頁。
- 事件頁選項若指定 `targetSceneId`，可同時帶有 `timeCostSeconds`、`effects`、`resultMessage` 與 `progressLabel`，用於移動確認、折返、付出代價後改變目的地等結果。
- 事件頁或事件選項可指定 `transition: "fade"`，表示從目前頁面切到下一頁、另一事件或指定場景前播放全畫面淡入淡出過場。事件最後一頁可指定 `finishTransition: "fade"`，表示結束事件並返回或接到後續事件前播放過場。開場、昏迷、醒來、長時間養傷等過場可使用這些欄位。
- 事件頁的道具、生命、技能、好感與旗標變化在該頁出現時立即套用。
- 事件結束後回到觸發它的原場景；若最後一頁有可見狀態變化，可保留在返回後的狀態變化區塊。
- 看板等設施若提供已看過事件回顧，回顧入口顯示在該設施頁的獨立分組。回顧只重播事件文字，不重新套用事件效果、每日旗標、時間、獎勵或狀態變化，也不得讓回顧占用每日新留言觸發次數。

主角命名規則：

- 新遊戲開場事件會在艾妲詢問名字後進入命名介面，命名介面仍使用主畫面結構，不另外彈出視窗。
- 主角預設名稱為「旅人」，命名介面初始顯示此名稱，玩家可在開場改成其他名字。
- 命名介面在指令區塊顯示單行輸入與確認按鈕；名字最多 12 個字，送出空白名字時在狀態變化區塊提示失敗原因。
- 確認後名字寫入 save data 的 `player.name`。後續主文字區塊文案可用 `{playerName}` 或 `{protagonistName}` 作為佔位文字，由程式顯示時替換為主角名字；舊存檔缺少名字時補用「旅人」。
- 對話框內若角色呼喚主角，可以自然加入主角名字；不要為了展示變數在每句對白都重複呼喚。

## 狀態變化區塊

狀態變化區塊只承載「上一個動作造成的實際影響」或「操作失敗原因」。它不是場景資訊欄，也不是敘事欄；打開頁面、進入子選單、查看人物或查看設施時，不應因為頁面本身有資訊就顯示狀態變化區塊。

狀態變化區塊用來記錄上一個動作影響了哪些事，例如自身狀態變化、物品變化、場景物件狀況、場景人物狀況、時間變化、位置變化、螢石變化、好感變化、設施狀態改變或交易庫存變化。旗標預設屬於內部進度資料，不顯示 raw flag id；只有 `flags/flag-sources.json` 明確標記 `playerVisible: true` 的旗標可在一般狀態變化區塊顯示，且只能顯示 `playerDescription` 或中文說明，不顯示旗標 id。NPC 互動的回應文字不應塞進狀態變化區塊。

位置：

- 放在主要場景區的主文字區塊與指令區塊之間；設施頁若顯示設施狀態區塊，順序為主文字區塊、設施狀態區塊、狀態變化區塊、指令區塊。不取代存檔訊息。
- 主文字區塊保留環境、對象本身敘事、NPC 互動後的回應文字，以及操作後的故事性回饋。角色外觀短句、委託提示、設施容量、深入程度、當前危機感知、交易持有螢石、丟棄後負重等若只是查看資訊，不得放入狀態變化區塊；需要顯示時應整合到主文字區塊、指令區塊標題、按鈕成本、側邊狀態或專用介面。
- 側邊狀態區順序為：狀態、背包、存檔。
- 若上一個動作沒有造成可顯示變化，狀態變化區塊可隱藏。

觸發條件：

- 真正執行行動後顯示，例如交談、送禮、交易結算、使用道具、等待確認、睡覺、取出設施內容物、採集、探索、移動、委託回報。
- 交談後若同時產生時間或好感變化，狀態變化區塊只顯示數值變化；對方回應顯示在人物頁主文字區塊，不得塞進狀態變化區塊。
- 單純打開選單、進入下一層、返回上一層、調整交易草稿、展開/收合背包、打開道具說明視窗，不更新狀態變化區塊。
- 單純刷新目前頁面、打開子頁、返回或調整草稿等沒有完成正式行動的操作，通常應清除上一個狀態變化區塊，避免舊行動結果殘留在新畫面。
- 行動失敗或條件不足時，訊息框顯示失敗原因，但不得套用任何狀態變化。

## 主文字區塊

主文字區塊的任務是提供視覺、情境與對話情報，讓場景、人物、事件與操作回饋具有故事性、易讀性與沉浸感。所有行為若需要敘事回饋，優先放在主文字區塊，而不是塞進狀態變化區塊。

玩家可見主文案的完整敘事準則集中記錄於 `docs/narrative-guidelines.md`。本節描述介面呈現責任；撰寫或審查實際文案時，應同時依該準則檢查沉浸感、具體度、角色演出、村莊人口限制與換行節奏。

寫作規則：

- 主文字區塊最優先維持沉浸感。只要資訊放進主文字區塊，不論是場景說明、場景物件、角色台詞、道具說明、事件正文、戰鬥情報、敵人外觀、戰勝可能、敵人弱點回想或操作後回饋，都必須先被寫成主角當下的觀察、感受、判斷或角色自然說出口的話。主文字區塊不是資料欄、報表或百科條目；必要情報可以明確，但不得用欄位排列取代敘事。
- 使用主角主觀視點，預設以「你」稱呼主角，敘述主角看見、聽見、感覺到、理解到的內容。
- 用說故事的方式描述互動，不把玩家操作後的敘事回饋寫成系統報告。
- 盡量不要顯示數值、機率、內部欄位、重置時間、資料表規則或明確系統影響。
- 玩家可見的敘事文字不得寫成欄位列表或資料卡，例如 `墓碑資訊`、`銘文`、`撰文者`、`用途`、`狀態`、`來源`、`管理者`、`效果`、`取得方式`、`角色定位`、`好感需求`。這條適用於場景物件描述、道具說明、角色台詞、事件正文、知識對話、送禮反應與操作後敘事。玩家點擊墓碑、紀念物、舊物、遺跡痕跡、看板、箱子、床、工作台、藥草棚器具、田地、水渠、路標或其他可互動物件時，主文字區塊應以主角看見的材質、磨損、擺放方式、字跡、氣味、聲音、使用痕跡與當下聯想來組成敘事。道具說明應描述主角拿在手上的重量、質地、氣味、用途聯想或過往使用痕跡；角色台詞應用角色自然語氣把資訊說出來，不把話拆成條列報告。若需要玩家知道撰文者、用途、管理者、過往使用者、隱含背景、道具來源或功能方向，也應由字跡、措辭、留下的物件、角色習慣、主角已知經驗或角色口吻自然帶出。只有真正的操作介面、狀態摘要、Debug、背包、倉庫、交易、製作清單、可用按鈕列表或狀態變化區塊可以使用列項。
- 所有玩家可見主文案都要先讓玩家聽懂核心意思，再考慮含蓄、詩性或弦外之音。這條不只適用於角色台詞與事件正文，也適用於場景說明、場景物件、道具說明、知識對話、戰鬥情報、敵人外觀、操作後回饋與好感事件。主文字區塊不應連續使用抽象比喻、話中有話或高概念句子來替代具體資訊；重要內容應直說，例如危險原因、物件用途、角色擔心、主角被要求做的事、角色為何信任主角、或村莊哪裡確實變好了。若一句話只留下氣氛，卻不能判斷實際狀況、角色意圖或操作線索，應改寫。
- 角色對白要在直觀之外保留角色演出。主文字區塊顯示台詞時，應依角色設定調整句長、停頓、反應速度、稱呼方式和語氣詞；可適時使用「嗯」「唉」「嘖」「喂」「欸」「……」等狀聲詞或語氣停頓，讓角色更像真人在當下反應。這些語氣元素只能少量使用，並且要符合角色性格與情境；不得把每個角色都寫成同一種解說口吻，也不得讓語氣詞取代具體資訊。
- 村莊場景的敘事必須符合山谷村目前只剩六位女孩的總人口設定。主文字區塊不得把村莊廣場、宿舍、開墾區或其他村內地點寫成有路人、人潮、泛泛的村民群眾或熱鬧市集；若寫「大家」，語意必須明確指六位女孩或當下實際在場的少數人。村莊的生活感應來自空位、舊物、遠處一兩個熟悉聲音、被維持的習慣與分頭工作的痕跡，而不是人口很多。
- 剛到達一個場景時，主文字區塊應顯示「場景印象情報」，也就是主角此刻對地點的視覺、聲音、氣味、光線、空曠感、在場者氣氛或危險感的觀察。場景印象情報不是固定場景介紹；主程式可依時間、地點、旗標、每日旗標或其他條件，從 `locations[].impressionInfos`、`scenes[].impressionInfos` 或探索場景的 `sceneryDescriptions` 提取不同敘事。沒有命中條件印象時，才使用原本 `description`。
- 初次進入人物互動頁時，主文字區塊應顯示「印象情報」，也就是主角此刻對這位人物的稱謂、神態、外觀、所在位置與細微狀態的觀察。印象情報不是固定人物介紹；主程式可依時間、地點、旗標、每日旗標、好感或其他條件，從 `villagers[].impressionInfos` 或 `villagers[].appearance.impressionInfos` 提取不同敘事。沒有命中條件印象時，才使用 `appearance.presenceSummary` 或其他外觀欄位產生保守 fallback。
- 印象情報只作為進入人物頁的初始敘事，後續交談、送禮、委託或角色功能回應會覆蓋它，不應和回應文字並存。
- 後續操作後，主文字區塊應改為該操作的情境回饋；不要讓初次到達場景的資訊在每次行動後持續殘留成多餘內容。
- 若操作同時產生故事回饋與實際影響，故事回饋放主文字區塊，實際影響放狀態變化區塊。
- 若資訊是為了幫助玩家操作但不屬於故事敘事，例如交易持有螢石、整理後負重、設施容量，優先放在指令區塊標題、按鈕成本或專用整理介面，不放在狀態變化區塊。
- 主文字區塊的換行以閱讀節奏為優先，不以句號機械切分。程式不得在每個中文句號後自動插入換行；瀏覽器自然換行負責處理行寬，資料作者負責用 `\n` 或 `\n\n` 標出真正需要分開的語意段落。
- 換行應服務於「玩家如何理解這段文字」：同一個動作、同一則留言、同一個角色連續說出的短句，應保持在同一段內；視角切換、時間推進、動作結果、另一個角色接話、敘事轉折或情緒停頓，才適合另起一段。
- 對話、看板留言、墓碑字跡、紙條、知識問答與戰鬥回饋都應避免被自動拆成零碎短行。若需要呈現多則留言，應以「一則留言一段」為單位分段，例如 `米菈回：「有。不要問。」` 應維持為同一則留言，而不是在 `有。` 後切成下一行。
- 長段落應由文案本身重寫或人工分段，而不是靠程式逐句斷行。一般主文字段落建議維持 1 到 4 句；超過這個長度時，優先檢查是否存在動作、視角、資訊焦點或情緒節奏的自然分段點。

## 狀態變化區塊格式

內容格式：

```text
時間：經過 1 分鐘
體力：7 → 10（+3）
物品：乾糧 -1（目前 0）
位置：移動到工坊倉庫
```

顯示規則：

- 不顯示行動名稱列，只顯示實際變化或失敗結果。
- 數值變化使用「前值 → 後值（差值）」格式，例如 `生命：6 → 8（+2）`。
- 若主角實際所在位置改變，顯示 `位置：移動到地點名`。
- 物品變化使用「道具名稱 +N / -N（目前 X）」格式，例如 `螢石 -4（目前 6）`、`小治療藥 +1（目前 1）`。
- 多個物品同時變動時，以逗號或分行列出，不塞進主文字區塊。
- 沒有變化的欄位不顯示，例如生命沒有變就不顯示生命列。
- 時間若有推進，顯示實際經過時間；若跨日，顯示 `時間：經過 6 小時，進入第 2 天 06:00`。
- 好感變化格式為 `米菈好感：3 → 5（+2）`。
- 交易結算需至少顯示螢石、買入物品、賣出物品與耗時。
- 探索失敗若造成物品遺失，必須逐項列出實際遺失道具，不只顯示「部分道具遺失」。
- 每次新行動完成後覆蓋上一則訊息；第一版不需要完整歷史紀錄清單。

資料來源與計算方式：

- 主程式在套用行動前先建立狀態快照，包含 `player.life`、`player.stamina`、`player.inventory`、`time`、`villagers`、`facilities`、`tradeStocks`、`flags`。
- 行動套用完成後，用前後快照計算差異，生成 `lastActionResult`。
- `lastActionResult` 是 runtime / save data，可寫入本機存檔；畫面只在最新行動剛產生可顯示變化或失敗時顯示，不因切換選單長期殘留。
- 不得把 `lastActionResult` 寫回 `docs/data/*.json`。

建議 save data 形狀：

```json
{
  "lastActionResult": {
    "actionId": "use_dry_ration",
    "label": "使用乾糧",
    "status": "success",
    "timeDeltaSeconds": 60,
    "changes": {
      "life": null,
      "stamina": { "before": 7, "after": 10, "delta": 3 },
      "items": [
        { "itemId": "dry_ration", "name": "乾糧", "delta": -1 }
      ],
      "affection": [],
      "flags": [],
      "facilities": []
    },
    "message": "你吃下乾糧，體力恢復。",
    "sceneDescription": "你把乾糧慢慢嚥下，讓身體重新找回一點力氣。"
  }
}
```

視覺規格：

- 狀態變化區塊應比主文字區塊更像狀態回饋，不使用大標題。
- 可用淡色邊框或淺底色區分，但不得做成搶過主要行動按鈕的提示框。
- 成功、失敗、警告可用不同細節色，但不要依賴顏色作為唯一辨識；文字必須清楚。
- 窄畫面時側邊狀態區移到主介面下方；主介面內的主文字區塊、設施狀態區塊、狀態變化區塊與指令區塊仍保持同一順序。

## 指令區塊

到達一個場景時，主要場景區下方顯示指令區塊。地點場景的固定分組為：

```text
行動
  道具        等待

看見的人物
  米菈        托莉
  藥草師      開墾管理人

看見的物件
  補給箱      蘋果樹      看板

移動到
  宿舍        藥草棚      工坊倉庫
```

分組規則：

- `行動`：來自目前場景的 `baseCommandSet` 與 `localActions`，例如道具、等待、探索、採集、返回村莊。
- `道具` 指令應永遠可進入，不因目前沒有可用效果而在上一層被阻擋。
- 宿舍夜晚可出現期間限定地點行動 `參與紡織工作`。入口只在 `夜晚` 顯示，確認後啟動 `docs/data/events/events.json` 中的宿舍紡織事件；事件第一頁只播放演出開頭，第二頁收束演出並耗時 `30` 分鐘、結算獎勵。主文字區塊顯示女孩們彼此拌嘴、玩笑、關切主角，或對主角展現好奇的互動，主角只用動作、眼神或敘事概述回應，不出現主角明確台詞。紡織演出固定至少 `6` 段，抽選時採一輪六段制：同一輪內不重複，六段都播過後才重置，新一輪第一段仍避免與上一段相同。結果隨機取得 `繩索 x1` 或 `布條 x1`，實際物品與時間變化放在第二頁後的狀態變化區塊。
- 像工坊倉庫這種已經能從 `看見的物件` 直接進入工作台、倉庫箱或木材堆的地點，不應再額外放一個泛用的地點指令，例如 `整理工坊倉庫`。
- `看見的人物`：來自目前 `location.id` 與 `timeBlock` 匹配出的 NPC。按鈕主文字顯示人物名稱，副標顯示 `villagers[].role` 身分，點擊後進入該角色的下一層互動選單。
- `看見的物件`：來自 `facilities[].locationIds` 匹配出的設施或物件。這裡只顯示物件名稱，點擊後進入該設施的下一層選單。
- `移動到`：來自 `locations[].exits` 或場景內的移動型選項。

空分組處理：

- 如果某分組沒有項目，該分組不顯示。
- 如果目前指令區塊只剩一個一般指令分組，不顯示按鈕上方的類別小字，例如事件只剩 `繼續` 分組時不顯示 `繼續`；有多個分組時才顯示 `行動`、`看見的人物`、`看見的物件`、`移動到` 等類別小字。
- 如果整個場景沒有任何可選項目，顯示一個返回或等待類 fallback 行動。

項目按鈕格式：

```text
行動文字                         耗時
```

項目按鈕必須支援：

- 不可選狀態：條件不足時 disable，但仍可看見。
- 耗時顯示：無耗時顯示 `不耗時`，有耗時顯示分鐘或小時。
- 若該行動會直接消耗體力，按鈕右側必須一併顯示體力需求，例如 `30 分鐘 / 需體力 1`。
- 來源可追蹤：每個按鈕應保留內部 `id`、`actionType`、必要目標 id。

體力需求規則：

- 會直接消耗體力的行動應明確附帶體力需求。
- 體力不足時，這類行動原則上應直接 disabled，並提示需求與目前體力。
- 若未來有低體力仍可執行的特殊例外，必須明確在規則上標示，不沿用一般行為。

## 附帶介面行動流程

若一個指令需要先進入附帶介面才能完成，例如整理草稿、選擇時間、選擇數量、多步驟確認或結果整理，主程式必須把它視為 `pendingActionFlow`，不得把入口當成正式行動。

共同規則：

- 入口只建立流程狀態與必要草稿，不消耗時間、體力、道具或其他成本，也不播放假讀條。
- 流程狀態至少要能記錄 `id`、`type`、`sourceSceneId`、`interfaceSceneId`、`returnSceneId`、`cancelSceneId`、`confirmAction` 與草稿識別。
- 附帶介面必須有返回或取消路徑；返回只取消草稿並回到指定上一層，不套用效果、不觸發完成敘事、不讓 NPC 離開、不消耗成本。
- 只有確認按鈕才算真正執行指令；確認時才套用 `confirmAction.timeCostSeconds`、體力或道具成本、假讀條、狀態變化、完成敘事與完成後場景。
- 讀條必須綁定確認按鈕，而不是綁定入口按鈕；若確認行動原本需要時間或特殊演出，確認按鈕必須保留對應 `progressLabel`、`progressScope` 與成本顯示。
- 調整草稿、切換頁籤、返回上一層或取消流程通常應清除舊狀態變化區塊，避免玩家誤以為行動已完成。

第一版流程類型：

- `draft`：交易、倉庫、黑貓帶回東西、戰利品整理、遺落道具整理。
- `selection`：等待時間、睡覺時間、數量選擇。
- `confirmation`：高成本或高風險行動的最後確認。
- `resultReview`：戰鬥結果、搜尋結果等已產生結果但還需要整理的介面。
- `nestedFlow`：未來需要選目標、選道具、再確認的多步驟指令。

## 行動中狀態

玩家選擇會推進時間的選項後，主程式在套用行動結果前，必須短暫顯示正在執行的狀態。

觸發條件：

- `timeCostSeconds > 0` 的行動。
- 會呼叫時間推進邏輯的動態行動，例如等待、睡覺、採集、移動、治療、交易以外的耗時對話。

不觸發條件：

- `timeCostSeconds` 為 `0` 或未設定。
- 單純進入下一層選單，例如點擊人物、點擊設施、打開背包、返回。
- 匯出/匯入存檔。

顯示規格：

```text
正在移動到宿舍...
[ fake progress bar 0.3s ]
```

介面行為：

- 顯示文字使用玩家剛選的行動 label，例如 `正在等待...`、`正在睡覺...`、`正在前往森林外圍...`。
- 顯示一條假的進度條，動畫時間固定 `0.3` 秒。
- 進度條期間暫時禁用所有其他按鈕，避免重複點擊。
- 進度條期間只保留導覽、標題與讀條；標題以下原本的主文字區塊、狀態變化區塊與指令區塊都先隱藏。
- `0.3` 秒結束後才套用效果、推進時間、處理每日重置、重新渲染場景；第一版改為手動存檔，這些變更不應自動寫入本機存檔。
- 如果行動失敗或條件不足，不播放進度條，直接顯示錯誤訊息。

過場淡入淡出：

- 快速跨越長時間的行動，例如等待或睡覺達 `1` 小時以上，使用全畫面淡入淡出過場，不只顯示一般 fake progress bar。
- 過場先淡入遮住主畫面，再套用時間推進與場景重新渲染，最後淡出回到更新後畫面。
- 過場不顯示文字，只用畫面淡入淡出表現時間跨越。
- 過場期間禁用其他指令，避免重複點擊造成時間或狀態重複套用。

## 側邊狀態區

側邊狀態區第一版固定包含五個區塊，順序為：狀態、委託、技能、背包、系統。

- 側欄區塊都應支援收合。
- 預設只有 `狀態` 展開；`委託`、`技能`、`背包`、`系統` 預設收合。
- 可收合區塊的狀態文字顯示為 `[展開]` / `[收合]`，不使用純箭頭符號。

### 狀態

顯示玩家狀態。日期與時間不放在側邊狀態區，固定移到主介面的導覽上方：

- 生命：以血條視覺顯示 `life / maxLife`，條內比例等於目前生命百分比；數字仍需直接可見，不能只靠顏色判讀。
- 體力：以體力條視覺顯示 `stamina / maxStamina`，條內比例等於目前體力百分比；數字仍需直接可見，不能只靠顏色判讀。

### 技能

- 技能改為獨立面板，不再放在 `狀態` 區塊內。
- 面板只顯示目前數值大於 `0` 的技能；若全部技能都是 `0`，整個技能面板不顯示。
- 每個技能列都必須可點擊；點擊後像道具一樣浮出完整介紹視窗。
- 技能介紹視窗至少顯示：技能名稱、目前等級、技能說明、目前規則摘要。

### 委託

- 側欄新增 `委託` 欄位，用來顯示目前已接到的委託。
- 若 `quests.active` 為空，整個 `委託` 欄位隱藏，不顯示空框。
- 每條委託至少顯示：委託名稱、發布者、委託目標與目前進度。
- 委託目標與進度應直接使用玩家可理解的句子，例如 `向諾絲提交 2 份木材（1/2）`。
- 側欄委託欄位不得顯示獎勵內容。

### 背包

顯示玩家背包內所有 `count > 0` 的道具：

```text
螢石 x 10                     總重量 1
```

- 每個背包道具列應直接顯示為「道具名稱 x 數量」，右側再顯示該堆疊目前佔用的 `總重量`。
- 交易、倉庫、遺落道具、採集整理等列式介面中的道具名稱也應可點擊，點擊後開啟與背包相同的道具資訊視窗；調整數量仍由該列右側按鈕負責。
- 背包內容上方提供 `查看已知配方` 按鈕，打開只讀配方視窗，不推進時間、不寫入行動結果。
- 配方視窗只列出玩家已知配方，顯示成品、材料需求、耗時、製作場所；不顯示目前持有數量、材料不足狀態或是否可製作。
- 配方視窗不直接製作道具；真正製作仍走 `道具 > 製作道具` 或對應工作台。

背包資料來源是 `save.player.inventory`，格式為：

```json
[
  { "itemId": "apple", "count": 1 }
]
```

### Debug 工具

`系統` 區塊放在背包下方，合併原本的存檔與 debug 工具入口，並且必須可收合。視覺上要像同一組工具欄位，不要拆成兩塊鬆散面板。

`Debug` 只保留在 `系統` 區塊中的入口按鈕。按下後要像道具說明一樣浮出完整 modal，只作為本機除錯用，不寫回規劃資料。

- Debug modal 不使用右上角 `X`；改為視窗底部固定一顆 `關閉` 按鈕，且該按鈕不得被捲進視窗內容區。
- Debug modal 開啟時主頁不得跟著捲動；modal 內部內容區必須自行提供垂直捲軸，標題與底部關閉按鈕固定在視窗內。

第一版 modal 支援：

- 新增未持有道具：只列出目前不在背包裡的道具，用下拉選單選擇並輸入數量後新增。
- 增加道具時仍必須檢查負重上限；超重時不得加入。
- 配方道具工具：可從下拉選單選擇配方，並列出該配方的成品與所需材料。
- 配方道具清單中的每一列都應顯示目前背包持有數量，方便直接比對缺口。
- 配方清單中的成品與材料都可直接加入背包，加入時同樣必須檢查負重上限；超重時不得加入。
- 已持有道具調整：列出目前背包裡的每個道具，提供 `+` 與 `-` 按鈕逐次調整數量。
- 已持有道具調整列在畫面上應固定對齊為「道具名 / 操作按鈕 / 重量」欄位；`+`、`-` 按鈕不可做成過扁的小條狀。
- Debug modal 內必須直接顯示目前負重狀態；物品本身與堆疊則顯示為 `重量`，不要把兩者混用。
- 若 Debug 的加道具操作被負重限制擋下，畫面必須明確回報失敗原因，不能只靠 disabled 或 tooltip。
- 生命調整：可直接把 `player.life` 設成指定值，範圍限制在 `0 ~ maxLife`。
- 體力調整：可直接把 `player.stamina` 設成指定值，範圍限制在 `0 ~ maxStamina`。
- 旗標刪除：列出目前 `player.flags`，每個旗標都能單獨刪除；Debug 查閱可顯示 `flags/flag-sources.json` 中的中文說明。一般玩家畫面只顯示 `playerVisible: true` 的旗標說明，且不顯示 raw flag id。

這個區塊的操作也應產生 `lastActionResult`，讓玩家能看到實際變更結果。

顯示名稱從 `docs/data/items/items.json` 解析；找不到道具定義時，顯示原始 `itemId`。

背包必須顯示目前負重：

- 顯示格式：`（負重：目前 / 上限）`，例如 `（負重：3.5 / 10）`。
- 目前負重使用 `items[].weight * count` 加總。
- 上限使用 `save.player.maxCarryWeight`，再加上 `carrySkill` 每級提供的 `+5`。
- 玩家可見的負重、容量與道具重量最多顯示到小數點後二位，並去掉多餘的尾端 `0`。
- 找不到道具重量時視為 `0`，但仍保留道具顯示。
- 若目前負重超過上限，負重文字需有警示樣式。
- 這行 `負重` 摘要放在背包標題同一行，不屬於可收合內容；背包收起來時仍要看得到。

背包區塊必須可以收合：

- 點擊背包標題列可展開或收合背包內容。
- 收合後仍保留 `負重` 摘要，只收起道具清單本身。
- 收合背包不算遊戲行動，不推進時間、不寫入 `lastActionResult`。
- 收合狀態只屬於介面狀態，第一版不需要寫入 save data。

道具說明視窗：

- 點擊背包中的道具列，浮出覆蓋頁面的道具說明視窗。
- 視窗至少顯示：道具類型、持有數量、道具名稱、`items[].description`、重量、使用耗時、效果、售價、標籤。
- 視窗不得顯示內部規則欄位：`canGift` / 可送禮、`canDisappearOnExplorationFailure` / 探索失敗遺失、`source` / 來源。
- 道具與技能說明視窗不使用右上角 `X`；改為視窗底部固定一顆 `關閉` 按鈕，且該按鈕不得被捲進視窗內容區。
- 查看已知配方視窗也遵守同一 modal 規則：不使用右上角 `X`，底部固定 `關閉` 按鈕，內容區自行捲動。
- 點擊關閉按鈕、背景遮罩或按 `Esc` 可關閉視窗。
- 打開或關閉道具說明視窗不算遊戲行動，不推進時間、不寫入 `lastActionResult`。

### 存檔

必須提供：

- 重新開始：先跳出確認詢問；只有玩家確認後，才清除目前本機存檔並建立新 `save@v1`。
- 存檔：提供明確的 `存檔` 按鈕，玩家按下後才把目前狀態寫入本機存檔；第一版只使用 `localStorage`。
- 手動存檔範圍至少包含：角色數值、背包、倉庫內容物、村莊固定設施狀態、探索狀態、旗標、委託與交易庫存。
- 匯出記錄檔：下載目前完整 save JSON。
- 匯入記錄檔：讀取 JSON，覆蓋目前本機存檔。
- 存檔訊息：顯示最近一次手動存檔、匯入匯出結果，或目前仍有未存檔變更。
- 存檔區按鈕順序固定為：`存檔` 在最上方，接著 `匯出記錄檔`、`匯入記錄檔`，`重新開始` 放在最後。

匯入失敗時，不得覆蓋現有本機存檔。

### 道具

- `道具` 指令進入道具選單，不因目前沒有可用效果而在上一層被阻擋。
- 道具選單至少包含三部分：可使用的消耗品、`製作道具` 入口，以及最後一個 `丟棄道具` 入口。
- 可使用的消耗品仍需判定「使用後是否真的造成任何狀態變化」；若不會產生變化，該道具按鈕 disabled。
- 在 `道具` 選單中，滑鼠移到可使用道具時應浮出效果摘要，至少包含恢復效果與返還道具等可直接理解的使用結果。
- 這個判定不能只檢查單一生命或體力欄位，必須以整體效果判定，例如生命、體力、返還道具、旗標、設施狀態、探索危險度等。
- 若目前沒有可直接使用的消耗品，道具選單內顯示不可用提示，但 `道具` 入口仍可進入。
- `製作道具` 進入隨身製作介面，功能沿用工作台製作流程，但只顯示 `recipes[].portableCrafting: true` 的簡易配方，例如銳利碎石與箭矢。
- 隨身製作仍需檢查材料、負重、耗時與行動結果；負重必須用「扣除材料並加入成品後」的背包結果判定，不能只檢查成品重量。不能顯示需要工作台、製藥台或其他設施才能完成的配方。
- `丟棄道具` 不建立獨立丟棄頁面；它必須直接開啟當前場景的 `遺落的道具` 編輯介面。
- 若當前場景已經有 `遺落的道具`，`丟棄道具` 入口直接連到同一份地面容器草稿，讓玩家可同時取回既有遺落物或把背包物品放到地上。
- 若當前場景還沒有 `遺落的道具`，`丟棄道具` 入口仍開啟同一個空地面容器編輯介面；只有按下 `確認整理` 且地上有內容時，才真正生成可見的 `遺落的道具`。
- 按下 `確認整理` 後必須回到原場景，不停留在 `遺落的道具` 編輯介面。
- 這個入口沿用 `遺落的道具` 介面的 `地上的道具`、`背包（整理後負重：目前 / 上限）`、`本次整理`、`確認整理` 與 `返回` 規則，不另開第二套草稿邏輯。
- 丟棄完成後，當前場景必須生成一個名為 `遺落的道具` 的設施，所有剛丟下的道具都放在這個設施內。
- `遺落的道具` 設施只顯示在實際丟棄過道具且內容不為空的場景中。
- `遺落的道具` 自丟下當刻起只保留 `1` 小時；到時後整個設施刪除，一般物品消失，裝備與委託道具移到宿舍的 `失物箱`。
- `遺落的道具` 若位於冒險區，主角變換探索位置或離開探索區時也會被清除；清除時同樣套用失物箱規則。
- 探索失敗懲罰若抽中帶有 `explorationFailureLossTarget: "lost_and_found_box"` 的道具，該道具不永久消失，改進入宿舍失物箱；空藥水瓶使用此規則。
- 若當前場景原本就已有 `遺落的道具`，新的丟棄內容直接加入同一個設施內，不另開第二個，並把刪除倒數重設為 `1` 小時。
- 若 `遺落的道具` 內部物品被全部取回，該設施必須立刻刪除，不保留空殼。
- `遺落的道具` 從場景中點擊後應直接進入內容畫面，不再先經過一層 `查看內容` 設施選單。

## 場景類型

主程式至少要支援以下場景類型。

### 故事場景

資料來源：`docs/data/scenes/scenes.json`

用途：

- 開場、劇情、特殊事件。
- 可保留手寫選項，但不能重複手寫通用地點行動或通用對話選單。

### 村莊地點場景

資料來源：`docs/data/locations/locations.json`

主程式進入地點時，必須生成以下分組：

- `行動`：從 `baseCommandSet` 與 `locations[].localActions` 生成。
- `看見的人物`：從 `villagers[].defaultLocations[timeBlock]` 匹配目前 `location.id` 生成。
- `看見的物件`：從 `facilities[].locationIds` 匹配目前 `location.id` 生成。
- `移動到`：從 `locations[].exits` 生成。
- `locations[].exits[]` 可指定 `eventBeforeMove.eventId`，代表玩家按下該出口時先播放移動前事件；事件選項再決定最後要前往 `targetSceneId`、折返原地，或接到其他事件。
- 若移動前事件已完成或條件不成立，該出口回到一般移動流程，依出口的 `to` 與 `timeCostSeconds` 前往目標地點。

顯示順序固定為：

1. 行動
2. 看見的人物
3. 看見的物件
4. 移動到

### 探索場景

資料來源可為 `scenes.json` 或未來拆出的探索資料表。

探索場景最上方先顯示 `移動` 分組，再顯示一般 `行動` 分組。探索場景使用 `baseCommandSets.exploration`，至少支援：

- `exploreDeeper`
- `exploreDeeperSlow`
- `retreat`
- `retreatSlow`
- `forage`
- `wait`
- `useItem`
- `shoutAttractEnemy`
  - 玩家大喊後，當前危機感知先提高 `20%`，再用提高後的危機感知檢定是否遭遇敵人；不再必定遇敵。
  - `大喊` 耗時為 `1` 分鐘。

戰鬥結束後若主角沒有倒下，當前危機感知降低 `30%`，最低不低於 `0%`；逃跑與煙霧彈等脫身行為可使用自己的危險度變化規則。

其中 `向深處移動` 與 `向村莊移動` 目前都應顯示為 `10 分鐘 / 需體力 1`。`向深處緩慢移動` 與 `向村莊緩慢移動` 顯示為 `10 分鐘`，不顯示體力需求，分別讓深入程度 `+1%` 與 `-1%`。

探索場景不可顯示村莊限定指令。

探索主文字區塊顯示目前位置的環境感受，避免固定提及陪同角色或內部規則。若探索場景資料有 `impressionInfos` 或 `sceneryDescriptions`，進入探索位置時依時間、旗標等條件抽一則保存到 `save.exploration.sceneryText`，同一位置重繪時不重新抽，直到向深處移動、向村莊移動或重新進入探索位置。`sceneryDescriptions` 可保留字串陣列；若需要條件，改用物件 `{ id, weight, conditions, text }`。

探索場景可在條件符合時顯示可移動安全地點。第一版 `廢棄小屋` 只在山谷森林外圍且深入程度介於 `45%` 到 `55%` 時，於 `移動到` 分組顯示。進入廢棄小屋後不顯示冒險狀態列，不觸發森林遭遇；可使用道具與等待。離開廢棄小屋時回到森林外圍，並把深入程度固定為 `50%`。

探索狀態資訊列至少要直接顯示：

- 目前森林層級。
- 當前深入程度。
- 當前危機感知。

探索場景中的 `搜尋` 規則：

- 每次移動到新的探索位置後，只能執行一次 `搜尋`。
- 若該位置已經搜尋過，`搜尋` 按鈕需維持可見但 disabled。
- `搜尋` 執行後即使沒有取得道具，也必須在狀態變化區塊顯示結果，例如 `附近沒有找到值得帶走的物資。`，不能只顯示經過時間。
- `搜尋` 發現道具時，先開啟暫時的道具整理介面，不直接把道具塞進背包。
- 搜尋整理介面上方顯示 `新找到的道具`，下方顯示 `背包（整理後負重：目前 / 上限）`，列式操作邏輯與倉庫/遺落道具一致，每次只移動 `1` 個堆疊數量。
- `新找到的道具` 每列按鈕文字為 `放入背包`；背包每列按鈕文字為 `放到遺落道具`。
- 按下 `確認整理` 後才套用背包變化，未放入背包的新道具與玩家從背包放下的道具都會進入目前探索位置的 `遺落的道具`。
- 搜尋整理介面的 `返回` 代表不整理剛找到的道具；剛找到的道具會全部放進目前探索位置的 `遺落的道具`，再回到探索場景。
- disabled 提示文字固定為 `附近已經沒有物品。`

探索場景中的隱藏地點：

- 隱藏地點由探索搜尋揭露，不在未發現前顯示於移動分組。
- 第一版在森林外圍深入程度 `10%` 時執行 `搜尋`，會揭露 `隱藏的洞穴` 並在結果欄提示主角發現被藤蔓遮住的洞口。
- 隱藏地點被揭露後，只有位於對應探索位置時，才在 `移動到` 分組顯示入口。
- `隱藏的洞穴` 是安全地點，等待與整理道具不觸發探索敵人。
- 隱藏地點內的固定設施由 `facilities[].locationIds` 指向該隱藏地點 id；其互動狀態必須保存於 save data，不可像探索臨時資源點一樣每次進入重新生成。
- `隱藏的洞穴` 目前顯示 `礦脈` 與 `土堆`。礦脈採 `small_storage` 每日重置規則；土堆沿用 `infinite_source`。

探索場景中的臨時資源點：

- `莓果叢`、`月露草叢`、`雜草叢`、`碎石堆`、`土堆` 可作為探索中的臨時資源點，顯示在探索場景的 `看見的物件` 分組。
- 這些探索版資源點不是村莊內固定設施本體，而是每次前進或後退後重新抽出的臨時物件。
- 離開探索區後，探索版資源點必須直接清空，不保留到村莊或下一次探索。

探索場景中的 `向村莊移動` 規則：

- `向村莊移動` 不能從任何進度直接回村。
- `向村莊移動` 先降低目前層級的深入程度。
- 只有當目前層級進度已到 `0%` 時，下一次 `向村莊移動` 才能退回上一層。
- 只有位於最外層且進度為 `0%` 時，按鈕名稱改顯示為 `返回村莊`，不再顯示 `向村莊移動`；這個 `返回村莊` 只消耗 `10` 分鐘，不消耗體力。
- 從冒險區主動 `返回村莊` 的落點固定為 `村莊廣場`。
- `向村莊緩慢移動` 在一般位置只讓深入程度 `-1%`，不消耗體力；若位於最外層且進度為 `0%`，不另外顯示第二個回村按鈕。

### 遭遇場景

- 探索中的危險動作若觸發敵人，畫面要切到獨立的遭遇場景。
- 遭遇場景標題固定顯示 `遭遇戰鬥`，主文字區塊需依序整合資訊：先用中標題顯示敵人名稱，不額外顯示 `敵人` 小標籤，再顯示遭遇敘事，接著用一般文字顯示目前是否有搏鬥把握。若還有 Debug 或額外情報，也依序列在同一個主文字區塊的情報列中，避免多個區塊互相搶資訊位置。
- 戰鬥中的主文字情報仍以沉浸感為優先。敵人外觀、壓迫感、戰勝可能、遠程命中、工具是否有效、賽拉的情報與敵人弱點回想，都要寫成主角看見敵人動作後的判斷、身體感受、記憶浮現或角色曾說過的自然台詞。可以直接保留 `命中`、`沒有命中`、`逃脫成功`、`逃脫失敗` 這類核心結果字樣，但不得寫成 `敵人弱點：...`、`目前戰力：...`、`有效道具：...`、`情報來源：...`、`狀態：...` 等欄位式資料。
- 遭遇場景狀態變化區塊不顯示戰勝可能或其他靜態敵人情報；它只在玩家實際採取戰鬥行動後顯示生命、體力、道具、位置或戰利品等變化。
- 敵人外觀與壓迫感文字優先讀取 `enemies[].appearanceDescription` 與 `enemies[].threatDescription`，但它們只屬於主文字區塊的情境敘事，不獨立做成機制欄位。
- 遭遇敘事應保持緊湊，不因每個句號自動拆成多段；敵人出現、外觀、壓迫感與對峙描述之間可用單換行 `\n` 分隔，但不要使用 `\n\n` 造成過大空白。
- 敵人出現時，程式需把敵人 id 記入隱藏存檔欄位 `player.encounteredEnemyIds`；此欄位只供後續知識、敘事或解鎖判斷使用，不在一般介面顯示，也不寫入狀態變化區塊。
- 搏鬥把握文字只判斷目前近身作戰是否可擊敗敵人，也就是背包中的近戰武器與 `徒手` 依目前敵方戰力能否達成 `險勝` 以上；不得把遠程削弱、戰術物品、逃跑、武器選擇暗示或未來機率納入提示。有任一近身作戰選項能達成 `大勝利`、`勝利` 或 `險勝` 時用綠色文字顯示 `你似乎有把握在近身戰勝對方。`，否則用紅色文字顯示 `你目前毫無把握戰勝。`。這段不使用底色、邊框或獨立資訊卡，只維持主文字區塊一般文字格式並改變文字顏色。
- 進入遭遇場景時不得顯示上一個觸發遭遇的 `結果` 欄，以免蓋掉敵人敘事與戰勝可能資訊。
- 遭遇指令分成 `近身作戰`、`遠程作戰`、`戰術物品`、`其他` 四組；`逃跑` 放在 `其他`。
- 遭遇指令的四個分類標題必須直接顯示在介面上；即使該分類目前沒有可用選項，也要顯示分類與空狀態，不得讓分類整組消失。
- 近身作戰不推進時間，但按下後要播放只覆蓋指令區塊的假讀條。老舊的劍進度文字為 `無畏揮砍...`；士兵長槍進度文字為 `衝鋒突刺...`；徒手進度文字為 `奮力搏鬥...`。
- 戰術物品不推進時間，但按下後要播放只覆蓋指令區塊的假讀條。陷阱進度文字為 `佈置陷阱...`；火把進度文字為 `點燃火把...`；補網進度文字為 `張開補網...`；煙霧彈進度文字為 `拋出煙霧彈...`。
- `遠程作戰` 顯示目前持有的遠程武器，並標示所需彈藥數量。獵弓消耗 `箭矢`，投石索消耗 `石頭`。
- 遠程作戰不推進時間，但按下後仍要播放假讀條。這類戰鬥中短讀條只覆蓋指令區塊；讀條期間要隱藏整個指令區塊內的原有指令類別與按鈕，只保留讀條，不隱藏日期、導覽、標題、主文字區塊或狀態變化區塊，避免畫面過度閃爍。獵弓進度文字為 `拉弓瞄準...`；投石索進度文字為 `投擲投擲...`。
- 戰鬥戰利品不在戰鬥結果頁立刻加入背包。戰鬥結束時先把本次戰利品存成待整理清單；狀態變化區塊以 `戰利品` 列顯示本次獲得的道具與數量，不使用 `物品` 欄位，也不再額外加 `戰利品：` 前綴。沒有戰利品時不顯示這列。
- 遠程武器不直接套用近戰結果，而是消耗 `1` 個對應彈藥嘗試削弱敵方本次戰力。若遠程命中值大於等於敵方遠程閃避，敵方本次戰力 `-1`；若命中值未達敵方遠程閃避，除了絕對閃避敵人外，仍有 `33%` 機率命中並削弱敵方本次戰力。若戰力降到 `0`，或本次遭遇累計遠程命中達 `3` 次，本次遭遇直接以 `大勝利` 結算。
- 遠程攻擊不論命中、失手、被敵人閃避或敵人免疫，都必須立刻在遭遇場景主文字區塊用故事方式回饋：說明主角用什麼武器朝敵人射擊或投擲、命中與否、目前累計命中幾次，以及敵人受到攻擊後的反應。不得把這段寫成 `剛才` 欄位式資訊。若敵人受到削弱，狀態變化區塊顯示敵人受傷、受到削弱；若沒有命中，狀態變化區塊顯示敵人沒有受到削弱。彈藥或體力變化仍照一般狀態變化顯示。
- 遭遇場景發生玩家行動後，主文字區塊應替換成該行動的新資訊，不要繼續重複初次遭遇的環境、外觀與對峙描述。行動回饋可以維持主文字區塊一般文字格式，不擅自加底色、邊框或欄位。
- 若玩家後續選擇逃跑並成功脫離，戰鬥結果頁不得保留逃跑前上一個戰鬥行動的回饋文字或顏色；結果頁只顯示逃脫結果敘事與實際狀態變化。
- 逃跑失敗時，遭遇場景主文字區塊必須刷新為本次逃跑結果。先用紅字顯示 `逃脫失敗。`，再用一般文字敘事說明沒有順利拉開距離、被敵人追擊並受傷。這段不得保留逃跑前上一個遠程或近戰行動回饋。
- 同一場遭遇中若連續逃跑失敗，失敗敘事必須依次數變化，避免只重複同一段受傷描述。第二次之後可描述被逼回原位、腳步打滑、傷勢加重或站穩變得吃力。
- 逃跑成功時，戰鬥結果頁先用綠字顯示 `逃脫成功。`，再接逃脫結果敘事；同樣不得保留逃跑前上一個戰鬥行動回饋。
- 遠程行動的 `命中` / `沒有命中` 是核心情報，必須直接明寫，不靠弦外之音讓玩家推測。`命中。` 以綠字獨立顯示，`沒有命中。` 以紅字獨立顯示；其餘遠程敘事維持一般文字格式。
- 同一場遭遇中，遠程命中的主文字回饋需依累計命中次數明顯變化。第二次命中不可只替換數字，應改寫成敵人節奏、姿態或攻勢被進一步打亂的段落，讓玩家感覺戰況正在推進；但仍必須直接保留 `命中` 與第幾次命中的明確文字。第三次命中會直接結算勝利，文案只需表達最後一擊讓敵人支撐不住，不再追加「第幾次命中」以外的持續壓制情報。
- 戰鬥結果結算時，本次遭遇每支射出的箭矢與每顆投出去的石頭都各自判定是否回收；基礎回收率為 `10%`，遠程技能每級再增加 `3%`。回收成功的彈藥要回到背包，並由行動結果變化框呈現道具數量變化。
- 即使敵人免疫遠程，也允許玩家射出第一發；該次必定浪費彈藥、不削弱敵方戰力，且之後本次遭遇不再能使用遠程牽制。
- 遠程命中或失手後若仍留在遭遇場景，主文字區塊必須立即用最新敵方戰力重算 `戰勝可能`，但仍只看近身作戰是否可擊敗；若該次行動有生命、體力或物品變化，變化內容只放在狀態變化區塊，不得蓋掉主文字區塊中的遭遇敘事與 `戰勝可能`。
- 遠程命中值使用 `遠程技能 + 彈藥克制加成`；若敵人 `effectiveAmmo` 等於該彈藥或為 `兩者`，加成為 `+2`。
- 若遠程嘗試命中，之後仍可繼續使用遠程牽制；若遠程嘗試未命中，本次遭遇不再能使用遠程牽制。
- Debug 模式開啟時，遭遇主文字區塊的情報列可追加 `Debug / 敵方戰力`、`Debug / 我方`、`Debug / 懼怕道具`、`Debug / 懼怕武器`、`Debug / 懼怕彈藥`、`Debug / 工具`、`Debug / 遠程`、`Debug / 近戰` 等系統資訊；Debug 未開啟時不得顯示這些系統欄位。
- Debug 模式開啟時，戰鬥工具、煙霧彈與遠程彈藥不消耗，也不加入彈藥回收池；Debug 關閉時維持正常消耗與回收判定。
- 遭遇場景要列出背包中目前持有的作戰工具與近戰武器，不只列出相剋工具；按鈕文字直接使用道具或武器名稱。
- 若選到無法剋制該敵人的作戰工具，該工具仍會消耗，但該次遭遇視為 `徒手` 結算，不套用工具的大勝效果。
- 若在遭遇場景使用 `煙霧彈`，必須直接消耗 `1` 個並結算為成功逃脫，不走一般工具剋制或大勝利判定。
- 近戰武器只提供判定，不因使用而消耗。
- 遭遇場景中，近戰武器與 `徒手` 不再有選擇前的體力門檻；只有 `逃跑` 需要 `1` 點體力，並且要像其他指令一樣在按鈕右側顯示 `需體力 1`，不得因遭遇按鈕的隱藏耗時設定而把體力需求一起藏掉。逃跑不推進時間，但按下後要播放只覆蓋指令區塊的假讀條，進度文字為 `尋找退路...`。
- 遭遇結束後不得立刻自動跳回原場景；必須先進入一個 `戰鬥結果` 畫面。
- `戰鬥結果` 畫面要先用中標題簡單顯示結果：`勝利`、`險勝`、`失敗`。`大勝利` 與 `勝利` 都顯示為 `勝利`；`大失敗` 與 `失敗` 都顯示為 `失敗`。逃脫可顯示為 `逃脫`。這個中標題不需要額外顯示 `戰鬥結果` 小標籤。
- `戰鬥結果` 畫面要顯示玩家選擇是否對上敵人弱點的短句，以及一段完整結果敘事，並保留行動結果變化框；不要另外加 `你面對的是...` 這類遭遇對象重述。
- 玩家選擇是否對上敵人弱點的短句屬於戰鬥結果正文，不得放進狀態變化區塊。狀態變化區塊只顯示生命、體力、物品、戰利品或其他實際數值變化。近身相剋顯示綠字 `使用{武器}作戰時發揮些許優勢。`，近身選錯顯示紅字 `使用{武器}作戰並不順遂。`；戰鬥工具相剋顯示 `使用{工具}應對{敵人}發揮了極大的功效。`，戰鬥工具選錯顯示 `使用{工具}應對{敵人}絲毫沒有作用。`
- 若遠程攻擊直接擊倒敵人並進入戰鬥結果頁，綠字短句只顯示 `你用{遠程武器}完美地壓制對手。`，不得把前一步遠程命中敘事整段搬到結果頁並染成綠字。
- 五種正式戰鬥結果必須各自使用對應敘事文案；文案要能描述整場狀況、主角承受的後果與戰後餘裕。除了 `大勝利` 可以集中描述壓制與採集餘裕外，其餘正式結果都要稍微描述主角與敵方的戰力差距，不得沿用遠程命中、工具使用等前一步行動訊息作為戰鬥結果描述。
- 探索區戰敗或生命歸零時，不直接在戰鬥函式中完成回村與道具懲罰；必須啟動事件播放回村敘事，並由事件效果負責結束探索、推進到隔天清晨與套用可遺失物品懲罰。
- `戰鬥結果` 畫面只保留一顆確認按鈕。若本次沒有戰利品，按鈕文字為 `確認`，按下後回到原本場景；若本次有戰利品，按鈕文字為 `蒐集戰利品`，按下後進入戰利品整理頁。
- `蒐集戰利品` 頁沿用搜尋整理介面邏輯，分組顯示 `戰利品`、`背包（整理後負重：目前 / 上限）`、`本次整理` 與 `確認`。玩家按下 `確認整理` 後才套用背包變化；沒帶走的戰利品與玩家從背包放下的道具都會進入目前探索位置的 `遺落的道具`。
- 從 `戰鬥結果` 回到場景，或完成戰利品整理回到場景時，不保留本次戰鬥的狀態變化資訊，避免回到場景後仍顯示戰鬥狀態變化區塊。

### 對話場景

對話場景由資料生成，不在角色 scene 裡硬寫舊選項。

資料來源：

- 指令定義：`commands.baseCommandSets.dialogueMenu`
- NPC 規則：`villagers[].socialRules`
- NPC 類型互動矩陣：`docs/data/npc-interactions/npc-interaction-rules.json`
- 委託可見性：`quests.json` + save quest progress
- 交易資料：`villagers[].carriedItems` 或特殊 NPC `tradeRules`
- 對話片段：`dialogues.json`

核心對話選單候選為：

- `chat`
- `gift`
- `trade`
- `quest`
- `leave`

但實際顯示必須依 NPC 規則過濾。

### NPC 互動介面

玩家在 `看見的人物` 分組點擊 NPC 後，進入該 NPC 的下一層互動選單。地點畫面只顯示人物名稱，不直接攤開交談、送禮、交易、委託等細項。

建議版面：

```text
米菈
藥草師

她看起來正在整理藥草與繃帶。

共同互動
  交談    送禮    交易    委託

角色功能
  請米菈協助治療

返回
  返回

回應
  「今天不要太勉強，傷口還沒完全好。」
```

NPC 互動區塊：

- `NPC 標題`：顯示 `villagers[].name`，不要寫成 `與某某互動`。
- `NPC 小標題`：顯示 `role`，特殊 NPC 可顯示 `npcType` 對應文字。
- `NPC 描述 / 印象情報`：進入人物互動頁時，主文字區塊簡短顯示主角當下看見的外觀、神態與此刻氛圍。句子應有主詞，例如「她」或「牠」，並以主角視點書寫；若包含兩句以上，換行仍以主文字區塊的閱讀節奏規則為準，不因句號機械拆段，也不得拆開同一段角色台詞。資料優先使用手寫的 `villagers[].impressionInfos` 或 `villagers[].appearance.impressionInfos`，條件可包含 `timeBlocks`、`locationIds`、`requiredFlags`、`blockedFlags`、`requiredDailyFlags`、`blockedDailyFlags`、`affectionMin`、`affectionMax` 等；沒有命中時才用 `appearance.presenceSummary`、其他 `appearance` 欄位或 `profileDescription` 做保守 fallback。不要直接顯示長背景、內部性格標籤或系統規則摘要，也不要放入狀態變化區塊。
- 產生交談、送禮、委託或其他人物互動回應後，人物互動頁改顯示該回應；原本的外觀短句應暫時移除，避免重複佔用畫面。
- 人物互動頁的好感信賴事件統一在玩家按下 `返回`、離開該角色互動頁時檢查。正式完成 `chat`、`gift`、委託、治療或其他角色互動後，畫面應先留在人物互動頁並顯示該互動自己的回應；不得立刻用信賴事件蓋掉送禮台詞或交談台詞。當玩家按下角色互動頁的 `返回` 時，若好感已達門檻且旗標尚未寫入，先播放對應事件，事件結束後回到原本要返回的地點。這類信賴事件的正文仍屬主文字區塊，應寫成角色自然提起村莊近期變化，不顯示 `好感需求`、旗標或觸發規則。
- `共同互動`：從 `commands.baseCommandSets.dialogueMenu` 生成，並依 NPC 規則過濾；不包含 `leave`，返回獨立成組。
- `角色功能`：從 `npc-interaction-rules[].specialActions` 與 `villagers[].interactionActions` 生成，用於特殊情報、治療、訓練、鑑定、護送、占卜等 NPC 專屬功能。
- 角色功能若有 `requirements`，按鈕右側必須顯示需求摘要，例如 `需月露草 x1`；條件不足時按鈕保持可見但 disabled，提示目前缺少的條件。
- `設施協助`：保留給 NPC 專屬非委託功能，例如治療、鑑定或特殊入口協助；設施升級第一版改走委託，不從設施資料直接生成升級按鈕。
- `返回`：回到進入 NPC 互動前的 `returnSceneId`。
- `回應`：顯示最近一次 `chat`、`gift`、`trade`、`quest` 或特殊互動結果。
- 不可把同一地點所有 NPC 的細項同時展開在地點畫面。

NPC 互動不應假設所有 NPC 都只使用同一排 `交談 / 送禮 / 交易 / 委託`。`npc-interaction-rules.json` 可用 `interactionMode` 指定互動畫面類型；目前至少支援 `menuView`。

- `menuView`：一般 NPC 選單，分成共同互動、角色功能、設施協助與返回。
- 地點上的人物入口一律先走 `openDialogue`，再由 NPC 規則決定實際互動內容；不要在地點入口層針對單一 NPC 硬寫特殊按鈕。
- 未知的 `interactionMode` 必須安全退回 `menuView`，避免新增 NPC 類型時整個畫面中斷。

普通核心村民的互動候選：

- 夜晚若核心女孩位於共同宿舍，仍顯示人物入口並允許互動。深夜若角色位於共同宿舍，視為不可互動，不顯示人物入口；非共同宿舍位置仍依資料判定是否可互動。若未來需要其他不可打擾狀態，必須由資料條件或事件明確指定。
- `交談`：抽取 `dialogues.json` 中符合時間、地點、好感、旗標與委託狀態的台詞。候選台詞先取最高 `priority`；最高優先級內若有多筆，依 `weight` 隨機抽選。若交談會增加好感，每名角色每天最多增加 `1` 點；達上限後仍可交談，但狀態變化不再顯示好感增加。
- `送禮`：進入送禮道具選擇，只列出玩家背包中 `canGift: true` 且數量大於 `0` 的道具；每名角色每天只能送一次禮物，當天已送過時按鈕維持可見但 disabled，提示 `今天已經送過禮了。`
- `交易`：進入該 NPC 的交易介面。
- `委託`：只有目前有可接、可回報或進行中的委託時顯示。
- `返回`：回到原地點場景。

特殊 NPC 規則：

- `talkative_cat`：玩家可見名稱為黑貓。牠不顯示普通交談、送禮、交易、委託；只在探索區每個探索位置以 `25%` 機率出現，出現時可顯示特殊行動與返回。
- 黑貓出現在目前探索位置時，主文字區塊需在場景敘事後增加一行沉浸式現身情報，使用綠色文字顯示。這段只描述主角看見黑貓與牠的姿態，不放進狀態變化區塊，也不寫成系統提示。
- `lizard_merchant`：不顯示普通送禮與普通好感互動。可顯示交易、特殊委託、返回。
- 特殊 NPC 不使用普通好感存檔；若台詞條件包含 `affectionMin`，依 `npc-interactions/npc-interaction-rules.json` 的 `dialogueConditionOverrides.ignoreAffectionMin` 忽略該條件。

NPC 額外互動：

- `villagers[].interactionActions` 可提供該 NPC 專屬互動，例如米菈的 `mira_treatment`。
- `npc-interaction-rules[].specialActions` 可提供同一 NPC 類型共用的特殊功能，例如黑貓的無言協助行動。
- 黑貓第一次在探索區被抽到出現時，必須立即觸發初見事件，不需要玩家先點人物互動。事件只播放一次，說明牠無敵意、友善，並讓主角可以單方面把牠的動作理解為想幫忙。
- 黑貓特殊行動包含：引開敵人注意，使目前探索危險度降低 `30`，耗時 `1` 分鐘；尋找臨時藏身處，體力恢復 `2` 且不額外改變危險度，耗時 `30` 分鐘；把東西帶回村子，開啟失物箱存入模式的整理介面，入口不耗時且不播放假讀條，確認整理後才播放只覆蓋指令區塊的假讀條、耗時 `1` 分鐘並把道具送入失物箱，讀條文字為 `黑貓正叼起東西...`；返回則只取消草稿並回到黑貓三個特殊行動選項，不觸發黑貓離開。所有黑貓特殊行動都需要留下情境回饋。任一黑貓特殊行動完成後，黑貓都會從目前探索位置消失；同一探索位置不重新抽出黑貓，直到玩家移動到新的探索位置才重新判定出現率。黑貓特殊行動完成後必須回到原本探索場景，並在主文字區塊以紅色文字追加沉浸式離開描述；帶東西回村只有在確認整理後才算完成並顯示黑貓離開。
- 第一次確認讓黑貓把東西帶回村子後，主程式寫入 `black_cat_first_delivery_pending`。之後玩家第一次進入 `village_square_hub` 時，必須觸發 `black_cat_first_delivery_notice` 事件；此事件不檢查艾妲當下是否在廣場排班，因為它是公共事件演出。事件完成後寫入 `black_cat_first_delivery_notice_seen`，避免重複播放。
- 這類互動只在玩家點進該 NPC 的互動介面後顯示，不在地點畫面的 `行動` 顯示。
- 若互動有 `availability.requiresPresentVillager`，必須確認該 NPC 目前確實在場。
- `recover` 類互動效果除固定數值外，也可用最大值百分比表達；目前米菈治療應恢復主角最大生命的 `70%`，再套用到當前生命並受最大生命上限限制。
- 若 NPC 專屬互動執行後不會產生任何變化，按鈕應維持可見但 disabled；像米菈治療在主角生命已滿時，提示應為 `目前生命已滿，不需要治療。`

伊蓮知識互動：

- 伊蓮的 `interactionActions` 新增 `knowledge`，玩家可從伊蓮互動頁進入教學與設定資訊。
- 知識分類的玩家可見選項必須使用口語問法，例如 `想尋找物品`、`詢問該注意的事`、`關於山谷的故事`，不得直接顯示 `道具獲取途徑`、`遊戲知識`、`背景故事` 這類破壞沉浸感的系統分類名。
- 知識文字不得依賴固定警句模板收尾，例如頻繁使用 `記住`、`不要把 X 當成 Y`、`真正的 X 不是 Y`。每則內容應先有具體情境或角色動作，再給出自然提示；賽拉偏實務觀察，伊蓮偏生活經驗與故事整理。
- 道具來源只列出材料類、資源類、可食用道具，以及可消耗戰鬥工具；不列出所有物品，也不列出純彈藥、裝備、線索、貨幣或委託關鍵物。
- 道具排序優先顯示目前已接委託需求物，其次顯示玩家曾經取得過的道具，再顯示其他符合分類的道具。程式以 `player.obtainedItemIds` 記錄曾經取得過的道具，舊存檔匯入時從目前背包補齊。
- 點進單一道具後，文字用伊蓮口吻說明取得方式，必須是自然對白，不得把來源顯示成冷冰冰的條列清單。內容由 `items.source`、設施產物、一次性箱子、村民交易、委託獎勵、事件獎勵、製作配方與探索搜尋等企劃資料推導，不另外維護一份手寫來源表。
- 一般道具說明視窗也不得把玩家可見文字寫成 `效果`、`來源`、`用途` 等資料卡。必要的效果與重量可以留在操作 UI 或狀態區，但道具本身的說明應用主角能感受到的材質、重量、氣味、磨損、包紮方式、可食用狀態、手感或使用聯想來寫。
- 伊蓮知識文案不得暴露開發或資料修正痕跡，例如「不存在的土堆」、「以前」、「被拿掉」或類似上帝視角。若資料來源曾被修改，玩家只應看到目前世界內合理存在的線索。
- 委託需求不是物品取得管道。伊蓮查詢道具來源時不得把「委託」當作來源提示；只有委託獎勵實際給出該物品時，才可用非系統化口吻描述為收到的謝禮或回禮。
- 遊戲知識第一版包含治療方式與夜間商人提示。夜間商人文字應保持神秘，不直接把完整觸發條件、時間規則或內部旗標攤開。
- 山谷舊事第一版先放三個可改寫的話題草案，集中在山谷村舊事、山谷之神與禁忌之書。
- 知識文本可使用 `§` 作為分頁符號；程式顯示時依分頁拆成多頁，使用 `繼續聽` 推進，不把長段文字一次塞滿畫面。
- 賽拉的 `interactionActions` 也可使用 `knowledge`，但內容走 `docs/data/knowledge/sela-knowledge.json`。賽拉知識分類第一版為 `關於敵人` 與 `森林的注意事項`。
- `關於敵人` 只列出 `player.encounteredEnemyIds` 記錄中已遭遇過的敵人。每個敵人使用兩頁說明：第一頁描述外觀與行為，第二頁以賽拉口吻含糊提示弱點。提示內容內部需檢查 `戰術物品`、`遠程應對`、`近身搏鬥`，必要時包含 `撤退`，但玩家可見文字不得顯示這些流程標籤，也不得使用「我按順序說」這類流程模板。每隻正式敵人應優先使用手寫情報，把弱點藏在該敵人的動作、距離、身體特徵與賽拉的觀察裡；若某階段沒有好解法，也要以自然台詞說出，不可寫成欄位式報告。賽拉台詞不得直接說出應對道具名稱，例如補網、陷阱、火把、箭矢、石頭、劍或槍。
- 若敵人分類為 `巨獸`，賽拉第二頁情報必須額外暗示撤退或拉開距離是合理選項，呼應巨獸逃跑成功率 `100%`；語氣仍維持角色台詞，不直接顯示機率。
- 玩家看過敵人第二頁後，程式需把該敵人 id 記入隱藏欄位 `player.knownEnemyWeaknessIds`。這表示主角已知曉該敵人的弱點，不直接顯示在一般狀態欄。
- 遭遇已知弱點的敵人時，`其他` 分組需顯示 `賽拉的情報`。此按鈕不推進時間、不消耗資源、不改變戰鬥結果，只開啟和道具說明相同形式的 modal。modal 不顯示 kicker，標題只顯示敵人名稱，不加 `眼前的` 等前綴；前導文字固定簡化為 `你記得賽拉曾提及{敵人名}：`。後續引用賽拉台詞與當時情境時使用斜體呈現，內容顯示賽拉提及該敵人弱點時的含糊台詞，不直接顯示應對道具名稱。
- `森林的注意事項` 第一版至少包含：廢棄小屋是休息點與中繼站、外圍約 `10%` 時搜尋可揭露隱藏洞穴、用對戰鬥工具可必定戰勝對應敵人，以及夜晚森林黑影的警告。這些說明可比一般場景文字更明確，因為它們來自賽拉主動教導；黑影情報仍不得直接說出機率或敵人替換規則。

NPC 互動介面必須保留 `returnSceneId`，返回後回到玩家原本所在場景。

### 台詞池

角色 `交談` 使用台詞池。每次交談時，主程式從 `docs/data/dialogues/dialogues.json` 讀取符合條件的候選台詞，再依優先級與權重抽出一段顯示。系統需用 `dialogueHistory.lastShownAtTotalSeconds` 記錄每段台詞上次被說出的總秒數；同一段台詞在一小時內不得再次被抽中。若當下所有符合條件的台詞都仍在一小時冷卻內，代表該角色此情境已沒有可抽台詞，主程式應釋放這批符合條件台詞的冷卻並重新抽選。對話候選還必須符合角色當前行程；除有明確特殊入口規則的 NPC 外，不得在角色不在場的時段或地點生成台詞。交談造成的好感增加受每日上限限制，玩家當天已從同一角色交談取得好感後，後續交談只顯示台詞與其他非好感效果。

台詞池的玩家可見內容不得把角色說話寫成欄位清單、任務報告或資料表解說，例如 `目前狀態`、`需求`、`可取得`、`注意事項`、`好感條件`。角色可以提供資訊，但必須先落在她此刻看見的東西、手上的動作、所在位置、過去經驗或對主角的反應，再用她自己的語氣自然說出；即使是教學、知識、委託提示或道具來源，也不能讓角色像在朗讀系統欄位。

台詞資料條件可包含：

- `villagerId`：角色 id。
- `command`：第一版交談使用 `chat`。
- `conditions.timeBlocks`：可讀取時段。
- `conditions.locationIds`：可讀取地點或探索 scene id。
- `conditions.affectionMin` / `conditions.affectionMax`：好感範圍。
- `conditions.requiredFlags`：玩家必須已有的旗標。
- `conditions.blockedFlags`：玩家已有任一旗標時不可讀取。

抽選規則：

- 先篩出符合角色、指令與條件的台詞。
- 優先取最高 `priority` 的候選群。
- 在同一優先度內依 `weight` 加權隨機抽一筆。
- 若最高優先度候選群有兩筆以上，抽選時排除上一次剛顯示的台詞，避免連續重複同一句。
- 若沒有符合台詞，使用角色互動的 fallback 文。

### 送禮介面

送禮從 NPC 互動介面進入下一層，不在 NPC 主選單直接消耗道具。

```text
送禮給米菈
藥草棚

選擇禮物
  莓果（2）    花朵（1）    返回
```

送禮規則：

- 每個道具必須有 `canGift` 屬性。
- 只有 `canGift: true` 的道具會出現在送禮清單。
- `canGift: false` 的道具不顯示，也不提示其不可送。
- 貨幣、彈藥、裝備、委託道具、線索、主線材料、會卡住委託或探索流程的補給與工具，預設 `canGift: false`。
- 螢石是貨幣，不可作為禮物，不出現在送禮清單。
- 第一版可送禮道具以 `items[].canGift` 為準，目前包含莓果、蘋果、生菜、花朵、礦石、獸角與骨系戰利品等。
- 送出禮物會消耗 1 個道具，推進 `gift.defaultTimeCostSeconds`，並依角色喜好調整好感；命中角色喜好的物品使好感 `+3`，其他可送禮物預設 `+1`。
- 送禮成功後，人物頁主文字區塊必須顯示角色性格化反應，不顯示 `某某收下了某物。好感 +N。` 這類硬式系統文。反應至少分成一般反應、本人喜歡的反應、以及此禮物更適合其他角色時的提示反應。
- 每名角色每天只能收一次禮物；成功送禮後，當天再次進入同一角色送禮流程時不得消耗道具。

## 設施互動

設施行動由 `facilities.json` 生成。

設施狀態不放在常駐側邊欄。玩家需要操作特定設施時，透過目前地點的設施互動按鈕進入該設施的下一層選單。

設施互動不應假設所有設施都共用同一種「說明 + 可互動按鈕」版型。`facilities.json` 可用 `interactionMode` 指定互動畫面類型；目前至少支援 `menuView`、`storageView`、`craftView`、`timeSelectView`。

- `menuView`：一般設施選單，顯示描述與可互動項目。
- `storageView`：直接進入倉庫或地面道具內容整理畫面，不先停留一層通用設施頁。
- `storageView` 的返回必須直接回到原本地點；不得返回 `facility:` 中介頁，避免又被 `storageView` 導回同一個倉庫畫面。
- `craftView`：直接顯示可製作配方與返回，不混進其他通用設施按鈕。
- `timeSelectView`：直接顯示時間選擇控制，例如睡覺。
- 地點上的設施入口一律先走 `openFacility`，再由 `interactionMode` 決定實際場景；不要在地點入口層針對單一設施硬寫特殊按鈕。
- 未知的 `interactionMode` 必須安全退回 `menuView`，避免資料新增時整個畫面中斷。

在地點場景中，設施只以名稱顯示在 `看見的物件` 分組。玩家點擊該物件後，不在原地點畫面直接攤開細項，而是進入該物件的設施選單。

設施選單版面：

```text
補給箱
村莊廣場

木箱就放在廣場邊，掀開時總能看見有人留下的備用品。裡頭的東西不一定多，但看得出村裡的人確實會把能分出的東西留在這裡。

設施狀態
狀態：可取出

可互動
  取出補給    返回
```

設施選單規則：

- 每個設施選單必須保留 `returnSceneId`，返回時回到原本地點。
- 設施選單標題顯示 `facilities[].name`。
- 小標題可顯示目前地點名稱或設施類型。
- 描述只顯示 `facilities[].description` 或行動後的敘事回饋，不混入已搬到設施狀態區塊的欄位資訊。同一份設施狀態資訊只能出現在一個區塊；搬到設施狀態區塊後，不得再留在主文字區塊重複顯示。
- 有等級設施進入設施選單後，設施狀態區塊顯示目前等級；若資料有最高等級，顯示為 `目前 / 最高`，例如田地顯示 `1 / 5`。升級行動不在設施選單中生成，改由 NPC 委託完成時提升等級。
- 產物暫存設施的設施狀態區塊只顯示目前最低限度狀態，例如 `狀態：可摘取`、`狀態：蘋果已被摘取`、`狀態：已翻找` 或 `狀態：已發現：螢石 x1`；不得顯示每天補幾個、何時刷新、還剩幾次刷新或內部規則。
- `可互動` 只顯示該設施可用行動，例如取出、睡覺、進入特殊入口。
- 不可把同一地點所有設施的細項同時展開在地點畫面。

通用行動：

產物暫存設施：

- `small_storage` 是程式運行概念，表示物件可保存已生成但尚未被玩家取得的道具，不是玩家可見的倉庫類型。
- 玩家可見文字必須依物件本身呈現；自然物件用「摘取」「採集」，箱子類物件才可用「取出」。
- 工坊倉庫的 `廢料堆` 也使用這個模型；玩家可見行動是翻找可用布條，不應顯示成倉庫或補貨物件；每天可發現 `2` 個布條。
- 墓地的 `鴉巢` 也使用這個模型；玩家可見行動是翻找巢中物，不應顯示成倉庫或補貨物件。
- 隱藏洞穴中的 `礦脈` 也使用這個模型；玩家可見行動是敲下礦石，不應顯示成倉庫或補貨物件。
- `withdraw`：把該物件目前可取得的產物移到背包。
- 這類規則也可用在固定補給箱，例如藥草棚的 `醫藥補給箱` 可每日補入 `1` 個月露草，若一直未取，最多累積到 `3` 個。
- 一次性寶箱也使用 `small_storage`，但以 `smallStorage.initialItems` 指定初始內容，並在內容被取完時寫入 `smallStorage.oneTimeClaimFlag`；存檔已有該旗標時不再重新生成內容。
- `舊箱子` 是可重複配置的一次性寶箱模式，不是宿舍專屬 UI；每個實例都應由自己的 `locationIds`、`description`、`initialItems` 與 `oneTimeClaimFlag` 決定內容與敘事。
- 已領取且內容為空的一次性寶箱不再顯示在地點的 `看見的物件` 分組；不要把空箱子留在場景中。
- 若一次性寶箱設定 `smallStorage.withdrawAllAtOnce: true`，畫面只顯示一個 `取出` 指令。執行時依序檢查箱內所有物品，負重可承受的加入背包，負重不可承受且設定 `smallStorage.overflowToDroppedItems: true` 的物品移到當前地點的 `遺落的道具`，並同時視為該寶箱已領取。
- 取出一次性寶箱時若有物品因負重不足移到 `遺落的道具`，行動結果必須告知玩家無法全部拾取。
- 同一設施中若同種道具是堆疊數量，例如莓果叢目前有 `berry x3` 時，或月露草叢目前有 `moondew_herb x3` 時，畫面只顯示一個採集按鈕並標示剩餘數量；每次點擊只取得 `1` 個。
- 若玩家取走最後一個產物，該設施內容清空後應自動返回上一層場景，不停留在空設施畫面。
- 每個取得按鈕都必須檢查玩家背包負重；若取得該物品後會超過 `save.player.maxCarryWeight`，該按鈕 disabled，滑鼠提示 `你已經無法負荷這個重量。`
- 不顯示 `deposit`。
- 如果沒有可取得產物，按鈕可以保留但點擊後顯示該物件對應的短句，例如「蘋果樹目前沒有成熟的蘋果。」
- 設施說明文字不得顯示「小倉庫」「補進內容物」等內部實作語句。
- 建議由 `facilities[].obtainAction` 提供玩家可見的 `label`、`contentsLabel`、`emptyMessage` 與 `successMessage`。
- 若同一設施狀態中的同種道具是堆疊數量，例如 `蘋果 x3`，採集選項只顯示一個按鈕，並標示剩餘數量；每按一次只取得 `1` 個，不展開成多顆重複按鈕。

未知採集設施：

- 若設施有 `randomGatherAction`，進入設施選單時不得顯示候選產物名稱或機率。
- 未搜尋時設施狀態區塊只顯示最低限度狀態，例如 `狀態：可尋找` 或 `狀態：可翻找`；已搜尋後直接顯示 `狀態：已尋找` 或 `狀態：已翻找`，不顯示未知產物、候選表或剩餘次數。
- 點擊 `尋找物品` 後才決定實際結果，可能得到道具，也可能沒有得到任何東西。
- 每次每日重置前只能尋找一次；已尋找過時，尋找按鈕 disabled，提示 `今天已經找過了。`
- 若 `randomGatherAction.maxDailyUses` 大於 `1`，可在同一天內翻找指定次數；可用與用完的判斷仍由指令按鈕狀態承擔，設施狀態區塊不顯示今日剩餘次數。
- 若找到的道具會讓玩家超過負重上限，該道具暫時留在該設施，畫面顯示為 `已發現：道具名稱 xN`，並提供取回按鈕。
- 取回按鈕同樣檢查玩家背包負重；超過時 disabled，滑鼠提示 `你已經無法負荷這個重量。`
- 每日重置時，該設施尚未取回的已發現道具會被刪除，不保留到下一天。
- 同一種未知採集設施可以掛在多個 `locationIds`，例如碎石堆可同時出現在村莊廣場與開墾區；這條只適用於村莊內固定設施。探索中臨時生成的同型資源點不共用村莊固定設施的 save data 狀態。
- 這類多地點的未知採集設施，其每日翻找次數與已發現物品也必須按 `locationId` 分開記錄；例如村莊廣場的碎石堆與開墾區的碎石堆不能共用同一個 `lastGatherDay`。
- 同樣原則也適用於多地點 `small_storage` 設施的成熟產物與每日補貨狀態；只要同一設施 id 掛在多個村莊地點，就不能直接共用同一份 `items` 或 `lastRestockDay`。

無限來源設施：

- `infinite_source` 表示可重複取得固定道具的設施，例如探索中的土堆或工坊倉庫的木材堆。
- 不使用 `smallStorage.items`，不需要每日生成或補貨，不會顯示可採數量。
- 按鈕由 `facilities[].infiniteSource` 生成，至少包含 `itemId`、`count`、`timeCostSeconds`、`label`。
- 每次點擊只取得指定數量，土堆第一版為每次取得 `soil x1`。
- 每次取得都必須檢查玩家背包負重；若取得後會超過 `save.player.maxCarryWeight`，按鈕 disabled，滑鼠提示 `你已經無法負荷這個重量。`
- 這類取得是正式行動，需推進時間、寫入 `lastActionResult`，並顯示物品增加。

容器設施：

- 例如 `storage_box` / 倉庫箱。
- 倉庫箱必須出現在所有 `locationIds` 指定地點的 `看見的物件` 分組；目前只設在工坊倉庫，不出現在開墾區。
- 設施主選單顯示目前容量，例如 `內容物：木材 x2；容量：4 / 20`。
- 點擊倉庫箱後進入單層整理介面，不再先選 `領取` 或 `放入`。
- 倉庫箱入口文案應直接表達用途，例如 `存放 / 取回`，不要使用偏描述性的 `整理倉庫`。
- 單層管理介面分組顯示 `倉庫內容`、`背包（整理後負重：目前 / 上限）`、`本次整理`、`確認`。
- `倉庫內容` 與 `背包可放入` 使用接近交易介面的列式排列：每個道具一列，左側顯示道具名稱與目前數量/總重量，右側固定顯示單一操作按鈕。
- `倉庫內容` 只列出整理草稿中倉庫內 `count > 0` 的道具，每列提供 `取回到背包`。
- `背包（整理後負重：目前 / 上限）` 只列出整理草稿中玩家背包內 `count > 0` 且可放入的一般道具，每列提供 `放入倉庫`。
- 第一版每次點擊一個道具只把 `1` 個堆疊數量移入本次整理草稿，並停留在同一個整理介面方便連續操作。
- 若某列在草稿中數量變為 `0`，該列不得殘留。
- `本次整理` 以文字摘要顯示取回與放入的草稿內容。
- `確認整理` 才真正套用倉庫與背包變化，只產生一次行動結果，並回到原場景。
- `返回` 視為取消本次整理，必須丟棄尚未確認的草稿變更，不產生行動結果。
- 若倉庫箱空，`倉庫內容` 分組顯示不可用提示。
- 若背包沒有可放入道具，`背包可放入` 分組顯示不可用提示。
- 放入前必須檢查容量；若放入後會超過 `capacityWeight`，按鈕 disabled，滑鼠提示容量不足。
- 容量計算使用 `items[].weight * count`。找不到道具重量時，第一版視為 `0`，但仍可顯示原始 item id。
- 裝備、委託道具、線索與主線關鍵材料是否可放入未來可再細分；第一版先允許玩家放入背包中有定義的非貨幣道具，螢石不顯示在 `背包可放入` 分組。
- 調整整理草稿不算真正行動，不寫入 `lastActionResult`；只有確認整理才算真正行動，需顯示物品增減。
- 從容器領取道具也必須檢查玩家背包負重；超過時按鈕 disabled，滑鼠提示 `你已經無法負荷這個重量。`
- `遺落的道具` 也屬於容器型設施，操作介面應和倉庫一樣採用列式排列，但採用草稿式編輯。
- `遺落的道具` 介面只分成 `地上的道具` 與 `背包（整理後負重：目前 / 上限）` 兩個分組，每列只有一顆操作按鈕，不使用 `-` / `+` 加減控制。
- `地上的道具` 分組的按鈕文字必須是 `取回到背包`，表示把地上的 1 個堆疊數量移進背包草稿。
- `背包` 分組的按鈕文字必須是 `放到遺落道具`，表示把背包中的 1 個堆疊數量移到地面草稿。
- 每次點擊只把 `1` 個堆疊數量移入本次整理草稿；若某列在草稿中數量變為 `0`，該列不得殘留。
- 在 `遺落的道具` 介面中，取回到背包與放到遺落道具按鈕只調整本次整理草稿，不立即修改背包、不立即修改地面容器，也不更新狀態變化區塊。
- `遺落的道具` 介面必須提供 `確認整理`；玩家按下後才一次套用所有取回與放下結果，只產生一次行動結果，並回到原場景。
- `遺落的道具` 介面的 `返回` 視為取消本次整理，必須丟棄所有尚未確認的草稿變更，直接回到原場景，不產生行動結果。
- `遺落的道具` 介面必須顯示警告，提醒放在這裡的東西只是暫時留在原地，時間一久或離開探索位置後可能永久消失。
- 這個地面容器不是永久存在；從最後一次丟棄開始計時，滿 `1` 小時後容器與內部物品一起消失。
- `失物箱` 也屬於容器型設施，固定放在宿舍，只接收過期 `遺落的道具` 中的裝備與委託道具；它不提供玩家手動放入，只提供取回。
- 若玩家把其中物品全部取走，容器立即消失。
- 這個地面容器不需要額外入口層；從 `看見的物件` 點下去後就直接顯示內容與取回按鈕。

恢復設施：

- 例如 `protagonist_bed`。
- 顯示恢復行動與耗時。
- 執行後依 `recovery.effects` 處理體力；睡眠類設施不得恢復生命，生命恢復必須走治療相關動作或道具。
- `protagonist_bed` 的玩家可見名稱為 `你的床`；進入後顯示 `睡覺`，可選擇快速度過 `1` 到 `6` 小時，並以 `1` 小時為最小刻度；也可直接睡到下一個指定時段。睡覺可大幅跳到較遠時段。若指定時段落在隔天，按鈕與結果訊息需顯示 `隔天` 字樣，例如 `睡到隔天清晨`。體力恢復效率與 `wait` 相同。
- `你的床` 不需要再停留一層設施說明頁；玩家在宿舍看到 `你的床` 時，點下去後就應直接看到睡覺時間選單。

製作設施：

- 例如 `alchemy_table`。
- 進入設施選單後，顯示 `可製作` 分組與 `返回`。
- `可製作` 只列出目前玩家已學會，且 `workstationFacilityId` 等於該設施 id 的配方。
- 每個配方按鈕至少顯示：成品名稱、成品數量、所需材料摘要、耗時；成品數量要直接接在製作名稱後，例如 `製作箭矢 x3`。
- 材料不足時，配方按鈕保持可見但 disabled，滑鼠提示缺少的材料，例如 `缺少：月露草 x1`。
- 未學會的配方第一版不顯示；若之後要顯示未學會配方，需另外加上明確規則來源，不先偷看完整配方表。
- 製作成功後，必須顯示材料扣除、成品增加與耗時，並寫入 `lastActionResult`。
- 製作前的負重檢查必須用「扣除材料並加入成品後」的背包結果判定；如果淨結果不超重，即使目前背包無法直接再塞入成品，也應允許製作。
- 製作失敗只允許出現在條件不足時，例如材料不足、工作台未解鎖或配方未學會；這些情況不得扣材料或推進時間。

特殊入口：

- 例如 `lizard_well`。
- 特殊入口可使用 `access.timeBlocks` 限制顯示時間；例如蜥蜴羅索的井底入口只在清晨、上午、中午、下午顯示，夜晚與深夜不顯示，避免同一 NPC 同時出現在井底與地面。
- 特殊入口的行動由 `facilities[].access` 生成，可包含 `label`、`timeCostSeconds`、`requiredFlag` 或 `requiredFlags`。
- `requiredFlag` / `requiredFlags` 是顯示條件，不是點擊後才檢查的失敗條件；條件未滿時，入口按鈕不顯示。
- 未解鎖狀態不得顯示「需要先完成相關委託」這類系統解釋；玩家只會看見一般設施描述與其他已可用行動。
- 已解鎖時可進入特殊 NPC 互動。

可見性條件：

- 所有行動可用 `visibility` 或 `visibleWhen` 宣告是否顯示，目前至少支援 `requiredFlag`、`requiredFlags`、`blockedFlag`、`blockedFlags`。
- 這類條件適合用於「玩家尚不知道或尚未解鎖的入口、話題、角色功能」；不要用 disabled 按鈕或失敗訊息提前暴露內容。
- disabled 只用於玩家已知道該行動存在，但暫時因資源、體力、容量或狀態不足而不能做的情況。

## 指令互動元件

### 道具

選擇 `useItem` 時，應進入道具選擇畫面。

顯示規則：

- 上一層指令文字顯示為 `道具`。
- 可使用清單只列出背包中 `count > 0` 且在目前場景有直接用途的消耗品。
- 沒有一般使用效果的戰鬥工具或彈藥，例如陷阱、火把、箭矢、石頭、補網，不顯示在一般道具使用清單；它們只在戰鬥、製作或其他專用流程中出現。
- 煙霧彈是探索區特殊例外：只在探索區道具清單顯示，用於降低當前危機感知；若危機感知為 `0`，按鈕保留但 disabled。村內安全地區不可使用煙霧彈。
- 每個項目顯示名稱、數量、效果摘要與耗時。
- 使用後扣除 1 個堆疊，套用 `items[].effects`。
- 使用道具成功後停留在道具選單，不自動返回上一層場景；若最後一個可用消耗品被用完，仍停留在道具選單並顯示目前沒有可直接使用的消耗品。
- 若道具資料含有 `useReturnsItems`，使用完成後必須把對應回收物加入背包，例如小治療藥使用後加入空藥水瓶。
- 道具選單必須顯示 `製作道具` 入口，最後顯示 `丟棄道具` 入口。

### 等待

`wait` 不應立即推進時間。正式介面應顯示時間選擇控制。

控制規格來自 `commands.commands.wait.timeInput`：

等待介面只保留時間滑桿、確認等待與返回，不提供跳到指定時段按鈕。跳時段功能只保留在床的睡覺介面。

- 最小值：`minSeconds`
- 最大值：`maxSeconds`
- 步進：`stepSeconds`
- 需要確認：`requiresConfirmation`

效果規則：

- 等待每經過 `1` 分鐘，隱藏累積 `1` 點休息值。
- 休息值滿 `30` 點時，恢復 `1` 點體力，並消耗對應的 `30` 點休息值。
- 休息值屬於 save data，可被記錄與儲存，但第一版不在介面中直接顯示。
- 等待不恢復生命。
- 等待確認完成後，介面必須自動返回原本開啟等待的上一層場景，不停留在等待選單。
- 等待完成後，主文字區塊必須顯示一段依目前地點或探索環境生成的情境回饋；狀態變化區塊只負責列出時間推進、體力恢復等具體變化。

第一版若尚未做滑桿，可使用固定快捷選項，但程式碼應保留可替換成滑桿的結構。

### 交易

交易介面從 NPC 互動介面進入，不作為常駐側邊欄。交易介面分為買入與賣出；普通核心村民第一版只支援玩家向 NPC 買入，蜥蜴羅索支援買入與賣出。

普通 NPC 交易版面：

```text
與托莉交易
持有螢石：10

商品
  莓果        庫存 4/6    1 螢石          買入 0   -   +

返回
```

蜥蜴羅索交易版面：

```text
與蜥蜴羅索交易
持有螢石：10

購買
  陷阱        庫存 1/3    7 螢石          買入 0   -   +
  火把        庫存 2/5    4 螢石          買入 0   -   +
  補網        庫存 1/2    8 螢石          買入 0   -   +

賣出
  碎骨        持有 3      1 螢石          賣出 0   -   +
  獸牙        持有 1      5 螢石          賣出 0   -   +

返回
```

交易介面應顯示：

- 商品名稱
- 剩餘庫存或持有數量
- 價格
- 貨幣道具
- 目前已選買入或賣出數量
- 每個道具各自一列，不得把同一道具拆成多顆加減按鈕分散顯示
- 每列右側固定提供 `-` 與 `+` 按鈕；玩家用同一列內的按鈕調整數量
- 若該次 `+` 或 `-` 目前無法使用，該按鈕直接 disabled，不得額外生成另一顆替代按鈕
- 購買或賣出後的結果訊息
- 交易列應盡量緊湊，優先把名稱、庫存與價格壓在同一列資訊流中，避免因單列過高導致頻繁捲動。
- 在 `完成交易` 之前，畫面需先用純文字摘要本次草稿，不用按鈕列出摘要項目。
- 摘要格式應接近：`買入：陷阱 x 2 、 火把 x 2。`、`賣出：碎骨 x 3。`
- 若某一側目前沒有任何項目，該行不顯示，不要寫 `無`。
- 若買入與賣出都沒有任何項目，整個 `本次交易` 區塊直接隱藏。
- 摘要最後一行需直接寫出本次淨花費或淨賺取的螢石，以及交易後玩家身上的螢石數量。
- 若買入與賣出剛好相抵，摘要最後一行可寫成 `你不會花費或賺取螢石，你身上的螢石數量將維持 X 顆。`
- 若結算後螢石會小於 `0`，摘要最後一行不要顯示負數，直接改成 `這筆交易無法被接受。`
- 交易摘要文案應盡量精簡，使用 `你身上的螢石將為 X 顆。`、`你身上的螢石將維持 X 顆。` 這類短句即可。
- 這段摘要是純文字，不做成按鈕，也不做成像另一個可互動卡片的外觀。
- `完成交易` 按鈕本身不再重複附帶 `結算後螢石` 這類摘要字樣。

交易資料來源：

- 核心村民：`villagers[].carriedItems`
- 蜥蜴羅索：`villagers[].tradeRules`
- 價格：優先讀 `carriedItems[].price`，售出規則讀 `items[].price`
- 庫存 runtime 狀態：`save.tradeStocks`

普通 NPC 買入規則：

- 商品清單來自 `villagers[].carriedItems`。
- 未達 `affectionMin` 的商品不顯示。介面不得顯示「好感不足」或未解鎖商品名稱，避免玩家提前知道 NPC 未來會賣什麼。
- 若全部商品都未解鎖，顯示短句：`目前沒有可交易的商品。`
- 購買列的 `+` 需檢查玩家 `fluorite` 數量、剩餘庫存與增加後是否超過負重上限。
- 購買列的 `-` 在目前未選擇該商品時 disabled。
- 購買成功後，玩家扣除螢石、背包增加商品、該 NPC 的 `save.tradeStocks[npcId][itemId]` 減少。
- 庫存為 `0` 或已達可買上限時，`+` 按鈕 disabled。

蜥蜴羅索買入規則：

- 販售商品來自 `lizard_merchant.carriedItems` 或特殊 `tradeRules` 計算結果。
- 不使用普通核心村民好感門檻。
- 目前販售陷阱、火把、補網，不販售彈藥與裝備。

蜥蜴羅索賣出規則：

- 可賣清單來自玩家背包與未來 `storage_box` 內容。
- 只顯示 `items[].price.sellable: true` 且玩家持有數量大於 `0` 的道具。
- 賣出價格讀 `items[].price.amount`。
- 賣出列的 `+` 在已達目前可賣上限時 disabled。
- 賣出列的 `-` 在目前未選擇該商品時 disabled。
- 調整賣出數量時不立即扣道具。
- 按下 `完成交易` 後，玩家才失去賣出的道具並獲得對應數量的 `fluorite`。

### 委託

委託介面只顯示目前可接、可回報或進行中的委託。

不顯示空的 `委託` 按鈕，除非該 NPC 目前有可見委託。

- 從 NPC 互動畫面的 `委託` 進入委託清單頁，不直接在對話主頁完成接取或提交。
- 委託清單頁至少分成 `可接委託`、`已接委託`、`可提交委託`、`返回`。
- `已接委託` 區塊只有在玩家目前真的接有該 NPC 的委託時才顯示。
- `已接委託` 每條要直接顯示委託完成條件與目前進度，例如提交或收集數量的 `目前/需求`。
- `已接委託` 只顯示條件與進度，不顯示獎勵內容。
- `已接委託` 是靜態資訊列，不是按鈕；不得用 disabled 按鈕呈現，避免玩家誤以為它是暫時被鎖住、未來可點擊的行動。
- 點擊 `可接委託` 後，進入獨立的委託對白頁，顯示委託內容與 `答應` / `拒絕`。
- `答應` 與 `拒絕` 都必須各自產生一段回應，之後直接返回該人物互動頁。
- 點擊 `可提交委託` 後，進入獨立的提交對白頁；玩家按下 `確認` 後才真正完成委託、扣除 `submitItems` 目標道具、發放獎勵，然後返回該人物互動頁。
- 委託獎勵發放時必須逐項檢查玩家負重。可承受的物品加入背包；超出負重的獎勵物品放到委託回報所在地的 `遺落的道具`，委託仍視為完成，行動結果需告知哪些獎勵暫時被留在地上。
- 委託清單頁不顯示獎勵預覽。
- 委託獎勵可直接修改玩家欄位，例如 `carrySkill`，這類變化也應進入狀態變化區塊。
- 委託獎勵可修改設施等級，例如 `rewards.facilities.storage_box.levelDelta: 1`；這類變化也應進入狀態變化區塊。

## 每日重置顯示

每日重置時間默認為 `06:00`。

介面需要能顯示或追蹤：

- 今日是否已重置。
- 產物暫存型設施是否已生成今日產物。
- 交易庫存是否已刷新。
- 每日旗標是否已清空。
- 看板留言是否已更新。

這些欄位來自 `save.dailyReset`、`save.time.lastDailyResetDay`、`save.facilities`、`save.tradeStocks`。

## 錯誤與缺資料狀態

主程式遇到資料缺失時，不應整頁崩潰。

建議 fallback：

- 找不到 item：顯示 `itemId`。
- 找不到 location：回到 `village_square_hub`。
- 找不到 villager：不生成該互動。
- 找不到 facility state：依 `facilities.json` 建立預設 runtime 狀態。
- 找不到 command：不生成該按鈕，並在 console 留下警告。

畫面顯示錯誤時使用短句，不顯示技術堆疊給玩家。

## 主程式實作檢查表

主程式介面完成一個功能時，應逐項確認：

- 地點互動使用 `location.id`，不是中文地點名。
- 設施按鈕由 `facilities[].locationIds` 生成。
- 對話選單由 `commands.baseCommandSets.dialogueMenu` 生成。
- 特殊 NPC 依 `npcType` 與 `socialRules` 排除普通社交。
- 黑貓特殊行動必須在主文字區塊留下情境回饋；其中引開敵人注意的回饋需明確讓玩家知道「你感知到的危機降低了一些」。引開敵人注意消耗 `1` 分鐘；請黑貓帶回東西在進入整理介面時不消耗時間，只有確認整理才消耗 `1` 分鐘；尋找臨時藏身處消耗 `30` 分鐘。
- 產物暫存型設施只可取得不可放入，且玩家可見文字不得把自然物件描述成倉庫。
- 補給箱總數達 6 時不再補給；每日重置時若未滿 6，從補給池隨機補入 2 樣，但不得補到超過 6。
- 背包使用 `save.player.inventory` 陣列格式。
- runtime 狀態只寫入本機存檔，不寫回 docs data。
- 匯出與匯入使用完整 save JSON。
- 窄畫面下指令區塊仍在狀態詳情之前。
