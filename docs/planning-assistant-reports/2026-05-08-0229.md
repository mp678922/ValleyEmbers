## 一、企劃運作矛盾
### 1. 禁忌之書主線的前置旗標沒有產生來源
- 位置：`docs/data/quests/quests.json:14`、`docs/data/scenes/scenes.json:79`、`docs/data/scenes/scenes.json:201`、`docs/data/scenes/scenes.json:286`
- 問題：`main_forbidden_books` 的前置條件要求 `asked_elaine_ruin_lore`，但當時場景資料只產生一批舊遺跡線索旗標與 `admitted_ruin_whispers`，沒有任何資料會寫入 `asked_elaine_ruin_lore`。
- 為什麼會卡住：委託系統若照 `prerequisites.flags` 判定，四本禁忌之書主線永遠不可接取。這不是內容量不足，而是主線入口旗標斷線。
- 建議修正方向：把前置旗標改成現有旗標之一，例如 `admitted_ruin_whispers`，或在伊蓮相關對話/事件中補上 `asked_elaine_ruin_lore`。若要更精準，可新增一個正式「向伊蓮詢問遺跡傳說」事件來產生該旗標。

### 2. 主角的床同時是地點行動與設施物件
- 位置：`docs/interface-design.md:67`、`docs/interface-design.md:69`、`docs/data/locations/locations.json:29`、`docs/data/facilities/facilities.json:26`
- 問題：介面規格要求 `localActions` 生成在「可執行指令」，設施則生成在「看見的物件」並進入下一層設施選單；但 `sleep_in_protagonist_bed` 已經放在 `dormitory.localActions`，同時 `protagonist_bed` 又作為設施存在。
- 為什麼會卡住：共同宿舍會同時出現「睡主角的床」行動與「主角的床」物件，兩者都指向同一個六小時全恢復規則，玩家與實作都會分不清哪個才是正式互動入口。
- 建議修正方向：保留其中一條路徑。若採現行介面規格，建議移除 `dormitory.localActions.sleep_in_protagonist_bed`，讓玩家先點「主角的床」設施，再看到睡眠行動。

## 二、程式運行模糊處
### 1. 特殊 NPC 不存好感，但台詞條件仍使用 `affectionMin`
- 位置：`docs/data/README.md:25`、`docs/data/save/save-template.json:90`、`docs/data/dialogues/dialogues.json:624`、`docs/data/dialogues/dialogues.json:678`、`docs/data/dialogues/dialogues.json:927`
- 模糊點：資料規則說 `talkative_cat` 這類特殊 NPC 排除普通好感、送禮、交易與委託流程；存檔也只保存六名核心村民的好感。但特殊 NPC 的台詞仍帶有 `affectionMin: 0`。
- 程式需要明確知道：遇到沒有普通好感存檔的 NPC 時，`affectionMin: 0` 是自動視為通過，還是因為找不到 `save.villagers[npcId].affection` 而失敗。
- 建議補規則：二選一即可。第一種是移除特殊 NPC 台詞中的 `affectionMin`；第二種是在介面/對話規格中明定「沒有核心好感的 NPC，若條件只要求 `affectionMin: 0`，視為通過」。

### 2. 請米菈治療是地點行動，但缺少米菈是否在場的判定
- 位置：`docs/game-design.md:916`、`docs/data/locations/locations.json:52`、`docs/data/villagers/villagers.json:313`
- 模糊點：企劃說「請米菈治療」需要米菈好感或消耗藥草；現行 `ask_for_treatment` 只檢查 `moondew_herb`，而且它是藥草棚的 `localActions`，不是 NPC 互動。米菈的行程下午在 `reclamation_area`，但藥草棚仍會固定生成這個治療行動。
- 程式需要明確知道：米菈不在藥草棚時，是否仍能治療；治療是藥草棚設施行動、米菈 NPC 行動，還是兩者皆可但效果不同。
- 建議補規則：若治療需要米菈本人，將行動移到 NPC 互動或補上 `requiresPresentVillager: "mira"`。若藥草棚可自助治療，則改名為「使用藥草棚藥材治療」，並移除對米菈在場的敘述。

### 3. `observe` 仍留在 Markdown 規則，但指令資料已不提供
- 位置：`docs/game-design.md:486`、`docs/game-design.md:806`、`docs/data/README.md:20`、`docs/data/commands/commands.json:3`
- 模糊點：Markdown 仍把 `observe` 列為地點共用指令，但 `commands.json` 的 `standardLocation` 只有 `useItem` 和 `wait`，且沒有 `observe` 指令定義。
- 程式需要明確知道：`observe` 是已移除的舊規則，還是未補回的正式指令。
- 建議補規則：若維持精簡指令面，從 Markdown 和 README 移除 `observe`。若仍要觀察功能，就在 `commands.json` 補回 `observe` 並加入 `standardLocation`。

## 優先處理
1. 先修 `asked_elaine_ruin_lore` 的來源或前置條件，否則主線委託入口不可達。
2. 決定「主角的床」只走設施入口或地點行動入口，避免同功能重複生成。
3. 明確特殊 NPC 的對話條件是否允許沒有好感存檔，避免牠們的台詞被條件系統濾掉。

## 三、五點規劃建議
1. 把每個主線旗標做成一張小型「旗標來源表」：旗標名稱、由哪個場景/對話/委託產生、解鎖什麼內容。
2. 將地點行動與設施行動分工固定下來：能被看見的物件優先放 `facilities.json`，只有不屬於物件的地點行為才放 `localActions`。
3. 特殊 NPC 建議建立一個最小規格表，列出可用互動：交談、特殊情報、交易、委託、送禮、好感，各自允許或禁止。
4. 清理 Markdown 中已被資料層取代的舊指令名，例如 `observe`，避免未來實作時回頭支援已刪除的命令。
5. 下一次企劃可集中補「主線入口到第一本禁忌之書」的可玩流程，先不展開四本書全部內容，只確保第一段能從事件、旗標、委託到獎勵完整跑通。
