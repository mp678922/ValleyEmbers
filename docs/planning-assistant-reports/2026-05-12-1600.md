## 一、企劃運作矛盾
### 1. 存檔策略的權威文件互相打架
- 位置：
  `docs/game-design.md:16-19,500,1165,1172`
  `docs/data/README.md:21`
  `docs/interface-design.md:344`
  `docs/data/save/save-template.json:8-24`
- 問題：
  `game-design.md` 和 `docs/data/README.md` 仍把第一版存檔描述成 `Cookie`，但 `interface-design.md` 與 `save-template.json` 已經明確改成 `localStorage + 手動存檔`。
- 為什麼會卡住：
  這不是單純文案落後，因為 `game-design.md` 還把 `storage` 欄位的語義寫成 Cookie 自動保存，會讓後續做企劃、補 save schema、設計新 UI 或檢查匯入匯出的人讀到不同版本的真規則。
- 建議修正方向：
  以 `docs/data/save/save-template.json` 與 `docs/interface-design.md` 為準，把 `game-design.md`、`docs/data/README.md` 中所有第一版存檔描述統一改成 `localStorage`、`手動存檔`、`overwriteCurrentLocalSave`。

### 2. 禁忌之書主線目前入口不可達，旗標登錄也指向不存在的來源
- 位置：
  `docs/data/quests/quests.json:3-14,49`
  `docs/data/flags/flag-sources.json:96-102`
  `docs/planning-assistant-reports/2026-05-08-0229.md:24-26`
- 問題：
  主線委託 `main_forbidden_books` 需要前置旗標 `asked_elaine_ruin_lore`，但目前 `docs/` 內沒有任何資料會寫入這個旗標；同時 `forbidden_books_truth_revealed` 的 `sourceId` 還寫成不存在的 `forbidden_books_first_truth.rewards`。
- 為什麼會卡住：
  玩家根本無法合法進入禁忌之書主線，旗標表也無法正確反查這條主線是由哪個委託或事件產生。這會直接讓主線入口、除錯、後續事件串接全部失去錨點。
- 建議修正方向：
  二選一即可：
  1. 在伊蓮的對話或事件中正式新增 `asked_elaine_ruin_lore` 的來源。
  2. 把主線前置條件改成目前已存在且語義相符的旗標。
  另外同步把 `forbidden_books_truth_revealed` 的 `sourceId` 改成目前真實存在的主線委託或事件 id。

### 3. 米菈委託的可回報條件與委託目標數量不一致
- 位置：
  `docs/data/quests/quests.json:141-158`
  `docs/data/dialogues/dialogues.json:74-90`
- 問題：
  `濕地的月露草` 委託本體要求提交 `月露草 x2`，但 `mira_moondew_ready_to_report` 的對話條件仍要求 `requiredItems.moondew_herb = 3`。
- 為什麼會卡住：
  玩家會先滿足委託模板，再被對話/回報層擋住，形成「委託說可以交，但互動畫面不讓交」的流程斷裂。
- 建議修正方向：
  以 `quests.json` 的委託模板為主，把 `dialogues.json` 的可回報條件同步改成 `2`；若想讓回報邏輯完全不重複數值，後續可改成由委託目標直接生成回報門檻。

## 二、程式運行模糊處
### 1. 禁忌之書的四次揭露有順序規則，但沒有明確的完成點歸屬
- 位置：
  `docs/game-design.md:104`
  `docs/data/quests/quests.json:17-46`
  `docs/data/save/save-template.json:203-207`
- 模糊點：
  目前只定了 `completeNextForbiddenBookTask` 與 `nextRevealIndex` 的概念，但還沒有明確說明「哪些事件、委託、場景互動」可以算作一次合法的禁忌之書推進完成點。
- 程式需要明確知道：
  哪些資料層會發出「完成一本禁忌之書相關委託」的訊號，以及這個訊號是由事件頁、委託提交、特殊探索結算，還是主線專用旗標來驅動。
- 建議補規則：
  定一個固定歸屬層。最乾淨的做法是讓特定主線事件頁或主線委託完成效果寫入 `forbiddenBooks.completedRelatedTaskCount +1`，並由同一層決定揭露哪一本。

### 2. 探索敗退後「誰在照顧主角」仍是企劃規則，但目前沒有狀態歸屬
- 位置：
  `docs/game-design.md:363,1236`
  `docs/data/events/events.json:322-339`
  `docs/data/save/save-template.json`
- 模糊點：
  企劃明確寫了生命歸零後醒來時會有一名隨機核心女孩照顧主角，但目前敗退事件只處理回村、跨日與掉裝，沒有定義照顧者是純敘事抽選，還是需要寫入 save / event state。
- 程式需要明確知道：
  這個照顧者是否只影響當下文字，還是會影響後續對話、好感、事件冷卻或當天 NPC 反應。
- 建議補規則：
  如果只是敘事 flavour，就明寫「僅在敗退事件當頁隨機決定，不寫入 save」；如果要影響後續互動，就新增一個明確欄位，例如 `events.lastExplorationDefeatCaregiverId` 或等效事件結果狀態。

### 3. 部分權威 JSON 帶 BOM，資料可攜性仍有風險
- 位置：
  `docs/data/facilities/facilities.json`
  `docs/data/flags/flag-sources.json`
  `docs/data/npc-interactions/npc-interaction-rules.json`
  `docs/data/quests/quests.json`
  `docs/data/villagers/villagers.json`
- 模糊點：
  這幾份檔案目前帶 UTF-8 BOM，直接用標準 `JSON.parse` 讀 raw string 會失敗；這代表「企劃 JSON 是可直接當作權威資料餵給任何解析器」這件事還不是完全成立。
- 程式需要明確知道：
  專案是否正式接受「載入層必須先去 BOM」作為前提，還是資料檔本身必須保持純 JSON。
- 建議補規則：
  二選一固定下來：
  1. 規定 `docs/data/*.json` 一律存成無 BOM UTF-8。
  2. 規定所有資料載入器都必須先做 BOM strip。
  如果不先定，之後每新增一個工具鏈都可能再踩一次。

## 優先處理
1. 先修禁忌之書主線入口與旗標來源，因為它現在是不可達狀態。
2. 立刻統一存檔策略文檔，避免之後 save 規格與 UI 企劃繼續分叉。
3. 修正米菈委託的回報數量，這是最容易直接卡玩家流程的資料衝突。

## 三、五點規劃建議
1. 之後凡是 `quest_ready`、`submitItems`、`rewardItems` 這類會重複帶數值的資料，優先只保留一個權威來源，其他畫面從委託模板推導，不要再手填第二份數量。
2. 主線相關旗標不要只記「已揭露真相」，也要記「由哪個事件或委託頁寫入」，否則後續加分支時很難追資料鏈。
3. 可以替 `events`、`quests`、`flags` 三層補一份小型對照規則，定義誰負責「入口」、誰負責「進行中」、誰負責「完成後鎖定」。
4. 敗退事件如果未來要做不同照顧者反應，最好把它當成可擴充模板，先定好「是否寫 save」這件事，再補人選差異文案。
5. 若你打算繼續把 `docs/data` 當真正可執行企劃層，建議下一輪順手做一次純資料規格清掃，把 BOM、舊欄位描述、錯誤 `sourceId` 這種非玩法問題先清乾淨，之後每次企劃 review 會更省力。
